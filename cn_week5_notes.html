Topic 1: TCP Flow Control Algorithm High Weightage

TCP sliding window — Go-Back-N principle

TCP uses a sliding window flow control algorithm with the Go-Back-N (GBN) ARQ principle. On a timeout, the sender retransmits ALL data currently inside the sender window — not just the lost segment. Sequence numbers start at the ISN established during 3WHS (we use 0 in examples only for simplicity).

Key explanation points — Basic sliding window example

1Example walkthrough — receiver buffer = 4 KB: App writes 2 KB → sent with seq=0. Receiver gets 2 KB → ACK=2048, window=2048 (2 KB free). App writes 2 KB more → sent from seq=2048. Receiver buffer now FULL → ACK=4096, window=0. Sender is blocked. App reads 2 KB from buffer → window reopens → ACK=4096, window=2048. Sender resumes sending.
2Sender retransmission buffer: Sender keeps a copy of every sent-but-unacknowledged segment in its buffer. If timeout occurs before ACK is received → retransmit all data in sender window (GBN principle). Data is only removed from sender buffer when the corresponding ACK arrives.
3Duplicate ACK (Dupack): When a receiver gets an out-of-order segment (e.g., seq=2048 arrives but seq=1024 is missing), it buffers the out-of-order segment AND sends an ACK for the LAST byte correctly received (e.g., ACK=1024 again). This repeated ACK for the same byte is called a duplicate acknowledgement (Dupack).
4Dupack example: Receiver has bytes 0,1,2 → ACK=3. Byte 3 lost. Bytes 4,5,6,7 arrive → receiver sends Dupack=3 three times. Eventually byte 3 retransmitted → receiver now has 0–7 → sends ACK=8 (one cumulative ACK jumping forward). Dupack triggers fast retransmit in congestion control.

Key explanation points — Telnet problem & solutions

5Telnet keystroke problem: Telnet reacts to every single keystroke. TCP creates a 21-byte segment for every 1-byte character typed: 20-byte TCP header + 1 byte data. This means only 1/21 ≈ 5% of the segment is useful data — 95% is overhead. Also: an ACK + window-update is sent for each character read by the receiving app. Massive bandwidth waste.
6Delayed Acknowledgement (solution 1 — receiver side): Receiver delays sending the ACK for up to 500 ms, hoping to receive more data packets within that interval. If more data arrives, one ACK covers multiple bytes. Prevents sender from immediately sending another tiny segment (sender won't send until it gets an ACK back with available window space).
7Nagle's Algorithm (solution 2 — sender side): When data comes into the sender in small pieces: (1) Send the FIRST piece immediately. (2) Buffer ALL subsequent small pieces until the ACK for the first piece arrives. By the time the first ACK comes back, multiple bytes have accumulated in the buffer → sender combines them into one larger segment. Only ONE small packet is outstanding in the network at any time.
8Nagle's Algorithm — when NOT to use: Nagle intentionally increases end-to-end delay (waits for ACK before sending next piece). For latency-sensitive applications (e.g., online gaming, interactive commands), this slow response time is unacceptable. Many apps disable Nagle using TCP_NODELAY socket option.
9Nagle + Delayed ACK = STARVATION: Nagle algorithm: sender waits for ACK before sending next piece. Delayed ACK: receiver delays the ACK by 500 ms. If both are enabled simultaneously: sender is waiting for receiver's ACK → receiver is waiting 500 ms before sending ACK → sender is permanently blocked for 500 ms. This causes significant response delay (starvation). Do NOT enable both together.
10Silly Window Syndrome (SWS): Sending app sends data in large blocks (10 Mbps rate). Receiving app reads data 1 byte at a time. Receiver buffer fills up → window=0 → sender blocked. App reads 1 byte → receiver sends window=1. Sender sends 1 byte with 20-byte header (again, tiny useful data). This becomes a vicious loop of 1-byte advertisements → same overhead problem as Telnet but from the receiver side.
11Clark's Solution (SWS fix — receiver side): Do NOT send a window update for 1 byte. Instead, receiver waits until "sufficient space" is available in its buffer, then sends the window update. "Sufficient space" = implementation-defined (e.g., at least MSS bytes free, or at least 50% of buffer is free). Receiver stays quiet (no window update) until enough buffer is available — prevents sender from sending many tiny segments.
12Nagle vs Clark — complementary, NOT conflicting: Nagle algorithm fixes tiny segments from the sender side (sending application delivers data 1 byte at a time). Clark's solution fixes tiny window updates from the receiver side (reading application reads data 1 byte at a time). They solve different halves of the same problem and work together without causing starvation (unlike Nagle + Delayed ACK).
13PSH flag (bypassing wait): Both Nagle and Clark add some delay: waiting for data accumulation or waiting for buffer to free. For real-time applications this is unacceptable. Set the PSH (Push) flag in the TCP header: (1) Tells the sender to create and send a segment immediately without waiting for more data. (2) Tells the receiver to deliver data to the application immediately without buffering. Bypasses both Nagle and Clark delays.

Topic 2: TCP Timers & RTO Estimation High Weightage

TCP Retransmission Timeout (RTO)

When a segment is sent, a retransmission timer starts. If ACK arrives before timeout → timer stops. If timer expires before ACK → segment is retransmitted AND congestion control is triggered. The challenge: what should the RTO value be? Too small → spurious retransmissions. Too large → slow recovery from loss.

Key explanation points — RTO Estimation

1Why RTT estimation is hard at transport layer: At the data link layer: two nodes are directly connected → propagation delay is fixed → RTT has low variance → mean is a good estimate. At the transport layer: two hosts communicate through the entire internet → many hops, queue delays, varying routes → RTT has high variance. The mean alone is insufficient — it can lead to spurious RTOs when actual RTT deviates significantly from the mean.
2Jacobson's algorithm (1988): Maintains a variable SRTT (Smoothed Round Trip Time) = best current estimate of RTT to the destination. TCP measures the actual RTT (R) for each ACK received = time between segment transmission and ACK receipt. Updates SRTT using EWMA (Exponentially Weighted Moving Average).
3EWMA formula for SRTT: SRTT = α × SRTT + (1 − α) × R. Where R = measured RTT for this ACK. α = smoothing factor = 7/8 (= 0.875). High α means old values are forgotten slowly (more weight on history). Low α = very reactive to new measurements. Jacobson chose α = 7/8 for stability.
4Problem with SRTT alone for RTO: Early TCP used RTO = 2 × SRTT. Under high network load, RTT variance increases sharply. If actual RTT deviates significantly from SRTT, RTO = 2×SRTT is inflexible → too many spurious RTOs. Solution: also track the VARIANCE of RTT.
5RTTVAR — RTT Variance tracking: RTTVAR = β × RTTVAR + (1 − β) × |SRTT − R|. Where |SRTT − R| = current deviation between estimated and measured RTT. β = 3/4. Jacobson sets β = 3/4 to give more weight to historical variance but adapt to changes.
6RTO formula with variance: RTO = SRTT + 4 × RTTVAR. The "4" was chosen by Jacobson using clever arithmetic — he used integer add/subtract/bit-shift operations to make computation lightweight. 4 = 2², so the multiply can be done with a left-shift (fast CPU operation). Under high load → RTTVAR increases → RTO increases automatically → adapts to network dynamics.
7Karn's Algorithm: Problem: if a segment is retransmitted, you cannot tell which transmission the ACK refers to (ambiguity). Using an ACK from a retransmitted segment to estimate RTT would corrupt SRTT. Karn's rule: Do NOT update SRTT on any retransmitted segment. Additionally: double the RTO on each successive timeout (exponential backoff) until the segment gets through. Once a new (non-retransmitted) ACK arrives, reset RTO to SRTT + 4×RTTVAR.
8Karn's exponential backoff: First timeout → retransmit → RTO = 2×RTO. Second timeout → RTO = 4×RTO. Third timeout → RTO = 8×RTO. This keeps increasing until ACK is received. Purpose: prevents flooding the network with retransmissions when it is heavily congested. Gives the network time to recover.

Key explanation points — Other TCP Timers

9Persistent Timer: Prevents deadlock when receiver advertises window=0. After sender receives window=0 advertisement and is blocked, the persistent timer fires periodically. When it fires, sender sends a Window Probe (probe packet) to the receiver to ask for an updated window size. Receiver replies with current window → if > 0, sender resumes. Prevents permanent deadlock from lost window-update ACKs.
10Keepalive Timer: Closes a connection that has been idle for too long. If no data has been sent for a long duration (typically 2 hours), the keepalive timer fires → sender sends a keepalive probe. If no response → connection is closed. Prevents zombie connections from wasting server resources when one side crashes silently.
11Time Wait Timer: Used during connection closure by the active closer. Duration = 2 × MSL (Maximum Segment Lifetime). Already covered in Week 4. Ensures delayed packets from the closed connection expire before the port is reused, and allows resending the final ACK if needed.

Topic 3: TCP Congestion Control — Slow Start, Tahoe & Reno Most Important

TCP Congestion Control History

In 1986, the growing internet experienced the first congestion collapse — goodput dropped by a factor of >100 as retransmissions made congestion worse. Van Jacobson investigated this and proposed TCP congestion control in 1988, designed to be compatible with existing TCP implementations (no protocol changes needed). Key insight: in wired networks, packet loss ≈ congestion.

Key explanation points — Core Algorithm

1Congestion Window (cwnd): TCP maintains a variable cwnd = maximum bytes the sender may have unacknowledged in the network at any time. Sending rate ≈ cwnd / RTT. Sender window = min(cwnd, rwnd). cwnd is managed by the congestion control algorithm; rwnd is managed by flow control (receiver-advertised window).
2ACK clocking: Key Jacobson insight: acknowledgements arrive at the sender at the RATE of the slowest link in the path (bottleneck). Example: fast link → slow link → receiver. Packets burst through fast link; the slow link spaces them out. ACKs return at the rate of the slow link. Sender uses ACK arrival rate as a clock to trigger sending new segments at the network's actual rate.
3Why simple additive increase is too slow: Example: 10 Mbps link × 100 ms RTT → BDP = 1 Mb → need 100 packets in window. Starting at cwnd=1, increasing by 1 per RTT → takes 100 RTTs × 100 ms = 10 seconds to reach capacity. Most HTTP connections close before 10 seconds → never reaches full speed. Need faster initial growth.
4Slow Start: Despite the name, Slow Start is NOT slow — it starts slowly but grows EXPONENTIALLY. For every ACK received, cwnd increases by 1. This means cwnd DOUBLES every RTT: 1→2→4→8→16... RTT 1: send 1, get 1 ACK → cwnd=2. RTT 2: send 2, get 2 ACKs → cwnd=4. RTT 3: send 4, get 4 ACKs → cwnd=8. Reaches BDP in log₂(100) ≈ 7 RTTs instead of 100 RTTs.
5ssthresh (Slow Start Threshold): Slow Start grows exponentially until cwnd reaches ssthresh. At ssthresh, TCP switches from Slow Start (exponential) to Congestion Avoidance (additive increase, linear). Initially ssthresh = BDP or an arbitrarily large value (maximum cwnd). Updated when congestion is detected.
6Congestion Avoidance (AIMD additive increase): Once cwnd ≥ ssthresh, switch to additive increase. cwnd increases by MSS per RTT (not per ACK). Formula approximation: cwnd += MSS × (MSS / cwnd) per ACK. This gives approximately +1 MSS per RTT regardless of current cwnd size. Grows linearly toward link capacity.
7Two congestion signals in TCP: (1) Retransmission Timeout (RTO): no ACK received — sure sign of severe congestion, network may be completely choked. (2) Triple Duplicate ACK (3 DupACK): receiving 3 duplicate ACKs — some packets still flowing (receiver is receiving and ACKing out-of-order segments). Less severe than RTO — link is not completely choked.

Key explanation points — TCP Tahoe & TCP Reno

8TCP Tahoe — on RTO: Set ssthresh = cwnd / 2. Set cwnd = 1 MSS. Restart Slow Start from cwnd=1. Grow exponentially to new ssthresh, then additive increase.
9TCP Tahoe — on 3 DupACK (Fast Retransmit): Retransmit the lost segment immediately without waiting for timeout (fast retransmit). Set ssthresh = cwnd / 2. Set cwnd = 1 MSS. Restart Slow Start. TCP Tahoe treats 3 DupACK the same as RTO — always drops to cwnd=1 and restarts Slow Start.
10TCP Reno — improvement over Tahoe: Observation: 3 DupACKs means some packets ARE flowing (receiver is getting bytes 5,6,7 even though byte 4 is missing). The network is NOT completely choked. Dropping cwnd to 1 and doing slow start is too aggressive. TCP Reno uses Fast Recovery instead.
11TCP Reno — Fast Retransmit + Fast Recovery (on 3 DupACK): Step 1: Immediately retransmit the lost segment (fast retransmit). Step 2: Set ssthresh = cwnd / 2. Step 3: Set cwnd = ssthresh + 3 MSS (the +3 accounts for the 3 packets that the receiver already confirmed receiving via DupACKs). Step 4: For each additional DupACK received → cwnd += 1 MSS, send new segment if allowed. Step 5: When a NEW ACK (non-duplicate) arrives → set cwnd = ssthresh → resume Congestion Avoidance (directly, skipping Slow Start).
12TCP Reno — on RTO (still drops to cwnd=1): If congestion is detected via RTO (timeout, no ACK at all) → severe congestion. Set ssthresh = cwnd / 2. Set cwnd = 1 MSS. Restart Slow Start. RTO always causes full restart regardless of TCP version — timeout means severe congestion.
13TCP NewReno: Improved version of TCP Reno. Handles multiple packet losses in one window more efficiently. Stays in fast recovery until all lost packets in that window are ACKed. Reduces unnecessary Slow Start entries.
14TCP SACK (Selective ACK — RFC 2018): Standard TCP uses cumulative ACK (GBN principle) — on timeout, retransmit all segments in window. TCP SACK uses Selective Repeat ARQ — receiver explicitly tells sender WHICH segments were received (out-of-order) via SACK option in ACK. Sender retransmits ONLY the missing segments. More efficient under high loss.

Topic 4: UDP & QUIC Medium Weightage

Why UDP exists alongside TCP

TCP has significant overhead: 3WHS (3 messages for setup + 3 for close), Slow Start phase, retransmissions (block new data), per-connection state. For short messages or real-time apps, these overheads are unacceptable. UDP = minimal wrapper over IP — just adds port numbers and checksum. Fast, connectionless, unreliable.

Key explanation points — UDP

1UDP features: No connection establishment. No reliability, no ACK. No flow control. No congestion control. No ordered delivery. Works well when the network has low loss probability. Application must implement its own loss recovery if needed. Extremely low overhead — just adds 4 fields to IP.
2UDP header (4 fields only): Source Port (16 bits) — sender's port. Destination Port (16 bits) — receiver's port. Length (16 bits) — total UDP datagram length (header + data). Checksum (16 bits) — error detection. That's it — 8 bytes total vs TCP's 20+ bytes. Rest is data payload.
3UDP Checksum with pseudo-header: During checksum computation, TCP/UDP create a pseudo-header NOT transmitted with the packet — used only for computation. Pseudo-header fields (from IP header): Source IP, Destination IP, Protocol field, Reserved bits. After computing checksum, pseudo-header is discarded. At receiver, reconstruct pseudo-header, recompute checksum, verify match.
4Why include pseudo-header in checksum? IP header has its own checksum, but the IP checksum CHANGES at every router hop (routers update TTL → recompute IP checksum). The transport layer checksum is end-to-end and never changes at intermediate hops. Including the pseudo-header provides double validation of src IP, dst IP, and protocol — ensures no router error corrupted these critical routing fields.
5Internet Checksum computation: Divide the message into fixed-size blocks → perform ones-complement addition → result = checksum. This is NOT a cryptographic hash — designed only to detect common transmission errors (bit flips during signal processing, ADC/DAC quantization errors). Computationally lightweight by design.
6Checksum is NOT security: Checksum detects random transmission errors (bit flips from physical layer noise/quantization). It does NOT protect against intentional attacks. An attacker can recompute and forge a matching checksum. For security, use cryptographic hashes (HMAC, SHA-256) or TLS.
7UDP use cases: DNS — short query/response, no connection needed. BOOTP/DHCP — network configuration, short messages. TFTP — Trivial FTP for small file transfers. SNMP — Simple Network Management Protocol, small query/traps, also cuts through congestion (UDP bypasses congestion control — useful for network management during congestion when TCP flows back off). VoIP, video streaming, gaming — real-time, tolerates some loss.

Key explanation points — QUIC

8What is QUIC? QUIC = Quick UDP Internet Connections. Developed by Google, detailed paper published 2017. Runs on top of UDP (for direct IP access without TCP overhead). Implements reliability, flow control, congestion control, and security INSIDE the application layer on top of UDP. No slow start delay for subsequent connections to the same server.
9QUIC architecture: vs TCP: HTTP → (SSL/TLS) → TCP → IP. QUIC: HTTP → QUIC (with embedded security) → UDP → IP. Security is built into QUIC itself — no separate TLS/SSL handshake layer needed. Every QUIC packet is end-to-end encrypted by default.
10QUIC handshake — 1-RTT initial: Client sends equate-CHLO (Client Hello, no credentials). Server sends REJECT (client credentials invalid) + server's security certificate. Client now has certificate. Client sends complete client-CHLO (with credentials) + starts sending encrypted requests. Server sends server-CHLO (server hello) + encrypted responses. Total: 1 RTT for handshake. Data flows immediately.
11QUIC — 0-RTT for subsequent connections: After the first connection, client already has the server's security certificate cached. Client can send complete client-CHLO immediately without waiting for the server's REJECT+certificate cycle. → 0-RTT for all subsequent connections to the same server. Versus TCP: requires 3WHS for EVERY new connection. QUIC: 0-RTT after the first connection.
12Head-of-Line (HOL) Blocking — the TCP problem: HTTP/2 multiplexes multiple streams over ONE TCP connection. If a single packet from any stream is lost → TCP BLOCKS delivery of ALL streams (cumulative ACK, GBN principle) even if other streams' packets are fully received and buffered. One lost red packet blocks all blue and black stream data waiting in buffer.
13QUIC solves HOL blocking: QUIC uses UDP (which has no blocking) + implements stream-wise flow control and congestion control. Each stream is independent. A lost packet in Stream A does NOT block Streams B, C, D. QUIC delivers other streams' data immediately while retransmitting the lost packet for Stream A only.
14QUIC retransmission — new seq numbers: Unlike TCP, QUIC gives a NEW sequence number to every retransmitted packet (different from original transmission). This fully removes the Karn ambiguity problem — QUIC can always determine exact RTT from any ACK, including those for retransmitted packets.

Topic 5: Socket Programming Medium Weightage

What is socket programming?

The entire TCP/IP stack (Transport + Network layer) is implemented in the OS kernel. Applications in user space communicate with the kernel protocol stack via system calls. A socket = a logical pipe from one process to another process, represented as a file descriptor in UNIX. Socket programming = using system call APIs to create and manage these logical pipes.

Key explanation points — Concepts

1Where is the protocol stack? Physical layer → NIC hardware. Lower Data Link (MAC) → NIC firmware/driver. Upper Data Link (LLC) + Network (IP) + Transport (TCP/UDP) → OS kernel (Unix/Linux: under /usr/src/linux/net/). Application layer → user space. User space applications access the kernel stack through system calls.
2Port-based multiplexing: Multiple processes on the same machine communicate over the network simultaneously. IP address identifies the machine. Port number identifies the specific process/application on that machine. Example: Machine at 202.141.81.2, port 8081 (running HTTP client) connects to 203.12.23.43 port 80 (HTTP server). Each connection is a separate logical pipe (socket).
3Stream socket vs Datagram socket: SOCK_STREAM = TCP socket (reliable, connection-oriented, ordered). SOCK_DGRAM = UDP socket (unreliable, connectionless). SOCK_RAW = raw socket (bypasses transport layer, talks directly to IP — not covered for exam).
4Active open vs Passive open: Server = passive open: announces address (bind+listen), waits for connections. Client = active open: initiates a connection when data needs to be sent. Client knows the server IP + port (e.g., well-known HTTP port 80). Client's own port is randomly chosen at runtime.
5Host byte order vs Network byte order: Computers can be little-endian (store least significant byte first: 0D, 0C, 0B, 0A) or big-endian (most significant first: 0A, 0B, 0C, 0D). Network uses big-endian (network byte order). Before putting port number in socket address struct, convert from host byte order to network byte order with htons() (host-to-network short). At receiver: ntohs() to convert back.

Key explanation points — TCP Socket API

6socket() system call: Creates a socket file descriptor. Parameters: domain (AF_INET for IPv4), type (SOCK_STREAM for TCP, SOCK_DGRAM for UDP), protocol (0 for default). Returns: socket file descriptor (integer). Usage: int s = socket(AF_INET, SOCK_STREAM, 0);
7bind() — server only: Binds the socket to a specific IP + port on the local machine. Parameters: socket fd, pointer to sockaddr_in struct (containing AF_INET, INADDR_ANY for local IP, htons(port)). INADDR_ANY = use whichever IP is assigned to this machine's NIC. Error if port already in use by another process.
8listen() — server only: Puts socket into listening state — announces "I accept connections on this port." Parameter: socket fd + backlog (max number of connections queued while server is busy). Backlog=5 means up to 5 pending connections can wait. Excess connections are dropped.
9connect() — client only: Initiates TCP 3-way handshake with the server. Parameters: socket fd, server address struct (server IP + port). This call BLOCKS until the handshake is complete (or fails). Inside connect(): SYN is sent → SYN+ACK received → ACK sent.
10accept() — server only: Blocks until a client connection arrives. When client connects, accept() returns a NEW socket file descriptor (connfd) specific to THAT client. The original server socket keeps listening for more clients. All data exchange with this specific client is through connfd (not the original socket fd).
11send/recv (TCP) and sendto/recvfrom (UDP): TCP (stream socket): use send(connfd, buffer, len, flags) and recv(connfd, buffer, len, flags). recv() returns 0 when connection is closed by remote side. UDP (datagram socket): use sendto() and recvfrom() — include the remote address in each call since there's no established connection. Client's port changes randomly each run (randomly assigned by OS).
12close() — both sides: After data transfer is done, close(fd) closes the connection. For TCP: triggers the FIN sequence (TCP state machine: ESTABLISHED → FIN_WAIT_1 → ... → CLOSED). Server should close(connfd) after finishing with a client; client should close its socket after receiving all data.

Key explanation points — Server Design Patterns

13Iterative Server: Server loop: accept() → send/receive with client → close(connfd) → accept() again. Problem: while sending/receiving with Client 1, all other clients are BLOCKED in the backlog queue. Cannot handle concurrent clients efficiently. OK only for very low request rates.
14Concurrent Server (fork): After accept() returns connfd → call fork() to create a child process. Child process: handles send/recv with connfd (closes original server socket). Parent process: immediately returns from fork(), closes connfd, calls accept() again to handle next client. Multiple clients handled in parallel by different child processes. Risk: zombie child processes if parent doesn't wait() for them.
15Select() — I/O Multiplexing: Used when a single process must monitor multiple file descriptors (e.g., chat app: multiple client sockets + keyboard input). select(nfds, &readfds, &writefds, &exceptfds, &timeout) waits until ANY of the monitored file descriptors becomes ready for read/write/exception. Returns number of ready fds. Use FD_SET to add fds; FD_ISSET to check if ready after select returns. No child processes needed → no zombie risk. Advantage over fork: single process, simpler resource management.
16select() return values: −1 = error. 0 = timeout expired (no fds ready). >0 = number of sockets with pending events. After select returns >0: use FD_ISSET(fd, &readfds) to identify which fd is ready, then call accept()/recv() on that fd.

  • PSH flag: bypass both Nagle and Clark → send/deliver immediately
  • Dupack: receiver sends same ACK repeatedly for out-of-order segments. Trigger: 3 Dupacks = congestion in TCP Tahoe/Reno
  • RTO: SRTT = (7/8)×SRTT + (1/8)×R. RTTVAR = (3/4)×RTTVAR + (1/4)×|SRTT−R|. RTO = SRTT + 4×RTTVAR
  • Karn's algorithm: no SRTT update on retransmitted segments; double RTO on each timeout
  • Persistent timer: fires when window=0 → sends window probe → deadlock prevention
  • Keepalive timer: fires after long idle period (~2 hours) → sends probe → closes if no response
  • Congestion collapse: 1986 internet event. Van Jacobson invented TCP congestion control in 1988
  • Slow Start: cwnd doubles every RTT (NOT slow — exponential growth). Ironic name
  • ssthresh: when cwnd reaches ssthresh → switch from exponential to linear (Congestion Avoidance)
  • On RTO (both Tahoe & Reno): cwnd=1, ssthresh=cwnd/2, restart Slow Start
  • On 3 DupACK — Tahoe: cwnd=1, ssthresh=cwnd/2, restart Slow Start
  • On 3 DupACK — Reno: Fast Recovery: ssthresh=cwnd/2, cwnd=ssthresh+3, then linear; no Slow Start restart
  • UDP header: 4 fields only (src port, dst port, length, checksum) = 8 bytes total
  • Pseudo-header (not transmitted): src IP, dst IP, protocol, reserved — for checksum validation of IP fields
  • Checksum = ones-complement addition of message blocks. NOT cryptographic
  • QUIC: Google 2017. UDP-based. 1-RTT initial, 0-RTT subsequent. Embedded security. HOL-blocking free. New seq# for retransmissions
  • socket() → bind() → listen() → accept() → send/recv → close() (TCP server)
  • socket() → connect() → send/recv → close() (TCP client — no bind/listen)
  • Iterative server: handle one client at a time. Concurrent: fork() for each client. select(): monitor multiple fds in one process
  • htons(): host byte order → network byte order (big-endian). ntohs(): reverse

RTO estimation formulas

SRTT= α×SRTT + (1−α)×R    α = 7/8 (EWMA — exponentially weighted moving average)
RTTVAR= β×RTTVAR + (1−β)×|SRTT−R|    β = 3/4
RTO= SRTT + 4×RTTVAR    (4 = 2² = left-shift, fast computation)
KarnNo SRTT update on retransmission. Each timeout: RTO = 2×RTO (exponential backoff)

TCP Congestion Control phases (Reno)

Phase 1Slow Start: cwnd starts at 1 MSS. Doubles every RTT (+1 per ACK). Until cwnd = ssthresh
Phase 2Congestion Avoidance: +1 MSS per RTT (linear). Formula: cwnd += MSS²/cwnd per ACK
On RTOssthresh = cwnd/2. cwnd = 1. Go to Phase 1 (Slow Start)
On 3DupACK (Reno)Fast Retransmit: retransmit lost segment. ssthresh=cwnd/2. cwnd=ssthresh+3. Each more Dupack: cwnd+=1. On new ACK: cwnd=ssthresh → Phase 2
On 3DupACK (Tahoe)ssthresh=cwnd/2. cwnd=1. Go to Phase 1 (Slow Start like RTO)

Tricky NPTEL MCQ traps — don't get these wrong

1Slow Start is NOT slow — it is exponential growth starting from a slow initial value
2Nagle + Delayed ACK = starvation. Nagle + Clark = safe (complementary)
3Tahoe and Reno BOTH drop to cwnd=1 on RTO. They DIFFER only on 3 DupACK handling
4Reno Fast Recovery: cwnd = ssthresh + 3 (not ssthresh + 0 or ssthresh alone)
5Karn: do NOT update SRTT on retransmission. Do exponential backoff on timeout
6Pseudo-header is NOT transmitted — used only for checksum computation
7UDP header = 4 fields, 8 bytes. TCP header = minimum 20 bytes (much larger)
8QUIC: 1-RTT handshake for FIRST connection; 0-RTT for all subsequent connections to same server
9TCP SACK uses Selective Repeat. Standard TCP uses Go-Back-N (cumulative ACK)
10accept() returns a NEW fd for each client — original server socket stays in LISTEN state
11Client does NOT call bind() or listen() — only server does
12ACK clocking: ACKs arrive at sender at the rate of the SLOWEST link in the path
13Persistent timer prevents window=0 deadlock. Keepalive timer closes idle connections
14QUIC assigns NEW sequence numbers to retransmissions → solves Karn ambiguity completely
