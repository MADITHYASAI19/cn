Topic 1: Bandwidth-Delay Product & Buffer Management High Weightage

What is Bandwidth-Delay Product (BDP)?

BDP = Link Bandwidth × One-way Propagation Delay. It represents how many bits can be "in the pipe" (in flight) at any point in time. It is the single most important parameter for deciding which ARQ protocol to use and what window size to configure for maximum link utilization.

Key explanation points — BDP & Window Sizing

1BDP formula & example: BDP = Bandwidth × One-way delay. Example: 50 Kbps × 250 ms = 12,500 bits = 12.5 Kbits. If segment size = 1000 bits → BDP = 12.5 segments. This is how many segments can be simultaneously in the pipe in one direction.
2Round Trip Time (RTT): RTT ≈ 2 × one-way propagation delay (assuming no congestion or intermediate delay). It is the time from sending a segment to receiving its ACK. Measured as the time difference between transmission and acknowledgement reception.
3Pipe analogy: Think of the network link as a water pipe. Latency = length of the pipe. Bandwidth = cross-section (diameter) of the pipe. BDP = volume of the pipe (how much water it can hold). For two-way communication there are two pipes: one for data (forward) and one for ACKs (reverse).
4Maximum outstanding segments (two-way): The data pipe holds up to BD segments. The ACK pipe holds another BD segments (ACKs for data already received by the receiver and sitting in receiver buffer). Total outstanding = 2 × BD. So if BDP = 12.5 segments → max outstanding = 25 segments.
5Optimal sender window size: W = 2×BD + 1. The "+1" accounts for one ACK just received at the sender but not yet processed. This ensures BOTH the data pipe and ACK pipe are always fully utilized → maximum link utilization. This is the theoretical upper bound on window size.
6Window size → Sequence number bits: Once W is known: For Go-Back-N: W ≤ 2ⁿ − 1 → n ≥ log₂(W+1). For Selective Repeat: W ≤ 2ⁿ/2 → n ≥ log₂(2W). BDP thus determines both window size AND minimum bits needed in the sequence number field.
7Protocol selection using BDP (example): Bandwidth = 1 Mbps, delay = 1 ms → BDP = 1 Mbps × 1 ms = 1 Kbit = 1024 bits. Segment = 1 KB = 8192 bits. BDP is SMALLER than one segment. The pipe cannot even hold one complete segment → no benefit from sending multiple segments in parallel.
8When to use Stop-and-Wait: When BDP < segment size — pipe cannot hold a full segment. Sliding window provides no parallelism advantage here. Stop-and-Wait has the LEAST complexity (no sender/receiver window maintenance, no sequence number management) → always prefer it in this case.
9When to use Sliding Window: When BDP >> segment size — many segments can be in-flight simultaneously. Set window W = 2×BD+1 to fill the pipe completely. Then choose Go-Back-N or Selective Repeat based on error rate and buffer complexity trade-off.

Key explanation points — Transport Layer Buffer Architecture

10Sender-side interface (TportSend): A hypothetical TportSend() function is called periodically by the transmission rate control module (based on current flow/congestion window). Application writes data to a source buffer at its own rate; TportSend reads from that buffer and passes segments to IP.
11Source buffer & write call blocking: Application may generate data at 10 Mbps but rate control allows only 2 Mbps. The difference accumulates in the source buffer. The write() system call BLOCKS the port when the buffer is full — prevents the application from overflowing the buffer. Connection-specific: one buffer per connection.
12Receiver-side interface (TportRecv): A hypothetical TportRecv() receives segments from the IP layer. It reads the destination port number in the transport header and routes the data to the correct application's receive queue. One receive queue per port/application.
13Read call blocking: The application makes a read() system call to get data from its receive queue. This call BLOCKS when the queue is empty — waits until new data arrives. An interrupt signal from TportRecv unblocks the call. The call also blocks until all requested data is retrieved.
14Buffer type 1 — Pool of identically-sized buffers: Each buffer slot holds exactly one segment. Works when all segments are the same size. Low complexity. Zero wasted space if segment sizes are uniform.
15Buffer type 2 — Chained fixed-size buffers (linked list): Individual buffers of max-segment-size connected as a linked list. Used for variable segment sizes. Problem: significant space wasted if segment sizes vary widely. Small segments waste most of a large buffer slot.
16Buffer type 3 — Variable-size buffers: Each buffer is dynamically sized to match the incoming segment. Best memory utilization. Disadvantage: complex implementation — dynamic memory allocation is required for every segment.
17Buffer type 4 — Single large circular buffer (preferred): One large circular buffer per connection. Segments of different sizes placed one after another in the circular space. Unused space can store the next arriving segment. Good utilization under heavy load. Balances complexity and efficiency.
18Dynamic buffer management: The receiver's available buffer space changes dynamically because the application reads at a different rate than the network delivers. If network delivers at 1 Mbps and app reads at 10 Kbps, the buffer fills up → would cause packet drops. Solution: receiver continuously ADVERTISES its available buffer space to the sender.
19Dynamic buffer management — example: In sliding window, sender and receiver dynamically adjust buffer allocation. Receiver advertises remaining buffer space with each ACK. Sender limits its window to that advertised size → never sends more data than receiver can hold. Example in lecture: A requests 8 buffers, B has only 4 → B advertises 4. A sends 3 at a time based on advertisement.
20Deadlock prevention (window = 0): When receiver advertises window = 0 (buffer full), sender stops sending. If receiver later frees space and sends an ACK announcing it, but that ACK is LOST → sender is permanently blocked (deadlock). Prevention: receiver must periodically send DUPLICATE ACKs after timeout, re-announcing available buffer space — even if no new data has arrived.

Topic 2: Congestion Control High Weightage

What is Congestion?

Congestion occurs when the total incoming rate (λ) at an intermediate router buffer exceeds its outgoing rate (μ). If λ > μ, the buffer fills up → packet drops. Retransmissions cause even more load → a vicious cycle called congestion collapse. The transport layer must detect and avoid this.

Key explanation points

1Max-Flow Min-Cut (centralized): In a centralized network, you can apply the Max-Flow Min-Cut theorem to find the bottleneck capacity and restrict flows to that maximum. In the real distributed internet, each router only has local information — this calculation is impossible in real-time. Congestion control must work in a decentralized, distributed way.
2Congestion avoidance vs congestion control: Since a priori estimation of the exact congestion point is difficult in a distributed environment, we do not perform strict "congestion control." Instead, we do congestion avoidance — detect congestion when it starts happening (infer from packet loss) and back off before it becomes severe.
3Packet loss as the congestion signal: In wired networks, physical bit errors are rare. If a packet is dropped, it is almost certainly due to buffer overflow at some intermediate router (λ > μ). Packet loss = signature of congestion. Transport layer use this to trigger rate reduction.
4Updated sender rate formula: Previously: Sender_rate = Receiver_rate (from flow control). Now with congestion control: Sender_rate = min(network_supported_rate, receiver_advertised_rate). Flow control sets the upper bound from the receiver side. Congestion control sets a separate upper bound from the network side. Sender uses the minimum.
5Goodput vs offered load curve: As offered load increases from 0, goodput increases proportionally up to the link capacity. At the saturation point, retransmissions begin → goodput drops sharply. This drop is called congestion collapse. End-to-end delay also spikes at this point as router queues fill up.
6Dynamic bandwidth sharing: Flows enter and exit the network dynamically. If Flow 1 and Flow 2 share a 1 Mbps bottleneck, each gets ~500 Kbps. If Flow 3 (100 Kbps) joins, it steals bandwidth → F1 and F2 each get ~450 Kbps. If F2 finishes, F1 captures ~900 Kbps. Congestion control must handle this dynamically without central coordination.
7Fairness: Bad congestion control can starve some flows entirely (some flows get 0 bandwidth while others hog all of it). Like cars stuck in a traffic jam forever. In networks, starvation means some applications cannot get any service. We need fairness guarantees.
8Max-Min Fairness: An allocation {λ₁, λ₂, ...} is max-min fair if the bandwidth of any flow CANNOT be increased without DECREASING the bandwidth of another flow that has equal or less bandwidth. Example: 3 flows on a bottleneck → each gets 1/3. To give Flow A more than 1/3, you must take from B or C.
9AIMD — Additive Increase Multiplicative Decrease (Chiu & Jain, 1989): The distributed algorithm that achieves both max-min fairness AND full link utilization simultaneously. Rule: If no congestion detected → w(t+1) = w(t) + a (additive increase). If congestion detected (packet loss) → w(t+1) = b × w(t), where 0 < b < 1 (multiplicative decrease). 'a' is the additive factor, 'b' is the multiplicative factor.
10Why AIMD converges to optimal (geometric proof): Plot User1 rate on x-axis, User2 rate on y-axis. Fairness line: x = y (45° line). Efficiency line: x + y = C (total capacity). AIMD additive increase moves diagonally at 45° (adds equally to both → toward fairness). Multiplicative decrease moves toward origin (radially). Together: spirals inward to the intersection point (max efficiency + max fairness) = optimal point.
11Why AIAD fails: AIAD = Additive Increase Additive Decrease. Additive increase moves parallel to fairness line. Additive decrease also moves parallel. Net result: oscillation ALONG the efficiency line. Never converges to the fairness point.
12Why MIMD fails: MIMD = Multiplicative Increase Multiplicative Decrease. Both increase and decrease are radial (toward/away from origin). Net result: oscillation radially along a line from origin. May eventually hit efficiency line but NOT the fairness point.
13TCP uses AIMD: TCP implements AIMD through the congestion window (cwnd). cwnd starts small and grows additively (congestion avoidance phase). On packet loss (congestion detected) → cwnd drops significantly (multiplicative decrease). This is expanded in TCP details with slow start, fast retransmit, etc.

Max-Min Fairness — example from lecture

Four flows (A, B, C, D) in a network. Flows B, C, D all pass through the same bottleneck link → each gets 1/3 capacity. Flow A shares a different link with only B → A gets 2/3 of that link's capacity. This is max-min fair: to give B more than 1/3, you must take from C or D.

Topic 3: Combining Transport Services & Socket Model Medium Weightage

Putting it all together

The full transport layer protocol combines all individual services into a coherent state machine. It requires (1) a unique identifier for each connection, (2) a stateful protocol that remembers connection state before allowing send/receive, and (3) a standard set of primitives (connect, send, receive, disconnect) that applications use.

Key explanation points

1Port number — identifying applications: On a single machine, multiple applications run simultaneously (multi-tasking). Eg: D1 runs A1, A2; D3 runs A1, A2, A3. The network layer IP address uniquely identifies the machine. The port number uniquely identifies the application on that machine. Transport layer header carries both source port and destination port.
2The logical pipe (socket): A transport-layer connection between two apps is a logical pipe — all of TCP/IP's services are applied over this pipe to make it reliable. In TCP/Unix systems, this logical pipe is called a socket.
3Socket = 6-tuple: A socket (unique connection identifier) is defined by: {Source IP, Source Port, Source ISN, Destination IP, Destination Port, Destination ISN}. All six parameters must match for a packet to belong to a connection. The ISNs prevent confusion with old connections that used the same IP+port pair.
4State-full protocol: The transport layer MUST remember the state of each connection (IDLE, CONNECTING, ESTABLISHED, CLOSING). A send() or recv() call is only valid in the ESTABLISHED state. Calling send() before establishing the connection is invalid. The protocol transitions between states based on received messages and application calls.
5Hypothetical transport primitives: Server: listen() → then recv() to receive data → send() to reply → disconnect(). Client: connect() → send() data → recv() reply → disconnect(). The connect() triggers 3-way handshake internally. These primitives are the basis of socket programming.
6Hypothetical state transition: IDLE → (connect() or listen()) → ACTIVE/PASSIVE ESTABLISHMENT → ESTABLISHED → (disconnect()) → IDLE. In ESTABLISHED state: send() and recv() are allowed. In any other state: send/recv calls wait or are rejected.
7Complete end-to-end service flow: (1) Connection establishment using 3WHS → agree on ISN. (2) Flow control using sliding window ARQ → maintain delivery order and reliability. (3) Congestion control using AIMD → respect network capacity. (4) Connection closure using symmetric release with timeout → clean shutdown.
8Sender rate — full formula: Sender_rate = min(network_rate, receiver_rate). Network rate is estimated by the AIMD congestion window. Receiver rate is advertised by the receiver's window field. Both are managed simultaneously — flow control and congestion control are tightly coupled in TCP.
9Rate control over time (conceptual graph): Initially: low rate. Gradually increases (AIMD additive increase). Hits receiver rate → saturates (stops increasing). If packet loss → drops sharply (AIMD multiplicative decrease). Starts increasing again. Periodically saturates at receiver rate. Cycle continues → dynamically tracks network capacity.

Why ISN is part of the socket 6-tuple

If a machine crashes and restarts, the same {src IP, src port, dst IP, dst port} pair may be reused for a new connection. Old delayed packets with old ISNs would then corrupt the new connection's data. By including the ISN in the socket identifier, old and new connections remain distinguishable even with the same IP+port pair.

Topic 4: TCP Protocol — Design & Segment Format Most Important

What is TCP?

TCP (Transmission Control Protocol) was designed to provide reliable, end-to-end byte streaming over an unreliable internet. First specified in RFC 793 (September 1981). Today >80% of all internet traffic uses TCP. It dynamically adapts to heterogeneous paths — wired, wireless, optical — and is robust to diverse network failure modes.

Key explanation points — TCP Service Model

1Full duplex, point-to-point only: TCP is strictly between exactly 2 hosts (point-to-point). Both directions are active simultaneously. TCP was NOT designed for multicast or broadcast — sending to a group of nodes. If you need that, use UDP or special multicast protocols.
2Byte stream — not message stream: TCP does NOT preserve application-level message boundaries. Sender writing 4 × 512-byte blocks to TCP socket may have them delivered as 2048 bytes in one chunk, or 1024 + 1024, or any other division. The receiver cannot determine how the sender chunked the data. Everything is one continuous byte stream.
3Why message boundaries are not preserved: TCP collects data from the sender buffer and creates segments based on current window size and MSS — not based on application write boundaries. Retransmission may also restructure segments (e.g., two 50B+150B segments retransmitted as two 100B segments). TCP sees it as bytes, not messages.
4inetd daemon (Unix): A single process "inetd" (Internet Daemon) runs at all well-known ports simultaneously. When a connection arrives on port 80, inetd forks an httpd child process. Port 21 → ftpd. Port 22 → sshd. After forking, inetd returns to listening. Applications do not need to always be running — inetd starts them on demand.
5TCP RFC evolution: RFC 793 (Sept 1981) — original TCP specification. RFC 1122 — clarifications and bug corrections. RFC 1323 — extensions for high performance (window scaling, timestamps). RFC 2018 — SACK (Selective ACK using Selective Repeat ARQ). RFC 2581 — congestion control algorithms. RFC 3168 — Explicit Congestion Notification (ECN). TCP continues to evolve.
6Byte sequence numbers (32-bit): Every individual byte in a TCP stream has its own 32-bit sequence number. The sequence number in the segment header is the byte offset of the FIRST byte in that segment's payload. 32 bits = 2³² ≈ 4 billion unique byte positions — large enough for most connections but can wrap at very high speeds.
7Cumulative ACK (32-bit): ACK number = the next byte the receiver EXPECTS. If ACK = 31245, it means all bytes up to 31244 have been received correctly. The receiver does NOT send one ACK per segment — it sends a cumulative ACK for all bytes received so far.

Key explanation points — TCP Segment Header

8Source Port & Destination Port (16-bit each): Identify the sending and receiving applications. Together with IP addresses, they uniquely identify the connection. Well-known ports: 80 (HTTP), 21 (FTP), 22 (SSH), 25 (SMTP), 443 (HTTPS).
9Header Length: Tells where the data payload starts, because TCP header length is variable (optional fields exist). Measured in 32-bit words. Minimum header = 20 bytes (no options).
10Flag bits (critical for NPTEL): SYN — connection initiation (set in first 2 messages of 3WHS). ACK — ACK field is valid (set in all messages except first raw SYN). FIN — connection close (sender has no more data). RST — reset (forceful close, reject connection). PSH — push data to application immediately without buffering. URG — urgent data present (urgent pointer field valid).
11Window Size (16-bit): Receiver-advertised window = available buffer space at receiver. Sender window = min(cwnd, rwnd). This implements dynamic buffer management and flow control simultaneously. Maximum without scaling = 65535 bytes. Can be scaled with TCP window scaling option (RFC 1323).
12Checksum: Mandatory in TCP (unlike UDP where it's optional in IPv4). Covers the TCP header + data + a pseudoheader (source IP, destination IP, protocol, segment length). Detects bit-level corruption of the segment.
13Urgent Pointer: When URG=1, points to the last byte of urgent data within the segment. Urgent data bypasses the normal FIFO transmit queue — sent immediately without waiting. Used for out-of-band signals, e.g., Ctrl+C in a Telnet session to interrupt a running remote process.
14Segment size limits: (1) IP payload limit = 65515 bytes maximum. (2) MTU (Maximum Transmission Unit) of the particular data link technology — Ethernet, Wi-Fi, optical fiber each have different MTUs. TCP uses Path MTU Discovery (PMTUD) — a mechanism in ICMP that finds the minimum MTU along the entire end-to-end path. Sets MSS accordingly during connection setup.
15Sender window = min(cwnd, rwnd): cwnd = congestion window (how much the network can handle — set by AIMD). rwnd = receiver window (how much buffer the receiver has — advertised in each ACK). Sender always uses the minimum of both, satisfying both flow control AND congestion control simultaneously.
16Out-of-order segment handling: TCP's network layer may route different packets via different paths → segments arrive out of order. TCP receiver should BUFFER out-of-order segments rather than discarding them. When the missing segment eventually arrives, the receiver can assemble all buffered segments and deliver them in order to the application.
17Retransmission may change segment boundaries: Example: original segments are 50B + 150B (bytes 201–250 and 251–400). Both are lost. Retransmission might be 100B + 100B (bytes 201–300 and 301–400). TCP sees it as "resend bytes 201–400" — not "resend those two specific segments." Segment boundaries are NOT preserved.

Topic 5: TCP Connection Establishment — SYN & ISN Most Important

TCP 3-Way Handshake

TCP connection establishment uses a 3-way handshake with a special message called SYN (Synchronize). Both hosts exchange Initial Sequence Numbers (ISN) during this handshake. The ISN selection mechanism must be carefully designed to prevent delayed duplicate packets from corrupting new connections.

Key explanation points — TCP 3-Way Handshake

1SYN segment (Step 1): Host A sends a SYN segment (SYN flag = 1) with its chosen Initial Sequence Number x. This is the connection request. SYN = Synchronize — used to synchronize sequence numbers between the two hosts before data transfer.
2SYN+ACK segment (Step 2): Host B receives SYN → sends SYN+ACK (both SYN flag=1 AND ACK flag=1). This message: (a) Acknowledges A's SYN with ACK number = x+1. (b) Proposes B's own ISN = y for the B→A direction. This "piggybacking" (combining SYN and ACK) saves one round trip compared to sending them separately.
3ACK segment (Step 3): Host A receives SYN+ACK → verifies ACK number = x+1 (matches its SYN) → accepts as valid. Sends final ACK: sequence number = x+1, ACK number = y+1. Connection is now ESTABLISHED. Both ISNs (x for A→B, y for B→A) are confirmed. Data transfer can begin.
4Why bidirectional SYN? TCP is FULL DUPLEX — both hosts send data to each other simultaneously. A→B direction needs ISN x. B→A direction needs ISN y. If only A sent SYN, only A's ISN would be established. The SYN+ACK from B establishes B's ISN y for the reverse direction.

Key explanation points — ISN Selection

5Why ISN selection matters: If a machine crashes and restarts, it reuses the same source IP and port. Old in-flight packets from the crashed connection arrive later. If the new connection uses overlapping sequence numbers, old packets corrupt new connection's data. ISN must be chosen to avoid overlap with previous connection's forbidden region.
6Two strategies to avoid forbidden region: (a) Time-based: wait for duration T (max packet lifetime) before starting the new connection — all old packets are guaranteed dead. (b) Sequence-number-based: choose new ISN high enough above the last sequence number used in the old connection — no overlap in sequence number space. TCP uses strategy (b).
7Original TCP — clock-based ISN: Clock ticks every 4 microseconds. Each tick generates a new sequence number. Clock cycles from 0 to 2³²−1. When new connection starts, ISN = current clock value → always higher than previous connection's last sequence number → no forbidden region overlap guaranteed.
8Problem with clock-based ISN — predictability: Clock ticks at exactly 4 μs → ISN is deterministic and predictable. An attacker who knows the clock rate can compute the current clock value and predict the next connection's ISN. This enables a SYN flood attack.
9SYN Flood Attack (DoS): Attacker sends thousands of SYN packets with spoofed source IPs. Server responds with SYN+ACK to fake IPs (which never respond). Server allocates memory for each "half-open" connection waiting for the third ACK. Eventually server's memory for half-open connections is exhausted → cannot handle legitimate connections → Denial of Service (DoS).
10Modern TCP — cryptographic ISN: Clock gives base value x. Apply a cryptographic hash function (e.g., MD5/HMAC) to generate final ISN y, where y > x. Since hash output is unpredictable (cannot be computed from clock value without the secret key), the attacker cannot guess y even knowing the clock rate. Still ensures y is above the forbidden region.
11Simultaneous open: Both client and server call connect() at the same time → both send SYN → both receive SYN (not SYN+ACK). Both then send SYN+ACK. Both move to SYN_RECEIVED state → both eventually move to ESTABLISHED. This rare edge case is handled correctly by TCP.

Topic 6: TCP Connection Release & State Machine Most Important

TCP Connection Release (FIN)

TCP uses a symmetric (half-close) release. Each side closes its own half independently using a FIN (finish) segment. The side that initiates closing is the "active closer." The other side is the "passive closer." The active closer enters TIME_WAIT to handle delayed packets before fully closing.

Key explanation points — Connection Release

1Active Close (FIN): Host A wants to stop sending → sends FIN segment (FIN flag=1) with current seq and ACK numbers. This closes the A→B half of the connection. FIN = "I have no more data to send."
2Passive Close options: Host B receives FIN → Option 1: Sends ACK+FIN immediately (FIN+ACK combined) — means B also has nothing more to send. Option 2: Sends only ACK (not FIN yet) — if B still has data to send to A. B continues sending data. When B finishes, it sends its own FIN.
3Final ACK + TIME_WAIT: Host A receives B's FIN → sends final ACK → enters TIME_WAIT state. TIME_WAIT duration = 2 × MSL (Maximum Segment Lifetime). During TIME_WAIT, A can still receive delayed packets. After 2×MSL, all delayed packets are guaranteed dead → A fully closes.
4Why active closer gets TIME_WAIT but passive closer doesn't: Active closer (A) may still receive delayed segments from B that were in-flight when A sent FIN. TIME_WAIT lets A receive them gracefully. Passive closer (B): B knows A sent FIN → A will definitely not send more data. B doesn't need to wait.
5Simultaneous close: Both A and B call close() at the same time → both send FIN. Both receive FIN from the other side (not ACK). Both send ACK for the received FIN → both enter CLOSING state → both wait for ACK of their own FIN → both enter TIME_WAIT → both eventually CLOSED.

TCP State Machine — all 11 states

6CLOSED: Initial and final state. No TCP socket exists. Both client and server start here before any connection is initiated.
7LISTEN: Server calls listen() on a port. Ready to accept incoming SYN. Server waits here for clients to connect. (inetd keeps well-known ports in LISTEN state permanently.)
8SYN_SENT: Client calls connect() → sends SYN(x) → enters SYN_SENT. Waiting for SYN+ACK from server. If client calls close() here → moves back to CLOSED.
9SYN_RECEIVED: Server received SYN → sent SYN+ACK(y, ack=x+1) → waiting for client's final ACK. Server can call reset() here → returns to LISTEN (used to defend against SYN flood — reject suspicious SYN packets).
10ESTABLISHED: Connection fully open. Data transfer active in both directions. Both client and server reach ESTABLISHED: client on receiving SYN+ACK; server on receiving final ACK. send() and recv() are only valid in this state.
11FIN_WAIT_1: Active closer calls close() → sends FIN → enters FIN_WAIT_1. Three possible next events: (a) Gets ACK only → FIN_WAIT_2. (b) Gets FIN+ACK → TIME_WAIT. (c) Gets FIN only (simultaneous close) → CLOSING.
12FIN_WAIT_2: Received ACK for own FIN. Passive closer still sending data. Waiting for passive closer's FIN. On receiving FIN → send ACK → TIME_WAIT.
13TIME_WAIT: Received FIN, sent final ACK. Waiting for 2×MSL to expire. Purpose: (1) let delayed packets die, (2) if passive closer's FIN+ACK was lost and it retransmits, active closer can resend the final ACK. After timeout → CLOSED.
14CLOSE_WAIT: Passive closer received FIN → sent ACK. Application may still be generating data. Server waits here until its application finishes and calls close(). On close() → sends own FIN → LAST_ACK.
15LAST_ACK: Passive closer sent its own FIN. Waiting for the active closer's final ACK. On receiving final ACK → CLOSED. No timeout needed — passive closer knows active closer will definitely close (it already sent FIN).
16CLOSING: Simultaneous close state. Both sent FIN at the same time. Client gets FIN from server instead of ACK → sends ACK → CLOSING. Waiting for ACK of its own FIN → TIME_WAIT → CLOSED.

Absolute must-know facts

  • BDP = Bandwidth × One-way delay. RTT ≈ 2 × one-way delay
  • Optimal window W = 2×BD + 1 (fills data pipe + ACK pipe + 1 just-received ACK)
  • If BDP < segment size → use Stop-and-Wait (no benefit from sliding window)
  • Write call (sender) blocks when buffer FULL; read call (receiver) blocks when buffer EMPTY
  • Preferred buffer type: Single large circular buffer (per connection)
  • Deadlock prevention: receiver sends periodic duplicate ACKs after timeout if window=0
  • Packet loss in wired networks = router buffer overflow (λ > μ) = congestion signal
  • Sender_rate = min(network_rate, receiver_rate) — both flow control and congestion control apply
  • AIMD proposed by Chiu and Jain (1989) — additive increase, multiplicative decrease
  • AIMD converges to optimal fair+efficient point. AIAD and MIMD do NOT
  • Max-Min Fairness: cannot increase one flow without decreasing another
  • Socket = 6-tuple: {src IP, src port, src ISN, dst IP, dst port, dst ISN}
  • TCP = byte stream, NOT message stream. Message boundaries NOT preserved
  • TCP ACK is CUMULATIVE: ACK=N means all bytes up to N−1 received; expecting byte N
  • TCP does NOT support multicast — point-to-point only
  • TCP ISN originally clock-based (4 μs tick) → predictable → SYN flood attack
  • Modern TCP: cryptographic hash for ISN → unpredictable → prevents SYN flood
  • 3WHS: SYN(x) → SYN+ACK(y, ack=x+1) → ACK(ack=y+1)
  • Sender window = min(cwnd, rwnd). cwnd = congestion window; rwnd = receiver window
  • Active close → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT (2×MSL) → CLOSED
  • Passive close → CLOSE_WAIT → LAST_ACK → CLOSED (no timeout needed)
  • Simultaneous close → CLOSING → TIME_WAIT → CLOSED
  • TCP SACK (RFC 2018) uses Selective Repeat. Standard TCP uses Go-Back-N (cumulative ACK)

Key formulas — BDP

BDP= Bandwidth × One-way propagation delay (number of frames/bytes in pipe)
RTT≈ 2 × one-way propagation delay (no congestion assumed)
W_opt= 2 × BD + 1 (BD = BD in units of segments)
GBNMax window = 2ⁿ − 1 (n = bits in sequence number)
SRMax window = 2ⁿ / 2 (n = bits in sequence number)
SWSender window = min(cwnd, rwnd)
S_rate= min(network_supported_rate, receiver_advertised_rate)

AIMD algorithm — key points

AINo congestion: w(t+1) = w(t) + a (additive increase — linear growth)
MDCongestion detected (packet loss): w(t+1) = b × w(t), 0 < b < 1 (multiplicative decrease — sharp drop)
AIADOscillates on efficiency line — never reaches fairness point
MIMDOscillates radially — never reaches optimal point
AIMDSpirals to intersection of efficiency line (x+y=C) and fairness line (x=y) = OPTIMAL

TCP header flags — use cases

SYNConnection initiation — set in first 2 messages of 3WHS
ACKACK number field is valid — set in all messages EXCEPT the very first SYN
FINSender has no more data — initiates graceful half-close
RSTForceful reset — reject connection, recover from error (server uses in SYN_RECEIVED to return to LISTEN)
PSHPush — deliver data to application immediately, don't buffer
URGUrgent — urgent pointer valid; bypass FIFO queue (e.g., Ctrl+C in Telnet)

Tricky NPTEL MCQ traps — don't get these wrong

1BDP < segment: use Stop-and-Wait. Never sliding window just because network is fast
2W_opt = 2×BD+1 not 2×BD (the +1 matters)
3GBN max window = 2ⁿ−1. SR max window = 2ⁿ/2. NOT the same!
4TCP uses byte sequence numbers — NOT packet sequence numbers
5ACK number = NEXT EXPECTED BYTE (not number of bytes received)
6TCP does NOT support multicast — strictly point-to-point
7Message boundaries are NOT preserved in TCP (it is a byte stream)
8TCP window = 16 bits → max rwnd = 65535 bytes (without scaling)
9ISN: original = clock (4 μs ticks, predictable). Modern = clock + cryptographic hash (unpredictable)
10TIME_WAIT = active closer. CLOSE_WAIT = passive closer. Don't confuse
11CLOSING state = simultaneous close only (both send FIN at same time)
12AIMD: Chiu & Jain (1989). Memorise this name and year
13Socket = 6-tuple (includes ISNs). 4-tuple (without ISNs) is NOT complete
14Sender window = min(cwnd, rwnd). Not max — the MINIMUM of both

Delayed Duplicate Problem (VERY IMPORTANT)

In packet switching networks, packets may be delayed due to congestion, leading to duplicate delivery.

  • Server may receive same request twice
  • Cannot distinguish:
    • New request
    • Old delayed duplicate
  • Occurs due to retransmission & buffering

Sequence Number Mechanism

  • Each byte/segment is assigned a unique sequence number
  • Used for:
    • Ordering data
    • Detecting duplicates
    • Ensuring reliability
  • Receiver identifies duplicates using sequence numbers

Forbidden Region Concept

  • Each packet has lifetime = T
  • Same sequence number must NOT repeat within T
  • If reused → confusion between old & new packets

Initial Sequence Number Selection

  • Must avoid overlap with old sequence numbers
  • Methods:
    • Wait for time T
    • Use higher sequence number
  • TCP uses clock-based mechanism

Packet Switching Network Issues

  • Packet loss (buffer overflow)
  • Packet corruption
  • Delay (congestion)
  • Duplicate packets

Why Simple Handshake Fails?

  • Packets may be delayed
  • Duplicate messages confuse server
  • Client crash creates ambiguity

Reliable Data Transfer (RDT)

  • Network layer → Unreliable
  • Transport layer → Reliable
  • Mechanisms:
    • ACK (Acknowledgement)
    • Retransmission
