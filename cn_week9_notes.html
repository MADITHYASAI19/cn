Topic 1: IP Router Design & Lifecycle Exam Favorite

What is a Router?

An IP router is a Layer 3 device that forwards packets from one network to another by examining the IP destination address and looking up the routing table. A router sits at the boundary between different networks (subnets or autonomous systems) and makes per-hop forwarding decisions.

Router Lifecycle — Key Stages

1Packet Arrival: Packet arrives on an input port (line card). The header is parsed and the IP destination address is extracted.
2Longest Prefix Match (LPM): The routing/forwarding table is looked up to find the best (longest matching prefix) next-hop for the destination IP address.
3Switching Fabric: The packet is passed through the internal switching fabric (crossbar, bus, or shared memory) to the correct output port.
4Output Queuing & Scheduling: At the output line card, packets are queued and scheduled (FIFO, WFQ, etc.) before being transmitted on the outgoing link.
5TTL Decrement & Checksum Update: Each router decrements the TTL (Time-To-Live) field by 1 and recomputes the IP header checksum before forwarding.

Major Components of an IP Router

Component Function Location
Input Line Cards Receive incoming packets; perform header parsing and LPM lookup Edge (input)
Forwarding Table (FIB) Stores prefix-to-next-hop mappings for fast lookups (hardware TCAM) Input line card / central
Switching Fabric Internal interconnect that moves packets from input to output ports Core
Output Line Cards Queue packets and schedule transmission on output links Edge (output)
Route Processor (RP) Runs routing protocols (OSPF, BGP), maintains RIB, updates FIB Control plane

Control Plane vs Data Plane

  • Control Plane: Runs routing protocols (OSPF, BGP, RIP). Builds and maintains the Routing Information Base (RIB). Slow, software-based. Runs on the Route Processor.
  • Data Plane (Forwarding Plane): Performs actual packet forwarding using the Forwarding Information Base (FIB). Must be extremely fast — hardware-based (ASIC/TCAM). Runs on line cards.
  • Key distinction: Control plane = "where should packets go?" | Data plane = "move this packet NOW"

RIB vs FIB — Critical Distinction

  • RIB (Routing Information Base): Complete routing table built by routing protocols. Contains all routes learned. Managed by control plane. Software-based.
  • FIB (Forwarding Information Base): Optimized subset of RIB used for actual packet forwarding. Hardware-based (TCAM). Only best routes per prefix. Consulted per-packet.
  • RIB → (best route selection) → FIB. FIB is derived from RIB but is much smaller and faster to look up.

5 Generations of Router Evolution

Router Technology Timeline

Gen 11980s — Standard Computer with Interfaces: Early routers were simply general-purpose computers (e.g., VAX or Sun workstations) running routing software with multiple network interface cards inserted. No custom hardware; all forwarding done in software by the CPU.
Gen 2Early 1990s — Dedicated Interfaces: Routers gained dedicated interface cards with some hardware offload (DMA, hardware interrupt handling), but the central route processor still did all forwarding decisions. Better throughput, still single CPU bottleneck.
Gen 3Late 1990s — Distributed Architecture: Forwarding moved onto line cards themselves (distributed forwarding). Each line card had its own processor and a local copy of the FIB. The central route processor was only used for control-plane tasks (running OSPF/BGP). Massive throughput improvement.
Gen 4Early 2000s — Rack-Based Multi-Chassis: Router chassis could be expanded across multiple physical racks interconnected by high-speed backplanes. Large ISP-grade routers (e.g., Cisco CRS-1) achieved terabit-per-second forwarding by distributing across many line cards and fabric blades.
Gen 5Current — SDN Routers: The control plane is decoupled from hardware and runs as software on commodity servers (the SDN controller). The router hardware becomes a "dumb" forwarding device controlled by OpenFlow or similar protocols. Enables software-defined, vendor-agnostic network management.

Static vs Dynamic Routing Entries

Static Routes

  • Manually added by the network administrator using commands like ip route add.
  • Do not change unless the admin manually modifies or removes them.
  • Advantages: simple, predictable, no protocol overhead.
  • Disadvantages: does not adapt to topology changes; not scalable for large networks.
  • Use cases: small networks, stub networks, specific policy overrides (e.g., force traffic for 10.0.0.0/8 through a specific gateway).

Dynamic Routes

  • Learned automatically by routing protocols: OSPF (link-state, IGP), BGP (path-vector, EGP), RIP (distance-vector, legacy IGP).
  • Routes update automatically when network topology changes (e.g., a link fails).
  • Advantages: adaptive, scalable, no manual intervention needed for topology changes.
  • Disadvantages: protocol overhead, convergence delay, configuration complexity.
  • Use cases: enterprise and ISP networks where topology changes frequently.

Linux Routing and IP Commands

Practical Reference — Linux Network Commands

Topic 2: Switching Fabric Types High Weightage

What is a Switching Fabric?

The switching fabric (also called the interconnection network) is the internal hardware mechanism that moves packets from input line cards to the correct output line cards. It is the heart of a router's data plane performance. Three primary types: Shared Memory, Bus-based, and Crossbar.

Type 1: Shared Memory Switching

Shared Memory

  • Input port writes packet to central shared memory; output port reads from it.
  • Route processor controls which memory area to read/write.
  • Bottleneck: Memory bandwidth limits the throughput — entire packet must be written AND read = 2× memory bandwidth consumed per packet.
  • Simple design, but does NOT scale to high speed (limited by memory bus speed).
  • Used in early/slower routers. Example: Cisco 2500 series.

Type 2: Bus-Based Switching

Shared Bus

  • All input and output line cards connected by a single shared bus.
  • One packet crosses the bus at a time — contention occurs when multiple inputs want to send simultaneously.
  • Only ONE transfer can happen at a time — serial access.
  • Bottleneck: Bus bandwidth is the limiting factor. All ports share the same bus bandwidth.
  • Better than shared memory for some designs, but still does not scale to very high line rates.
  • Example: Cisco Catalyst 5000 (32 Gbps bus).

Type 3: Crossbar (Crosspoint) Switching

Crossbar Switch Fabric

  • An N×N grid of crosspoints — each crosspoint can connect any input to any output.
  • Allows multiple simultaneous (non-conflicting) transfers in parallel.
  • If input i and input j want to send to DIFFERENT outputs, both can send simultaneously.
  • Head-of-Line (HOL) Blocking: A packet at the head of an input queue may block other packets behind it even if those other packets' output ports are free. This is a key limitation of input-queued crossbars.
  • SPEEDUP: Run the crossbar faster than line rate to reduce HOL blocking.
  • High performance — used in modern high-speed routers and switches.
  • Example: Cisco 12000 series GSR uses crossbar fabric.
Fabric Type Parallelism Bottleneck Scalability HOL Blocking?
Shared Memory None Memory bandwidth Low N/A
Shared Bus None (serial) Bus bandwidth Medium N/A
Crossbar High (N simultaneous) Scheduling algorithm High Yes (input-queued)

Topic 3: Queuing & Packet Scheduling Exam Favorite

Why Queuing?

Queuing occurs when packets arrive faster than the output link can transmit them. Both input queuing (at ingress) and output queuing (at egress) can occur. The scheduling discipline determines WHICH packet from the queue is sent next.

Input vs Output Queuing

Type Where Problem Solution
Input Queuing At input line cards before the fabric HOL Blocking Virtual Output Queues (VOQ); Speedup
Output Queuing At output line cards after the fabric Output contention when multiple inputs → same output Large buffers; QoS scheduling

Scheduling Algorithms

1FIFO (First-In First-Out): Packets served in strict arrival order. Simple. No priority. Tail-drop when buffer full. Used as the baseline.
2Priority Queuing: Multiple queues with priorities. High-priority queue always served first. Low-priority packets may STARVE if high-priority traffic is heavy.
3Round Robin (RR): Serve one packet from each queue in turn. Fair between flows. But does not account for different packet sizes.
4Weighted Fair Queuing (WFQ): Each flow gets a WEIGHT. Bandwidth allocated proportional to weight. Most popular QoS algorithm. Approximates bit-by-bit round robin (fluid model).
5Deficit Round Robin (DRR): Extension of RR that handles variable packet sizes fairly by maintaining a credit (deficit) counter per queue.

Buffer Sizing — Rule of Thumb

  • Traditional rule: Buffer size = RTT × Link capacity (e.g., 250ms × 10Gbps = 2.5Gb)
  • Modern rule (for many flows): Buffer = RTT × C / √N (where N = number of TCP flows)
  • Too little buffer → packet loss, throughput suffers
  • Too much buffer → excessive queuing delay (bufferbloat)

Active Queue Management (AQM)

  • Tail Drop: Drop incoming packet when buffer is full. Simple but causes TCP synchronization.
  • RED (Random Early Detection): Randomly drop packets BEFORE buffer is full, proportional to queue occupancy. Avoids global synchronization of TCP flows.
  • WRED (Weighted RED): Different drop probabilities for different traffic classes — higher-priority traffic dropped less aggressively.

Topic 4: Router Hardware — ASIC, TCAM & NetFPGA Important

Why Special Hardware?

At high line rates (10 Gbps, 100 Gbps, 1 Tbps), even the fastest CPUs cannot process packets fast enough in software. Dedicated hardware accelerators are needed to perform lookups, table searches, and forwarding at wire speed.

Key Hardware Components

1ASIC (Application-Specific Integrated Circuit): Custom silicon chips designed specifically for packet forwarding. Much faster than general-purpose CPUs. Not reprogrammable after manufacture. Used in all high-performance routers and switches.
2TCAM (Ternary Content Addressable Memory): Memory that can be searched by CONTENT rather than address. Supports three values per bit: 0, 1, or Don't Care (X). Enables longest-prefix-match lookup in a SINGLE clock cycle. Much more expensive than SRAM but orders of magnitude faster for LPM.
3NetFPGA: Field-Programmable Gate Array board designed for network research. Allows researchers to prototype high-speed router/switch functions in reconfigurable hardware. Open-source platform used by universities worldwide.
4NP (Network Processor): Programmable processor optimized for network tasks (header parsing, table lookups). Sits between general CPU and ASIC in terms of flexibility vs speed.

TCAM: How It Works

  • Conventional RAM: you give an address → get back content (one address at a time).
  • CAM (Content Addressable Memory): you give content → get back the address where it is stored. Parallel search of ALL entries simultaneously.
  • TCAM adds a "Don't Care" (X or *) bit — perfect for IP prefix matching where trailing bits are wildcards.
  • A routing table with N entries: TCAM finds the matching entry in O(1) time regardless of N.
  • Limitation: High power consumption and cost. Entries stored in priority order (highest priority = most-specific prefix first).

TCAM Entry Structure — Worked Example

Prefix + Mask → Wildcard (TCAM) Format

In TCAM, each routing prefix is stored as a prefix pattern + wildcard mask. The mask length tells how many leading bits are fixed (must match exactly); the remaining bits become "Don't Care" (X) in TCAM. A search word is broadcast to all TCAM entries simultaneously; all matching entries signal a hit, and a priority encoder selects the longest match (highest priority = most specific prefix = fewest X bits).

Prefix (Binary) Mask Length TCAM Entry (Ternary) Meaning
101 3 (all bits fixed) 101 Exact match: only 101
101 2 (last bit wildcard) 10X Matches 100 and 101
10 2 (both bits fixed) 10XX Matches 1000, 1001, 1010, 1011
(default) 0 (all wildcard) XXXX Matches any address — default route

Example search: Search word = 1011. This matches TCAM entry "10XX" (prefix 10, mask 2) and also "XXXX" (default). The priority encoder selects "10XX" because it is more specific (fewer X bits = longer prefix = higher priority). This gives O(1) longest-prefix-match in hardware.

Slotted Chassis Architecture

Physical Chassis with Insert-able Slots

Modern enterprise and carrier-grade routers use a slotted chassis design — a physical enclosure with multiple numbered slots into which individual cards are inserted like modules in a rack.

1Chassis (the case): Provides the physical backplane (high-speed bus/crossbar connecting all cards), power supplies, and cooling fans. The backplane is the internal switching fabric.
2Route Processor Card: One (or two, for redundancy) card slots hold the Route Processor — the CPU that runs OSPF, BGP, and manages the control plane. Inserted into a dedicated slot, typically slot 0.
3Interface / Line Cards: Each remaining slot holds one interface card (line card) providing physical ports (e.g., 4× 10GbE, 8× 1GbE). Cards can mix port types within the same chassis.
4Hot-Swap / In-Service Repair: A faulty line card can be physically pulled out and replaced with a new one without powering down the chassis. Other slots continue forwarding traffic — the router remains operational during repair.
5Capacity Upgrade: Adding capacity is as simple as inserting a new line card into an empty slot. No rewiring or downtime required for the existing slots. This makes chassis-based routers far more maintainable than fixed-port appliances.

MTU and Loopback Interface

MTU — Maximum Transmission Unit

  • Each network interface has a maximum packet size it can transmit in one frame, called the MTU.
  • Ethernet MTU = 1500 bytes (most common). This is the maximum IP datagram size on an Ethernet link (excluding the Ethernet frame header).
  • Loopback MTU = 65536 bytes — the loopback interface has a very large MTU because there is no physical medium limit.
  • If a packet is larger than the MTU of the outgoing interface, the router must either:
    • Fragment the packet (IP fragmentation) — split it into smaller fragments, each with its own IP header.
    • Drop the packet and send an ICMP "Fragmentation Needed" message back to the source — if the DF (Don't Fragment) bit is set in the IP header. This is how Path MTU Discovery (PMTUD) works.
  • Fragmentation is expensive (CPU overhead) and fragments must be reassembled at the destination.

Loopback Interface (127.0.0.1)

  • A virtual network interface that exists on every machine and does not correspond to any physical hardware.
  • IP address: 127.0.0.1 (IPv4), configured with the /8 prefix (entire 127.0.0.0/8 block is loopback).
  • Hostname "localhost" resolves to 127.0.0.1 — used by applications to communicate with services on the same machine without network traffic leaving the host.
  • Purpose: Test whether the local TCP/IP stack is working correctly without needing a physical network connection. If ping 127.0.0.1 works, the kernel network stack is functional.
  • Packets sent to 127.0.0.1 are looped back by the kernel immediately — they never leave the machine and never go on any physical medium.
  • MTU of loopback is 65536 bytes (no physical medium constraint).

Practical Router: Cisco IOS Router Components

  • Flash memory: Stores the IOS (operating system) image.
  • NVRAM: Stores the startup configuration file.
  • RAM: Stores running configuration and routing tables.
  • ROM: Bootstrap program (POST + ROM Monitor).
  • Interfaces: Physical ports (Ethernet, Serial, etc.) — each is a line card.

Topic 5: Software-Defined Networking & OpenFlow Exam Favorite

SDN Motivation — Why Move Away from Traditional Routing?

  • (a) Scalability limits: Distributed routing protocols (distance-vector, link-state) do not scale perfectly; convergence slows as networks grow.
  • (b) Per-device management: Policy changes (e.g., block a traffic type) require reconfiguring every router individually — tedious and error-prone at scale.
  • (c) Vendor lock-in: Each vendor has a proprietary OS (Cisco IOS, Juniper JunOS). Managing a multi-vendor network is complex; no standard API.
  • (d) Configuration inconsistencies: In a distributed system, different routers may have inconsistent views of the network momentarily, causing transient routing loops or black holes.
  • SDN solution: Centralize the control plane into a software controller. Switches become simple forwarding hardware with no routing intelligence of their own.

SDN Architecture — Three Layers

Layered SDN Model

TopApplication Layer: Network applications written by the operator — firewall, load balancer, traffic engineering, monitoring. These are normal software programs (Python, Java) that call the controller's northbound API to install/query forwarding rules.
MidControl Layer (SDN Controller): Centralized software running on a standard computer (or cluster). Collects network topology information, computes forwarding rules, and pushes them to switches via the southbound interface (OpenFlow). Acts as the "brain" of the network.
BotInfrastructure Layer (Data Plane Switches): Programmable "dumb" switches. They hold only a TCAM-based flow table and a forwarding engine. They have NO built-in routing logic — they simply execute rules installed by the controller. Cheap commodity hardware.

Closed vs Open Innovation

Traditional Closed vs SDN Open Model

Aspect Traditional (Closed) SDN (Open)
Hardware + Software Both from same vendor (e.g., Cisco IOS on Cisco ASR) Vendor supplies hardware only; operator chooses software
Control logic Built into router firmware, proprietary Runs as software on commodity server (open-source controller)
Innovation pace Slow — hardware upgrade cycle required Fast — update control software without touching hardware
Management Per-device CLI configuration Single controller manages all switches
Vendor lock-in Yes — switch hardware and OS bundled No — standard OpenFlow interface

Northbound and Southbound Interfaces

API Interfaces in SDN

  • Northbound Interface (NBI): The API between the SDN controller and the Application layer. Network programs (written in Python, Java, etc.) call the NBI to install forwarding rules, query topology, or set policies. Each SDN controller exposes its own NBI (e.g., REST API in OpenDaylight, Python API in Ryu).
  • Southbound Interface (SBI): The protocol between the controller and the physical/virtual switches in the Infrastructure layer. OpenFlow is the most widely used southbound protocol. Other options: NETCONF, gRPC, P4Runtime.
  • Think of it as: Applications talk "north" to the controller → Controller talks "south" to switches.

OpenFlow Protocol

OpenFlow — Controller-to-Switch Communication Standard

OpenFlow is the standardized protocol for an SDN controller to install, modify, and delete forwarding rules in a switch's flow table. Each entry in the flow table is a match-action pair:

  • Match fields (what to look for in packet headers): Source MAC, Destination MAC, Source IP, Destination IP, TCP/UDP source port, TCP/UDP destination port, VLAN ID, incoming switch port, Ethernet type, IP Type-of-Service (ToS).
  • Actions (what to do when a packet matches): Forward to a specific port, Drop the packet, Encapsulate and forward to controller (Packet-In), Modify a header field (e.g., rewrite MAC for NAT).
  • Statistics (maintained per rule): Packet counter, byte counter, time since last match — used for traffic monitoring and rule timeout.
  • Priority: When multiple rules match a packet, the rule with the highest priority number wins. Enables specific rules to override more general ones.

OpenFlow Flow Table Examples

Use Case Match Fields Action
Layer 2 Switching dst MAC = AA:BB:CC:DD:EE:FF Forward to port 3
Firewall (block SSH) TCP dst port = 22 Drop
IP Forwarding dst IP = 202.2.*.* (/16) Forward to port eth2
Flow-Specific Routing src IP = 10.0.0.1, dst IP = 10.0.0.2, src port = 5000, dst port = 80 Forward to switch port 4

OpenFlow Message Types

1Hello: Exchanged at connection startup. Controller and switch negotiate the OpenFlow protocol version to use.
2Features Request / Reply: Controller asks the switch what capabilities it supports (number of ports, actions supported, table sizes). Switch replies with its feature set.
3Packet-In: Switch sends an unmatched packet UP to the controller. Triggered when a packet arrives and no flow table rule matches it. The controller then decides what to do.
4Packet-Out: Controller sends a message DOWN to the switch telling it what to do with a buffered packet (forward to a port, drop, flood). Used to respond to Packet-In events before a flow rule is installed.
5Flow-Mod: Controller installs, modifies, or deletes a flow table entry in the switch. This is the core operation of SDN — pushing forwarding rules into the switch's TCAM. Types: ADD, MODIFY, DELETE.
6Echo Request / Reply: Keepalive mechanism. Controller sends Echo Request; switch responds with Echo Reply. Checks if the OpenFlow control channel is alive. If no reply, the connection is considered dead.
7Stats Request / Reply: Controller requests flow statistics (packet counts, byte counts, durations) from the switch. Used for traffic monitoring, load balancing decisions, and billing.

SDN Use Cases

What OpenFlow Match-Action Rules Can Implement

  • Firewall: Match src IP = 1.2.3.4 → Drop. Blocks specific sources without dedicated firewall hardware.
  • NAT (Network Address Translation): Match src IP = 10.x.x.x → Modify src IP to public IP and forward. Rewrites headers for address translation.
  • Flow-specific routing: Route packets from a specific application (identified by 5-tuple: src IP, dst IP, src port, dst port, protocol) along a chosen path through the network.
  • VLAN switching: Match VLAN ID + dst MAC → Forward to specific port. Implements Layer 2 switching with VLAN isolation.
  • Load balancer: Match dst IP = virtual IP → Modify dst IP to one of several real server IPs (round-robin or hash-based), distributing traffic across multiple servers.

Topic 6: Wireshark & Mininet Important

Wireshark — Packet Capture and Analysis Tool

  • What it does: Captures live network packets on a selected physical or virtual interface and displays them in real time.
  • Protocol stack display: Each captured packet is decoded and shown with its full protocol stack breakdown — Ethernet → IP → TCP → Application layer. Every header field is visible and expandable.
  • Fields visible at each layer:
    • Ethernet: Source MAC, Destination MAC, EtherType
    • IP: Source IP, Destination IP, TTL, Protocol, Header checksum, Fragment offset, DF bit
    • TCP: Source port, Destination port, Sequence number, Acknowledgement number, Flags (SYN/ACK/FIN/RST/PSH), Window size, Checksum
  • TCP three-way handshake visibility: Visible as three consecutive packets: (1) SYN [client → server], (2) SYN-ACK [server → client], (3) ACK [client → server]. Sequence and acknowledgement numbers increment visibly.
  • Filter syntax examples:
    • tcp — show only TCP packets
    • http — show only HTTP traffic
    • ip.addr == 10.0.0.1 — show packets involving 10.0.0.1
    • tcp.port == 22 — show SSH traffic
    • icmp — show ping packets

Mininet — Network Emulator

Mininet Architecture and Internals

  • What it does: Creates an entire virtual network inside a single Linux machine, with real hosts, switches, links, and an SDN controller — all simulated in software.
  • Virtual Hosts: Implemented as Linux network namespaces. Each namespace has its own TCP/IP stack (routing table, ARP table, sockets). This is NOT a VM — it is a lightweight OS-level isolation mechanism.
  • Virtual Switches: Implemented using Open vSwitch (OVS) — an open-source, OpenFlow-capable software switch. OVS runs partly in the Linux kernel (fast path) and partly in user space.
  • Virtual Links: Implemented using veth (virtual Ethernet) pairs. A veth pair is like a virtual cable — packets written to one end appear on the other end. One end is attached to a host namespace, the other to OVS.
  • Advantage over pure simulation: Mininet uses the REAL Linux kernel TCP/IP stack, so experiments produce realistic results (real TCP congestion control, real ARP, real routing). Pure simulators use simplified models.

Mininet Topology Commands

1mn --topo single,3 — Creates one OVS switch with 3 hosts connected to it in a star topology. Hosts are h1, h2, h3; switch is s1. All hosts share the single switch.
2mn --topo linear,3 — Creates 3 switches (s1–s2–s3) in a linear chain, each with 1 host (h1 on s1, h2 on s2, h3 on s3). Packets from h1 to h3 traverse two switches.
3mn --topo linear,2,3 — Creates 2 switches, each with 3 hosts attached. Total: 6 hosts, 2 switches, 1 inter-switch link.
4--controller remote — Connects all OVS switches to an external SDN controller (e.g., Ryu running on the same machine at 127.0.0.1:6633) instead of the built-in Mininet reference controller.
5--switch ovsk — Specifies Open vSwitch kernel module as the switch implementation. The kernel module provides faster packet processing than the user-space OVS daemon.
6h1 ping h2 (inside Mininet CLI) — Runs the ping command from host h1 to h2's IP address. Tests connectivity and measures RTT. The first ping may take noticeably longer due to SDN first-packet delay.

First-Packet Delay in SDN Networks

Why the First Packet Is Slower

In a Mininet demo with a Ryu controller, pinging from h1 to h2 shows a pattern like:

  • Ping 1: 4.84 ms (much higher than expected)
  • Ping 2–5: 0.02 – 0.16 ms (fast)

Reason for first-packet delay:

  1. First ICMP packet from h1 arrives at OVS switch s1.
  2. No matching flow rule exists yet → OVS generates a Packet-In → sends packet to Ryu controller.
  3. Ryu controller processes the Packet-In: learns MAC of h1, looks up MAC of h2, determines output port.
  4. Ryu sends a Flow-Mod to install a bidirectional flow rule in OVS.
  5. Ryu sends a Packet-Out to forward this specific packet to h2.
  6. All subsequent ICMP packets hit the installed TCAM flow rule → forwarded at hardware speed, no controller involvement → sub-millisecond RTT.

Open vSwitch (OVS)

Open vSwitch — Software SDN Switch

  • Definition: An open-source, production-quality implementation of a multilayer virtual network switch.
  • OpenFlow support: Fully supports OpenFlow 1.0 through 1.3 — can be controlled by any OpenFlow controller (Ryu, POX, OpenDaylight, ONOS).
  • Kernel module: OVS has a kernel module (openvswitch.ko) for the fast path — packets matching existing flow rules are forwarded entirely within the kernel without context switching.
  • Use in Mininet: OVS is the default switch type used by Mininet. The --switch ovsk flag selects the kernel module variant.
  • Production use: OVS is used in major cloud platforms — OpenStack (Neutron), VMware NSX, Kubernetes CNI plugins. Not just a research tool.

simple_switch_13.py — Ryu Learning Switch Controller

How the simple_switch_13.py Controller Works

This Ryu application implements Layer 2 MAC address learning — the same behavior as a traditional Ethernet switch, but implemented in software on the SDN controller.

1MAC Table: The controller maintains a dictionary mapping (dpid, MAC address) → output port. Initially empty.
2packet_in_handler: Called when a Packet-In event arrives. Extracts the source MAC and the incoming port from the frame. Stores src_mac → in_port in the MAC table (learning).
3Lookup: Looks up the destination MAC in the MAC table. If found → set out_port to the known port. If NOT found → set out_port = FLOOD (send to all ports except incoming).
4add_flow: If out_port is known (not flood), installs a Flow-Mod rule in the OVS switch: "match dst_mac = X → forward to port P". Future packets for this destination are forwarded at hardware speed without involving the controller.
5Packet-Out: Forwards the current (first) packet to the determined output port, either directly via Packet-Out or by letting the newly installed flow rule handle it.

MOST FREQUENTLY ASKED — Memorize These

  • Control Plane = routing protocols (OSPF, BGP) → builds RIB → slow, software
  • Data Plane = packet forwarding using FIB → fast, hardware (ASIC/TCAM)
  • RIB = all routes from all protocols; FIB = best routes only for line-speed lookup
  • Router decrements TTL by 1 at each hop; packet discarded if TTL = 0
  • Switching fabrics: Shared Memory → Bus → Crossbar (increasing performance)
  • HOL Blocking = crossbar limitation; packet at head of input queue blocks others
  • TCAM = searches ALL entries in O(1) by content — perfect for LPM; ternary = 0/1/X (don't care)
  • Cisco router memory: Flash=IOS, NVRAM=startup config, RAM=running config, ROM=bootstrap
  • SDN 3 layers: Infrastructure (dumb switches) → Control (SDN controller) → Application (network apps)
  • OpenFlow = match-action rules in TCAM flow table; match on 10 fields; actions = forward/drop/modify
  • Key OpenFlow messages: Packet-In (switch→controller, no match), Flow-Mod (controller→switch, install rule), Packet-Out (controller→switch, handle buffered packet)
  • Controller examples: Ryu (Python, lightweight), OpenDaylight (Java, enterprise), ONOS (Java, carrier-grade)
  • Southbound = OpenFlow (controller ↔ switches); Northbound = REST/Python API (apps ↔ controller)
  • First-packet SDN delay: caused by Packet-In → controller → Flow-Mod round trip; subsequent packets hit TCAM at hardware speed
  • Mininet = Linux network namespaces as hosts + Open vSwitch + veth pairs; uses real kernel TCP/IP stack

Scheduling Algorithm Summary

→FIFO: Simple, no priority, tail-drop. Baseline scheduler.
→Priority Queuing: High priority always first. Risk of starvation for low-priority.
→Round Robin: Serve one packet per queue in turn. Fair but ignores packet size.
→WFQ: Weighted fair sharing of bandwidth. Most popular QoS mechanism. No starvation.
→RED: Drop randomly before buffer fills → avoids global TCP synchronization.

Switching Fabric Comparison (One-Line Each)

→Shared Memory: Central memory, CPU controls, bottleneck = memory bus, low scalability.
→Shared Bus: Single bus, serial transfers, bottleneck = bus bandwidth, medium scalability.
→Crossbar: N×N grid, parallel transfers, HOL blocking, high scalability, used in modern routers.

SDN Quick Reference

→Packet-In: switch → controller (no matching rule found)
→Packet-Out: controller → switch (handle buffered packet, one-shot)
→Flow-Mod: controller → switch (install/modify/delete TCAM flow rule)
→Echo Req/Reply: keepalive on OpenFlow control channel
→Ryu simple_switch: learns MACs, installs flow rules for known dst, floods for unknown dst

Exam Strategy — Week 9

  • If asked about "where routing protocols run" → Control Plane / Route Processor
  • If asked about "where actual packet forwarding happens" → Data Plane / Line Cards / FIB
  • HOL Blocking is a crossbar-specific problem — do not say it affects bus or shared memory
  • TCAM key advantage = O(1) parallel search; key disadvantage = expensive, high power
  • WFQ provides QoS WITHOUT starvation — priority queuing CAN cause starvation
  • SDN switch = dumb TCAM + forwarding engine; ALL intelligence is in the controller
  • Packet-In goes UP (switch to controller); Flow-Mod goes DOWN (controller to switch)
  • First-packet SDN delay = controller round trip; not a bug, a fundamental SDN property
