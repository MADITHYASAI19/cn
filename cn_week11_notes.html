Topic 1: ARP & RARP — Address Resolution Most Important

The Address Resolution Problem

IP (Layer 3) operates with IP addresses. Ethernet/Wi-Fi (Layer 2) operates with MAC addresses. To send a packet, the sender needs the NEXT HOP's MAC address, not its IP address. ARP solves: "I know the IP address — what is the MAC address?"

ARP — Step-by-Step Working

1Purpose: Maps an IP address (Layer 3) to a MAC address (Layer 2) on the same local network segment.
2Check ARP Cache: The sender first checks its local ARP table for a cached IP→MAC mapping. If found and not expired, use it directly — no broadcast needed.
3ARP Request (Cache Miss): Broadcast frame (MAC: FF:FF:FF:FF:FF:FF, EtherType: 0x0806). "Who has IP address X.X.X.X? Tell me your MAC." All stations on the LAN receive this frame.
4ARP Reply: Unicast frame sent ONLY to the requester. "IP X.X.X.X is at MAC AA:BB:CC:DD:EE:FF." Only the host with that IP replies. All others ignore it.
5ARP Cache Update: Requester stores the new IP→MAC mapping in its ARP table with a TTL (typically 20 minutes). Saves repeated broadcasts for the same destination.
6Cross-layer position: ARP sits between Layer 2 (Data Link) and Layer 3 (Network). ARP packets are encapsulated directly in Ethernet frames (EtherType = 0x0806). ARP does NOT use IP headers.
7Gratuitous ARP: A host sends an ARP request for its OWN IP address. Used to: (a) detect IP conflicts — if someone replies, there's a conflict; (b) update other hosts' ARP caches when a MAC changes (e.g., NIC replacement); (c) used in HSRP/VRRP failover to announce new active router.

ARP Packet Format — Key Fields

  • Hardware Type: 1 = Ethernet
  • Protocol Type: 0x0800 = IPv4
  • Hardware Address Length: 6 (MAC = 6 bytes)
  • Protocol Address Length: 4 (IP = 4 bytes)
  • Operation: 1 = ARP Request; 2 = ARP Reply
  • Sender MAC and IP; Target MAC and IP
  • In ARP Request: Target MAC field = 00:00:00:00:00:00 (unknown — that's what we're asking for)

RARP — Reverse ARP

RARP: From MAC → IP Address

  • RARP solves the REVERSE problem: "I know my MAC address — what is my IP address?"
  • Used by: Diskless workstations that do NOT have persistent storage (no hard disk) to remember their IP address. On every boot, they ask the network for their IP using their MAC address as the only identity.
  • A RARP server on the network maintains a static table of MAC→IP mappings (configured by admin).
  • Client broadcasts a RARP request with its own MAC address as the target MAC.
  • RARP server looks up the MAC in its table and responds with the IP address.
  • Limitations: (1) Requires a RARP server on EACH subnet — RARP is broadcast-based, cannot cross routers. (2) Provides IP address only — no subnet mask, default gateway, or DNS info. (3) Static configuration — admin must manually enter each MAC→IP binding.
  • RARP is now entirely obsolete — replaced by BOOTP, then DHCP.

Topic 2: BOOTP & DHCP — Dynamic IP Configuration Very Important

The Problem: Automatic Network Configuration

When a new host joins a network, it needs: IP address, Subnet mask, Default gateway (router IP), DNS server IP, domain name, and more. Manually configuring each host is impractical in large networks. BOOTP and DHCP automate this process.

BOOTP — Bootstrap Protocol

BOOTP (RFC 951)

  • An improvement over RARP — provides more information: IP address, subnet mask, gateway IP, and a server filename for booting (TFTP boot file).
  • Uses UDP (port 67 for server, port 68 for client). Can cross routers via BOOTP Relay Agents — a major advantage over RARP.
  • Client sends broadcast request; BOOTP server responds with configuration.
  • Limitation: BOOTP bindings are STATIC — MAC→IP mapping must be pre-configured by admin. No dynamic/temporary allocation. Cannot reclaim and reuse IP addresses when a device leaves the network.
  • BOOTP was widely used for diskless workstations and network booting (PXE boot still uses BOOTP/DHCP protocol messages).

DHCP — Dynamic Host Configuration Protocol

DHCP (RFC 2131) — The Current Standard

  • An extension of BOOTP — backward compatible. Uses same ports (UDP 67/68).
  • Provides DYNAMIC IP address allocation with a configurable LEASE TIME — the IP is borrowed for a specific duration.
  • After lease expires, client must renew or release the IP. The IP pool (scope) is reused efficiently as devices join and leave the network.
  • Provides: IP address, Subnet mask, Default gateway, DNS server IPs, lease duration, domain name, NTP server, and more via DHCP options.
  • DHCP Scope: A configured pool (range) of IP addresses on the DHCP server that it is allowed to assign. For example: 192.168.1.100 – 192.168.1.200. The scope defines the available pool and can include exclusions (reserved addresses).
  • Lease Concept: Like renting an IP address — the client uses it for the lease duration (e.g., 24 hours). Before expiry, the client can renew. After expiry without renewal, the IP returns to the pool for reassignment.

DHCP DORA Process — 4-Step Handshake Frequently Asked

D-O-R-A: Discover → Offer → Request → Acknowledge

DDHCPDISCOVER: Client has no IP. Broadcasts to 255.255.255.255 (src IP = 0.0.0.0). "Is there a DHCP server? I need an IP address." All DHCP servers on the subnet (and via relay agents) receive this broadcast.
ODHCPOFFER: One or more DHCP servers respond with an offer: proposed IP address, lease time, subnet mask, default gateway, DNS servers. May be broadcast (if client has no IP yet) or unicast if client set the broadcast flag to 0. Server tentatively reserves the offered IP.
RDHCPREQUEST: Client broadcasts its acceptance of ONE server's offer (even if multiple servers replied). This broadcast tells all servers which offer was accepted — declined servers release their reserved IPs. Client also echoes back the offered IP to allow the server to verify.
ADHCPACK: Winning server confirms the allocation with an acknowledgment containing the final configuration. Client now officially has its IP and all network parameters and can begin communicating. Lease timer starts.

DHCP Lease Renewal Process

  • At 50% of lease time (T1): Client unicasts DHCPREQUEST to the original server to renew. If server responds with DHCPACK, lease is renewed. This is the preferred renewal path.
  • At 87.5% of lease time (T2): If the original server did not respond at T1, client now broadcasts DHCPREQUEST to ANY available DHCP server. This allows failover to a backup server.
  • At 100% expiry: Client has NOT renewed successfully. Client loses its IP address and must restart the entire DORA process from DHCPDISCOVER.
  • DHCPNAK: Server tells client the requested IP is invalid (e.g., client moved to a new subnet). Client must restart DORA.
  • DHCPRELEASE: Client voluntarily releases its IP before lease expires (e.g., when shutting down gracefully). IP immediately returns to the pool.
  • DHCPINFORM: Client already has a static IP but wants other config parameters (gateway, DNS) from the DHCP server.
  • DHCP Relay Agent: A router configured to forward DHCP broadcasts to DHCP server on another subnet, converting broadcast to unicast. Allows ONE DHCP server to serve MANY subnets.

Topic 3: Proxy ARP & Gratuitous ARP Important

What is Proxy ARP?

Proxy ARP is a technique where a router responds to ARP requests on behalf of a host on a DIFFERENT subnet. The router "pretends" to be the target by replying with its OWN MAC address. The sending host then sends frames to the router, which forwards the packet to the actual destination.

Proxy ARP — Detailed Operation

1When Used: Host A (192.168.1.10) wants to reach Host B (192.168.2.20). Host A incorrectly believes both are on the same subnet (wrong subnet mask configured). Host A sends an ARP request for 192.168.2.20.
2Router Intervenes: The router between the two subnets receives the ARP request. The router knows it has a route to 192.168.2.20. It replies with its OWN MAC address in the ARP reply.
3Host A's Behavior: Host A thinks it found the MAC of 192.168.2.20 (actually got the router's MAC). Host A sends frames to the router. The router routes the packet to the correct subnet and Host B.
4Advantages: Works without any reconfiguration of hosts. Useful when hosts are misconfigured with wrong subnet mask. Transparent to applications.
5Disadvantages: Increases ARP table size on hosts. Router must answer EVERY ARP request — overhead. Does not scale well. Security risk: can mask configuration errors that should be corrected.

Gratuitous ARP — Self-Announcement

  • A host sends an ARP request for its OWN IP address (sender IP = target IP).
  • No one should reply (the IP should be unique). If someone replies → IP address conflict detected!
  • Use cases: IP conflict detection at boot, announcing MAC change (NIC replacement), HSRP/VRRP failover, load balancer failover.

ARP Poisoning / ARP Spoofing — Security Issue

  • ARP has NO authentication. Any host can send fake ARP replies.
  • Attacker sends unsolicited ARP replies mapping a legitimate IP to the attacker's MAC address.
  • Other hosts update their ARP caches with the poisoned entry → man-in-the-middle attack.
  • Solution: Dynamic ARP Inspection (DAI) on switches — validates ARP packets against DHCP snooping binding table.

Topic 4: Layer 2 Switching & STP Exam Favorite

Layer 2 Switch Operation

An Ethernet switch forwards frames based on MAC addresses. Unlike a hub (which broadcasts to all ports), a switch selectively sends frames only to the port where the destination device is connected. Switches maintain a MAC Address Table (CAM table) to make forwarding decisions.

Switch MAC Table Operation

1Learning (Source Learning): When a frame arrives, the switch records the SOURCE MAC address and the incoming port in the MAC table (CAM table). This builds the table dynamically over time.
2Forwarding (Destination Lookup): Switch looks up the DESTINATION MAC in the MAC table:
• Found → forward frame ONLY to the specific port (unicast forwarding).
• NOT found → FLOOD frame to ALL ports except the incoming port (unknown unicast flooding).
3Filtering: If src and dst are on the SAME port (same segment), the frame is filtered (dropped) — no need to forward.
4Aging: MAC table entries age out after a timeout (default 300 seconds = 5 minutes). If no frame is seen from a MAC within the timeout, the entry is deleted to keep the table current.

STP — Root Cause and Solution Root Cause Often Missed

Why Layer 2 Loops are Catastrophic — Root Cause

The core problem: Ethernet frames have no TTL field (unlike IP packets). A router discards a packet after its TTL reaches 0, preventing infinite loops. But an Ethernet switch has no such mechanism — it will forward a frame forever if a loop exists.

When a loop exists in a switched network, three catastrophic events happen simultaneously:

  • Broadcast Storm: A single broadcast frame (e.g., ARP request) enters a loop. Each switch forwards it out all ports → the frame traverses the loop → re-enters the switches → forwarded again → the frame multiplies exponentially. Within seconds, thousands of copies flood the network, consuming 100% of bandwidth. The network becomes completely unusable.
  • Duplicate Frame Delivery: A unicast frame loops around and arrives at the destination MULTIPLE TIMES. The upper-layer protocol (IP, TCP) must deal with the duplicates — wasting processing power and potentially causing application errors.
  • MAC Table Instability (MAC Flapping): The same source MAC address keeps arriving on different ports as frames loop. The switch continuously updates its MAC table with conflicting port information for the same MAC — the table oscillates, making forwarding decisions unreliable. Also called "MAC flapping."

Why loops are introduced: Network admins add redundant physical links between switches for high availability — if one link fails, the other keeps the network up. But these redundant links create physical loops if not managed.

STP — Spanning Tree Protocol (IEEE 802.1D) — How It Solves Loops

  • STP Solution: STP logically BLOCKS redundant ports to create a loop-free spanning TREE topology, while keeping the redundant links physically connected. If the active path fails, STP unblocks the redundant path (failover).
  • Root Bridge Election: The switch with the lowest Bridge ID (priority field + MAC address) is elected as the root bridge. All topology decisions are made relative to the root bridge. Default priority = 32768; can be manually set lower to force a specific switch to be root.
  • Port Roles:
    • Root Port: The port on each non-root switch with the lowest cost path to the root bridge. One per switch.
    • Designated Port: The forwarding port on each network segment (link). One per segment.
    • Blocked Port: All other ports — logically disabled to prevent loops. Still receives BPDUs.
  • Port States (STP Convergence): Blocking → Listening → Learning → Forwarding → Disabled. Default convergence time: 30–50 seconds (too slow for modern networks).
  • RSTP (IEEE 802.1w): Rapid STP — converges in under 1 second. Modern standard. Uses Proposal/Agreement mechanism instead of fixed timers.
  • BPDU (Bridge Protocol Data Unit): STP control messages exchanged between switches to elect root bridge, determine port roles, and detect topology changes.

Topic 5: VLANs — Virtual Local Area Networks High Weightage

What is a VLAN?

A VLAN (Virtual LAN) is a logical grouping of network devices that behaves as if they are on the same physical network segment, regardless of their actual physical location. VLANs allow a single physical switch to be divided into multiple isolated logical broadcast domains.

Why VLANs? — Benefits

1Broadcast Domain Control: Each VLAN is its own broadcast domain. Broadcasts are confined within the VLAN — reduces unnecessary broadcast traffic across the entire network.
2Security (Traffic Isolation): Devices in different VLANs cannot communicate directly — they must go through a router (Layer 3). This provides logical separation for sensitive departments (Finance, HR, R&D) even on the same physical switch.
3Flexibility: A department can span multiple physical switches, or multiple departments can share a physical switch — all without physical rewiring. VLANs follow the user, not the port.
4Performance: Smaller broadcast domains → less broadcast traffic → better performance for each VLAN group.

VLAN Host Assignment — Two-Step Requirement Important Concept

Correctly Assigning a Host to a VLAN

Assigning a host to a VLAN correctly requires TWO things to both be correct simultaneously:

1Switch Port Assignment: The switch port that the host is physically connected to must be configured as an access port for the correct VLAN (e.g., switchport access vlan 20). This places the host's traffic in VLAN 20 at Layer 2.
2IP Subnet Configuration: The host's IP address must be within the IP subnet assigned to that VLAN. For example, VLAN 20 might use network 192.168.20.0/24. The host needs an IP in this range (e.g., 192.168.20.10) and the correct default gateway (the VLAN 20 SVI IP, e.g., 192.168.20.1).

Why BOTH are required:

  • Wrong switch port VLAN, correct IP: Host's frames arrive at the switch tagged with the wrong VLAN. The Layer 2 forwarding is wrong — frames go to the wrong broadcast domain. Even though the IP is technically correct, the host cannot communicate with its intended VLAN peers.
  • Correct switch port VLAN, wrong IP subnet: Host is in the right Layer 2 VLAN, but its IP address is from the wrong subnet. It cannot communicate with other hosts in the VLAN at Layer 3. Routing fails even within the VLAN. ARPs work at L2 but IP routing breaks.
  • Example: Host connected to port assigned to VLAN 20, but host configured with 192.168.10.5 (VLAN 10's subnet). The host sends traffic into VLAN 20's broadcast domain but thinks it belongs to VLAN 10's IP network — all communications fail.

VLAN Types / Deployment Models

End-to-End VLAN (Campus-Wide VLAN)

  • Definition: A VLAN that spans the ENTIRE campus network — from access layer switches all the way through distribution and core switches to the other side of the campus.
  • Use case: Users who frequently move between buildings but need to remain in the same VLAN (e.g., a company where Finance department users may sit in any building but always need VLAN 30).
  • Flexibility: Very high — VLAN follows the user wherever they plug in across the campus.
  • Complexity: High — requires consistent VLAN configuration on ALL switches throughout the campus. Trunk links must carry the VLAN everywhere. Difficult to manage in large deployments.
  • Broadcast domain: Can be very large, spanning the entire campus. This is a drawback — broadcast traffic propagates far and wide.

Location-Based VLAN (Local VLAN)

  • Definition: A VLAN that is confined to a specific geographic location (building, floor, or wiring closet). Users in a given location share a VLAN; users in another location use a different VLAN. VLANs are local — they do NOT span the entire campus.
  • Use case: Most modern enterprise deployments follow this model. Users are assigned to VLANs based on where they are physically located, not their organizational group. When a user moves buildings, they get a new IP address/VLAN.
  • Flexibility: Lower — user must reconfigure (or use DHCP) when moving locations. VLAN does NOT follow the user.
  • Complexity: Lower — VLANs are simpler to manage, smaller broadcast domains, fewer VLANs needed across the network core.
  • Broadcast domain: Small — limited to the local segment. Better performance.

IEEE 802.1Q — VLAN Tagging Standard

VLAN Tag Structure

  • 802.1Q inserts a 4-byte VLAN TAG into the Ethernet frame header between the Source MAC and EtherType fields.
  • VLAN Tag Format (4 bytes total):
    • TPID (Tag Protocol Identifier): 2 bytes = 0x8100 (identifies this as a tagged frame)
    • PCP (Priority Code Point): 3 bits — for QoS (802.1p priority levels 0–7)
    • DEI (Drop Eligible Indicator): 1 bit — marks frame as eligible for drop during congestion
    • VID (VLAN Identifier): 12 bits — identifies the VLAN (values 0–4095; usable: 1–4094)
  • Maximum VLAN ID = 4094 (0 and 4095 are reserved). 12 bits → 2^12 = 4096 total values.
  • VLAN ID (VID): Each VLAN has a unique 12-bit numeric identifier. Switches use the VID in the 802.1Q tag to determine which VLAN a tagged frame belongs to.

VLAN Trunk Links

  • Access Port: Belongs to ONE VLAN. Frames are untagged (tag is added/removed by switch). Used to connect end devices (PCs, printers, phones).
  • Trunk Port: Carries frames from MULTIPLE VLANs (tagged with 802.1Q tags). Used to connect switches to switches or switches to routers.
  • Native VLAN: Frames on a trunk port are tagged EXCEPT for the native VLAN (default VLAN 1). Native VLAN frames are sent UNTAGGED.
  • Router-on-a-Stick: A router with a single physical trunk link to a switch, using sub-interfaces (each tagged with a VLAN ID) to route between VLANs.

Inter-VLAN Routing

  • Devices in different VLANs cannot communicate at Layer 2 — they need a Layer 3 device.
  • Method 1: Use a router with one physical interface per VLAN (expensive, wastes ports).
  • Method 2: Router-on-a-Stick — one router trunk port with sub-interfaces per VLAN (moderate cost).
  • Method 3: Layer 3 Switch (SVI — Switched Virtual Interface) — most scalable and fastest. Each VLAN gets an SVI with an IP address. The L3 switch routes internally at hardware speed.

Topic 6: Wireless LAN — 802.11 Wi-Fi Very Important — Full Lecture

Why Wireless is Different from Wired Ethernet

Unique Challenges of Wireless Medium

  • Cannot detect collision while transmitting: In wired Ethernet, a station can listen to the channel while transmitting and detect if its signal is being garbled by a collision (CSMA/CD). In wireless, a transmitting station's own signal is much stronger than any received signal — it cannot "hear" a collision while sending. CSMA/CD is impossible.
  • Signal attenuation: Radio signals weaken with distance (path loss). Not all stations can hear all other stations. Two stations may both be able to communicate with the access point but cannot hear each other at all.
  • Interference: Other Wi-Fi networks, microwave ovens, Bluetooth devices, and cordless phones all share the 2.4 GHz band, causing interference that wired networks don't face.
  • Channel is shared: All stations in a Basic Service Set (BSS) share the same wireless channel — unlike a switch where each port has its own dedicated link.

Hidden Terminal Problem Exam Favorite

Hidden Terminal — The Core Problem of Wireless MAC

Scenario: Three nodes A, B, C arranged in a line. B is in the center. A and C are both within range of B but are OUT of range of each other (they cannot hear each other at all).

1A wants to send to B: A senses the channel — it hears nothing (because A cannot hear C). A decides the channel is free. A starts transmitting to B.
2C also wants to send to B: C senses the channel — C also hears nothing (because C cannot hear A's transmission). C also decides the channel is free. C starts transmitting to B simultaneously.
3Collision at B: Both A and C are transmitting to B at the same time. Their signals COLLIDE at B — B receives garbled data and cannot decode either frame. The collision happens at the receiver (B), not at the senders.
4Why CSMA fails: A and C each performed carrier sensing before transmitting, but they sensed each other's transmissions as NOTHING (they are hidden from each other). The "listen before talk" mechanism fails completely — the collision is undetectable by the senders.

A and C are "hidden terminals" from each other's perspective — each is hidden from the other's carrier sense, yet they interfere at the shared receiver B.

Exposed Terminal Problem

Exposed Terminal — Channel Underutilization

Scenario: Four nodes A, B, C, D in a line. B is transmitting to A. C is within range of B but wants to send to D (which is out of B's range).

  • C senses the channel and hears B transmitting. C concludes "channel is busy" and WAITS — even though C's transmission to D would NOT interfere with B→A (because D is out of B's range).
  • C is unnecessarily blocked. C could safely transmit to D simultaneously with B→A without any collision.
  • Result: Channel underutilization — C wastes time waiting unnecessarily. Throughput is reduced.
  • Difference from hidden terminal: The hidden terminal problem causes collisions (bad). The exposed terminal problem causes unnecessary idle time (inefficient). Both are caused by the asymmetric propagation ranges of wireless signals.

Why CSMA/CD Fails in Wireless

Three Reasons CSMA/CD Cannot Work in Wireless

1Cannot listen while transmitting: A wireless node's own transmitted signal is MUCH stronger (orders of magnitude) than any signal it might receive. A transmitting station cannot distinguish a received collision from its own signal — the received signal is completely swamped.
2Signal strength varies: Unlike wired Ethernet where a collision causes a deterministic voltage change detectable by every sender, wireless signal strength varies with distance, obstacles, and interference. No reliable collision detection signal exists.
3Hidden terminal problem: Even if a station could listen while transmitting, it still cannot detect collisions caused by hidden terminals it cannot hear. The collision happens at the receiver, not at the sender..

Solution: CSMA/CA (Collision Avoidance) — instead of detecting collisions after they happen, try to AVOID collisions before they happen.

CSMA/CA — Collision Avoidance

CSMA/CA Working — Step by Step

1Sense Channel (Physical Carrier Sensing): Before transmitting, the station listens to the channel. If the medium is busy, the station DEFERS and waits.
2Wait for DIFS: After the channel becomes idle, the station waits for a mandatory inter-frame gap called DIFS (Distributed IFS) before proceeding. This ensures the channel is truly clear.
3Random Backoff: After DIFS, the station picks a random backoff timer (from a contention window). It counts down the backoff ONLY while the channel is idle. If the channel becomes busy during backoff, the countdown PAUSES and resumes when idle again. This reduces collision probability compared to CSMA/CD's post-collision backoff.
4Transmit: When backoff reaches 0, the station transmits the ENTIRE frame without stopping (no collision detection possible).
5Wait for ACK: After transmitting, the station waits for an explicit ACK from the receiver. A short SIFS (Short IFS) gap is used between frame and ACK. If no ACK is received within a timeout, a collision is assumed → the frame is retransmitted with exponential backoff.

RTS/CTS Mechanism Very Important

RTS/CTS — Request to Send / Clear to Send

RTS/CTS is an optional handshake mechanism used before data transmission to reserve the channel and solve the hidden terminal problem.

1RTS (Request to Send): Before sending a large data frame, the sender (A) sends a short RTS frame to the receiver (B). The RTS includes: sender address, receiver address, and the DURATION of the upcoming data transmission (how long the channel will be needed). RTS is short — low cost if it collides.
2CTS (Clear to Send): If B is ready to receive, it broadcasts a CTS frame. CTS also contains the DURATION from the RTS. ALL nodes within range of B (including hidden terminal C) receive this CTS.
3NAV Setting by Hidden Terminals: Node C hears the CTS from B (even though C couldn't hear A's RTS). C reads the duration field in the CTS and sets its NAV (Network Allocation Vector) timer to that duration. C is now "virtually silent" for the duration of A→B's data transmission — it will not transmit even though it cannot hear A.
4Data Transmission: After receiving CTS, A transmits the full data frame to B.
5ACK: B sends an ACK to A after successfully receiving the data. The transaction is complete.

How RTS/CTS solves hidden terminal: C hears the CTS from B (even though C can't hear A) and defers via NAV. This prevents C from transmitting during A's data frame, eliminating the hidden terminal collision.

Overhead: RTS/CTS adds 2 extra frames before every data frame. Only worth using for large frames — small frames are sent without RTS/CTS to avoid disproportionate overhead.

NAV — Network Allocation Vector

NAV — Virtual Carrier Sensing

  • The NAV is a timer maintained by every wireless station. It represents how long the channel is expected to be busy based on duration fields in received frames (RTS, CTS, data frames).
  • Mechanism: When a station receives an RTS or CTS with a duration field, it sets its NAV countdown timer to that duration — even if it doesn't hear the data frame itself.
  • Effect: While NAV > 0, the station treats the channel as busy and does NOT attempt to transmit — even if physical carrier sensing detects silence. This is "virtual carrier sensing."
  • Dual carrier sensing: 802.11 uses BOTH physical carrier sensing (listening to the actual RF signal) AND virtual carrier sensing (NAV timer) to decide if the channel is busy. If EITHER says "busy," the station defers.

802.11 Infrastructure vs Ad-Hoc Mode

  • Infrastructure Mode (BSS — Basic Service Set): All stations communicate through an Access Point (AP). The AP acts as a base station. Most common Wi-Fi deployment. The AP can connect to a wired network (Distribution System).
  • Extended Service Set (ESS): Multiple BSSs (access points) connected via a Distribution System (usually a wired LAN). Allows roaming between access points — common in offices and campuses.
  • Ad-Hoc Mode (IBSS — Independent BSS): Stations communicate directly with each other without an access point. Used for peer-to-peer wireless networking (e.g., two laptops sharing a file directly). No infrastructure needed.
  • SSID (Service Set Identifier): The "network name" (up to 32 characters). Broadcast in Beacon frames by the AP. Stations associate with an AP with a matching SSID.

  • DHCP Relay Agent: forwards DHCP broadcasts across subnets (broadcasts → unicast to DHCP server)
  • Proxy ARP: router replies with OWN MAC for remote destination; used when host has wrong subnet mask
  • Gratuitous ARP: host ARPs for its own IP — detect IP conflicts, update other caches on MAC change

ARP vs RARP vs BOOTP vs DHCP

→ARP: IP → MAC. Broadcast request, unicast reply. Essential for every IP communication on LAN.
→RARP: MAC → IP. Broadcast only, one server per subnet, IP only. Obsolete.
→BOOTP: Static allocation, cross-router via relay, UDP 67/68, IP+mask+gateway. Legacy.
→DHCP: Dynamic leases (DORA), UDP 67/68, relay agents, scope/lease concept, all options. Current standard.

VLAN Key Facts

→VLAN = logical broadcast domain segmentation on a switch.
→802.1Q tag = 4 bytes: TPID(0x8100) + PCP(3b) + DEI(1b) + VID(12b) → 4094 usable VLANs.
→VLAN host: correct port VLAN + correct IP subnet — BOTH required, not just one.
→End-to-End VLAN: campus-wide, user mobility; Location-Based: local, simpler, preferred today.
→Inter-VLAN routing: L3 switch SVI (fastest); router-on-a-stick (simpler).

Wireless LAN Quick Reference

→Hidden terminal: A, C cannot hear each other → both sense idle → collision at B (receiver).
→Exposed terminal: C unnecessarily defers because it hears B→A, even though C→D would not interfere.
→RTS/CTS: RTS from sender → CTS from receiver (heard by hidden terminal) → hidden terminal sets NAV → defers.
→NAV = virtual carrier sense timer; set from duration field in RTS/CTS; prevents hidden terminal collisions.
→DCF: distributed, CSMA/CA, DIFS + random backoff; PCF: centralized, AP polls, PIFS, deterministic.
→802.11a=5GHz/54Mbps; 802.11b=2.4GHz/11Mbps; 802.11g=2.4GHz/54Mbps.

Exam Strategy — Week 11

  • Wireless questions are conceptual: hidden vs exposed terminal is scenario-based — read carefully who can hear who.
  • RTS/CTS: the KEY step is C hearing the CTS (not the RTS) and setting NAV. Many students confuse this.
  • DHCP DORA order is a standard numerical sequence question — memorize D-O-R-A.
  • DHCP ports 67 and 68 (UDP) — often confused with DNS (53) if unprepared.
  • VLAN assignment scenario: if port VLAN is correct but IP is wrong (or vice versa), communication FAILS — both must match.
  • STP loop root cause question: the answer is "no TTL in Ethernet frames" — not "too many VLANs" or similar.
  • DCF vs PCF: DCF = standard contention (CSMA/CA); PCF = optional centralized (AP polls). DCF is always there; PCF is optional.
  • 802.11a vs 802.11g: both are 54 Mbps but a=5GHz and g=2.4GHz. 802.11b=11 Mbps at 2.4 GHz.
  • ARP: request/reply direction is VERY frequently asked — Request = broadcast; Reply = unicast.
  • Proxy ARP: router's OWN MAC in reply — not the remote host's MAC. For hosts with wrong subnet mask.
