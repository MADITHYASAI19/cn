Topic 1: QoS Introduction — Why It Matters High Weightage

What is Quality of Service?

QoS (Quality of Service) refers to the ability of a network to provide different levels of service to different types of traffic — guaranteeing bandwidth, controlling delay, controlling jitter, and limiting packet loss for specific flows. The internet's original best-effort IP model treats all packets equally. With the explosion of multimedia apps (YouTube, Netflix, VoIP, gaming), best-effort is no longer sufficient for all traffic types.

Key explanation points — Why QoS is needed

1Best-effort vs QoS: Traditional IP = best-effort — all packets treated equally, no guarantees. Routers simply forward packets as fast as possible. Under congestion, packets are dropped arbitrarily. This is fine for elastic traffic (file transfer, email) which can tolerate delay and retransmission. But it is unacceptable for real-time multimedia traffic (VoIP, video conferencing) that needs bounded delay and jitter.
2The two traffic worlds: (1) Elastic/Data traffic — file transfer, web browsing, email. Can adapt to available bandwidth. TCP handles congestion with slow start. Tolerates delay up to seconds. Can retransmit. (2) Real-time/Multimedia traffic — VoIP, video conferencing, live streaming. Cannot tolerate delay >150ms (VoIP), or jitter, or packet loss (causes audible/visual glitches). Cannot retransmit late packets (already due for playback).
3Inelastic applications: Applications like VoIP, interactive video, teleconferencing are inelastic — they generate traffic at a fixed rate regardless of network congestion. Unlike TCP which backs off when congested, inelastic apps cannot reduce their rate and continue filling the network. This is dangerous — they can congest the network and starve other flows.
4QoS goals: (1) Bandwidth guarantee — reserve minimum bandwidth for a high-priority flow. (2) Delay bound — ensure end-to-end delay stays below a threshold (e.g., 150ms for VoIP). (3) Jitter bound — limit the variation in delay across packets in the same flow. (4) Loss guarantee — ensure packet loss rate stays below an acceptable level. Different apps have different QoS requirements.
5QoS approaches: Two major approaches: (1) Fine-grained / per-flow QoS (Integrated Services / IntServ) — reserve resources for each individual flow using signaling protocol RSVP. Provides hard guarantees. Doesn't scale to internet size. (2) Coarse-grained / aggregate QoS (Differentiated Services / DiffServ) — classify traffic into a few classes, treat each class differently. Scales well. Provides soft (relative) guarantees only.
6QoS mechanisms (four building blocks): (1) Traffic classification and marking — identify and mark traffic types at ingress. (2) Traffic shaping / policing — control the rate at which traffic enters the network (token bucket, leaky bucket). (3) Queue scheduling — decide which packets to serve first at router queues (priority queuing, fair queuing, WFQ). (4) Admission control — decide whether to accept a new flow (if resources are insufficient, reject).
7Jitter and its impact: Jitter = variation in packet arrival delay. Example: packet 1 arrives at t=10ms, packet 2 at t=25ms, packet 3 at t=12ms. The gaps between arrivals are different — this is jitter. For audio, jitter causes uneven playback timing → choppy sound. Solution: playout buffer / jitter buffer at the receiver — delays playback by a fixed amount to absorb jitter. Trade-off: larger buffer = better jitter absorption but higher end-to-end delay.
8Multimedia streaming vs interactive: Streaming (YouTube, Netflix) — large pre-buffering at client, tolerates high delay (seconds). Can adapt quality (adaptive bitrate — ABR). Interactive (VoIP, video call) — cannot pre-buffer, delay must stay <150ms for natural conversation. Loss <1–5%. Even with FEC (Forward Error Correction), interactive media has very tight QoS requirements.

Topic 2: Traffic Types & QoS Metrics High Weightage

Understanding traffic characteristics

Different applications generate network traffic with very different statistical patterns. Understanding these patterns is critical for designing QoS mechanisms that can provide the right guarantees for each traffic type without over-provisioning.

Key explanation points — Traffic Types

1CBR — Constant Bit Rate: Traffic is generated at a fixed, steady rate. Example: uncompressed voice over VoIP (G.711 codec) — one packet every 20ms, always the same size. Very predictable. Easy to provision bandwidth — just reserve the fixed rate. Traditional circuit-switched telephone networks were CBR. Simple to handle with QoS: just guarantee the fixed bandwidth.
2VBR — Variable Bit Rate: Traffic rate varies over time but within known bounds (peak rate and average rate). Example: compressed video (H.264) — scenes with little motion → small frames (low rate). Scenes with fast motion → large frames (high rate). Challenging for QoS: must guarantee peak rate (waste if average is low) OR allow statistical multiplexing (risk: if many flows peak simultaneously → congestion).
3Bursty / Data traffic: Traffic arrives in large short bursts separated by long idle periods. Example: web browsing — long pause (user reading page) → sudden burst (clicking link, downloading page). File transfer — large sustained burst then nothing. Extremely hard to predict. TCP handles this elastically. No real-time constraint. Best-effort IP is designed primarily for this type.
4On-Off traffic model: A simplified model for many real-world sources. Source alternates between ON periods (generating data at rate r for time ton) and OFF periods (silent for time toff). Average rate = r × ton / (ton + toff). During ON: if many sources are ON simultaneously → congestion. Statistical multiplexing relies on the fact that not all sources are ON at the same time.
5Statistical multiplexing gain: If N sources share a link, and each is ON only fraction p of the time, the probability all N are ON simultaneously is pᴺ (very small for large N). This means link capacity can be set much less than N × peak rate. The gain = N × peak rate / required link capacity. Example: 100 users each with 1Mbps peak, ON 10% of time — a 10Mbps link can serve them all since rarely more than 10 are ON at once.
6Traffic descriptor / flow specification: To provide QoS guarantees, a flow must describe its traffic to the network. Common parameters: Token Bucket params (r, b): sustained rate r (bytes/sec), burst size b (bytes). Peak rate p: maximum instantaneous rate. Minimum policed unit: smallest packet size considered. These parameters form the flow specification (flowspec) used in IntServ/RSVP.

Key explanation points — QoS Metrics

7Bandwidth (throughput): The data rate available to a flow (bits per second). QoS bandwidth guarantee = the network reserves at least this rate for the flow. Measured as: goodput (useful data delivered per second), not counting retransmissions or headers. Different from raw link speed — actual throughput depends on congestion, protocol overhead.
8Delay (latency): Total time from packet sent to packet received at destination. Components: (1) Propagation delay = distance / speed of light through medium. (2) Transmission delay = packet size / link bandwidth. (3) Queuing delay = time waiting in router buffers. (4) Processing delay = time for router to look up forwarding table. Propagation + transmission are fixed for a given link. Queuing delay is variable and dominates under congestion.
9Jitter (delay variation): The difference in delay between consecutive packets of the same flow. Causes: variable queuing delay at congested routers. Impact: for real-time streams, high jitter means packets arrive unevenly → audio/video playback becomes choppy. Measured as: standard deviation of one-way delay across packets, or max delay minus min delay across a window. Bound by jitter buffer size at receiver.
10Packet loss: Fraction of packets that are dropped (never received). Causes: router buffer overflow (most common), bit errors (rare on wired links), congestion. Impact: TCP handles loss by retransmission (acceptable for data). For real-time media: loss = missing audio/video frames → gaps/artifacts. Typical tolerable loss: VoIP <1–3%, video <0.1%. Loss above these thresholds causes perceptible quality degradation.
11QoS parameters summary — ITU-T G.114: For VoIP, ITU-T recommends: One-way delay < 150ms (preferred) or < 400ms (acceptable). Jitter < 10–20ms. Packet loss < 1%. These are the standard numbers to memorise for exams. Beyond 150ms delay, callers start using "push-to-talk" style pauses (walkie-talkie effect).
12Delay budget: The total allowed delay is distributed among network segments. Example: total allowed = 150ms. Propagation (sender to receiver) may take 50ms. Router queuing may take up to 20ms per hop × 3 hops = 60ms. Jitter buffer at receiver = 40ms. 50+60+40 = 150ms — the entire budget is consumed. QoS must ensure queuing delays stay within budget. This is why delay-bound guarantees are needed per hop.

Topic 3: Traffic Shaping — Leaky Bucket & Token Bucket Most Important

Why shape traffic?

Traffic shaping controls the rate and burstiness of traffic entering the network. Without shaping, a bursty source can flood the network with packets, causing congestion and harming other flows. Traffic shaping smooths out bursts to match the contracted rate. Two classic algorithms: Leaky Bucket (rigid rate limiting) and Token Bucket (allows burstiness up to a limit).

Key explanation points — Leaky Bucket

1Leaky Bucket concept: Imagine a bucket with a small hole at the bottom. Water (packets) poured in from top at any rate — can be bursty. Water leaks out from the bottom at a constant, fixed rate regardless of how fast it arrives. If bucket fills up → overflow (packets dropped). Result: output is always at a constant rate (smoothed/shaped traffic) even if input is bursty.
2Leaky Bucket — network implementation: A queue (buffer) acts as the bucket. Incoming packets enter the queue at any rate. A constant-rate server removes packets from the queue and sends them onto the network at rate r bytes/sec (fixed). If the queue is full → new packets are dropped (overflow). Output: always a steady stream at rate r — burst is absorbed/eliminated.
3Leaky Bucket — limitation: Even if the network has spare capacity, leaky bucket cannot exceed rate r. Example: user sends a 10MB file fast (burst). Leaky bucket slows this to rate r even if the link is idle. This is inflexible — cannot take advantage of momentarily available bandwidth. Good for CBR traffic (voice). Bad for bursty data where bursts can be accommodated.
4Leaky Bucket parameters: Rate r (output rate, e.g., bits/sec or packets/sec). Buffer size b (maximum queue length). If arrival rate < r — queue drains, no congestion. If arrival rate > r for sustained period — queue fills to capacity b, then packets drop. Maximum output burst in any interval T = r × T (perfectly smooth, no burst at output).

Key explanation points — Token Bucket

5Token Bucket concept: A bucket holds tokens. Tokens are added at a constant rate r tokens/sec. Bucket has a maximum capacity of b tokens (when full, new tokens are discarded). To send a packet of size L bytes, the sender must first remove L tokens from the bucket. If insufficient tokens → packet must wait (or be dropped). Key difference from leaky bucket: tokens can accumulate, allowing a burst when tokens have built up.
6Token Bucket — burst capability: If the flow is idle, tokens accumulate up to the maximum b. When data arrives, it can consume up to b tokens all at once → send a burst of up to b bytes instantly. After the burst, tokens must refill at rate r. This allows short bursts while maintaining long-term average rate r. This is more flexible and realistic than leaky bucket for data traffic.
7Token Bucket parameters: Token arrival rate = r (sustained rate, bytes/sec or packets/sec). Token bucket size = b (maximum burst size in bytes). Maximum burst duration = b/r seconds. In any time interval T, the maximum bytes that can be sent = min(b + r×T, peak_rate × T) (capped by peak rate if specified).
8Token Bucket + Leaky Bucket combination: Often used together: Token Bucket allows shaping with controlled bursts. A downstream Leaky Bucket then smooths out the token-bucket-allowed bursts to a peak rate p. Combined: guaranteed average rate r, allowed burst b, but peak output rate ≤ p (peak rate limiter). This is used in ATM traffic shaping (GCRA — Generic Cell Rate Algorithm).
9Traffic policing vs Traffic shaping: Shaping: delay excess packets (buffer them and send later) — smooths traffic but adds delay. Policing: drop or mark (re-mark DSCP) excess packets that exceed the contract rate — no delay added, but packets may be lost. Policing is done at the network edge (metering incoming traffic from customers). Shaping is done at the sender side (own traffic).
10Three-color marking (srTCM/trTCM): Used in DiffServ policing. Packets are marked Green (within CIR — Committed Information Rate), Yellow (exceeds CIR but within PBS/EBS), or Red (exceeds all limits). Green = forward normally. Yellow = forward if possible (drop on congestion). Red = drop. srTCM = single rate three color marker. trTCM = two rate three color marker (defines both CIR and PIR).

Idle accumulation Queue drains — no benefit from idle Tokens accumulate up to b during idle

Topic 4: Queue Scheduling Algorithms Most Important

Why scheduling matters

When multiple packets are queued at a router's output link, the scheduling algorithm determines the order in which they are served (transmitted). Scheduling is the core mechanism for differentiating traffic classes and providing QoS guarantees at routers. Different algorithms give different trade-offs between fairness, delay, and priority.

Key explanation points — Scheduling Algorithms

1FIFO (First In First Out) / FCFS: The default scheduling algorithm. Packets are served in the exact order they arrive — first packet in queue is first transmitted. Simple and fair in a naive sense. Problem: no differentiation between traffic classes. A large file download's packets and a VoIP call's packets wait in the same queue — the file download can cause high delay for VoIP. A single heavy flow can dominate ("head-of-line blocking").
2Priority Queuing (PQ): Multiple queues, each assigned a priority level. Packets are classified into priority queues (e.g., High, Medium, Low). Scheduler always serves the highest non-empty priority queue first. Only when the high-priority queue is empty does it serve medium, and so on. Advantage: low-priority high-priority traffic gets very low delay. Problem: starvation — if high-priority queue is always non-empty, lower queues never get served.
3Priority Queuing — preemptive vs non-preemptive: Non-preemptive (most common): if a high-priority packet arrives while a low-priority packet is being transmitted, wait until current transmission finishes, then serve high priority. Preemptive: interrupt current low-priority transmission immediately and start high-priority packet (impractical for packets — transmission cannot be interrupted mid-stream on a shared link).
4Round Robin (RR) scheduling: Multiple queues. Scheduler cycles through queues in a round-robin fashion — serve one packet from Queue 1, then one from Queue 2, then one from Queue 3, then back to Queue 1. Skips empty queues. Fair if all packets are the same size (each flow gets equal number of packets served). Problem: not byte-fair — if one flow sends large packets and another sends small packets, large-packet flow gets more bytes in each round.
5WFQ — Weighted Fair Queuing: Extends Round Robin with weights. Each queue Q_i is assigned a weight w_i. In each scheduling round, queue Q_i gets to send w_i bytes (proportional to its weight). Example: 3 queues with weights 1:2:3 — in each round, Q1 sends 1 unit, Q2 sends 2 units, Q3 sends 3 units. Result: each queue gets bandwidth proportional to its weight. Provides weighted fair bandwidth sharing while preventing starvation of any flow.
6WFQ — delay bound: With WFQ, if a flow i has weight w_i and the link rate is R bps, flow i is guaranteed at least bandwidth R × w_i / (Σw_j). Additionally, if the flow uses a token bucket (r, b), the maximum delay for any packet in the queue is bounded by: D_max = b / R_i where R_i is the guaranteed rate. WFQ provides both bandwidth and delay guarantees per flow — key for QoS.
7GPS — Generalized Processor Sharing: An ideal (mathematical) fluid model where all flows are served simultaneously in proportion to their weights. Physically impossible (you cannot transmit a fraction of a packet). WFQ is the practical packet-level approximation of GPS. GPS gives the best fairness — WFQ approximates it and is used in real routers.
8Deficit Round Robin (DRR): Addresses the packet-size unfairness in Round Robin. Each queue gets a quantum Q bytes per round. Track "deficit" (leftover bytes from previous round). If a queue's head packet ≤ remaining bytes → send. Otherwise carry forward the deficit. Result: over multiple rounds, each queue sends exactly Q bytes on average — truly byte-fair. Efficient (O(1) per packet).
9RED — Random Early Detection: Active Queue Management (AQM) algorithm. Instead of dropping packets only when queue is full (tail-drop), RED starts probabilistically dropping packets when queue occupancy exceeds a minimum threshold. Drop probability increases linearly until the queue reaches a maximum threshold (then drop all). Purpose: signal congestion to TCP senders early → TCP backs off before queue is completely full → prevents queue oscillation and global synchronisation of TCP flows.
10Tail-drop problem: Default queue management: drop packets only when queue is completely full (tail-drop). Problem: TCP flows using the same bottleneck link all detect loss simultaneously → all back off simultaneously → link goes idle → all increase together → congest again. This is global synchronisation. RED avoids this by staggering drops across different flows before the queue is full.
Algorithm Mechanism Advantage Disadvantage

Priority Queuing Always serve highest priority queue first Low delay for high-priority traffic Starvation of low-priority queues
Round Robin Cycle through queues, one packet each Fair (packet-wise); no starvation Not byte-fair for variable packet sizes

Topic 5: Integrated Services (IntServ) & Differentiated Services (DiffServ) High Weightage

Two QoS architectures

IntServ provides per-flow hard QoS guarantees using resource reservation (RSVP). DiffServ provides per-class soft QoS using traffic classification and marking — no per-flow state at interior routers. IntServ is ideal but does not scale to internet size. DiffServ is widely deployed in enterprise and ISP networks today.

Key explanation points — Integrated Services (IntServ)

1IntServ architecture: Defined in RFC 1633. Provides per-flow QoS guarantees by reserving resources (bandwidth, buffer space) at every router along the path before sending data. Three components: (1) Admission control — router decides if resources are available; rejects flow if not. (2) RSVP — Resource Reservation Protocol for signaling. (3) Per-flow queuing — WFQ or similar at each router maintains per-flow state.
2RSVP — Resource Reservation Protocol: Signaling protocol used by IntServ. Operates out-of-band (separate from data path). Receiver-initiated reservation (unusual — receiver is often the bottleneck, knows what it can handle). Steps: (1) Sender sends PATH message toward receiver (carries traffic description — flowspec: token bucket params r, b, peak rate p). (2) Receiver sends RESV message back toward sender, requesting resource reservation along each router. (3) Each router either accepts and reserves resources, or rejects (and sends error back).
3IntServ service classes: (1) Guaranteed Service — hard bandwidth and delay guarantee. For real-time applications that absolutely cannot tolerate delay violation (VoIP, teleconferencing). No packet loss (other than link errors). Delay is strictly bounded. Mathematically provable using network calculus. (2) Controlled Load Service — "best-effort quality under no-load conditions." Emulates what best-effort gives when the network is lightly loaded. Not a hard guarantee. For tolerant multimedia apps.
4IntServ scalability problem: Every router in the internet must maintain state for every active flow passing through it. Internet has millions of concurrent flows. Per-flow state = massive memory and processing overhead. RSVP soft state (must be refreshed periodically) → constant signaling traffic. Does not scale to internet backbone. IntServ is only practical in small/controlled networks (enterprise, research networks).
5RSVP soft state vs hard state: RSVP uses soft state — reservations are temporary and must be refreshed periodically by new PATH/RESV messages (typically every 30 seconds). If refresh stops, reservation automatically expires. Advantage: auto-cleanup when sessions end without explicit teardown. Disadvantage: constant refresh overhead. Compare: hard state (from TCP SYN → must explicitly tear down) is stateful and requires explicit cleanup.

Key explanation points — Differentiated Services (DiffServ)

6DiffServ architecture: Defined in RFC 2474/2475. Provides class-based QoS (not per-flow). Key principle: complex per-flow processing at the network edge (ingress/egress routers); simple aggregate processing at the core (interior routers). Interior routers only look at the DSCP field in IP header and apply Pre-configured Per-Hop Behaviors (PHBs). No RSVP, no per-flow state at core routers.
7DSCP — Differentiated Services Code Point: Top 6 bits of the TOS/DS field in the IP header. Defines 64 possible codepoints. Packets are marked by ingress edge router (or by the end host) with a DSCP value indicating their traffic class. Interior routers read the DSCP and apply the corresponding PHB (forwarding behaviour). DSCP marking is done without any signaling protocol — simple and scalable.
8Per-Hop Behavior (PHB): The forwarding treatment applied to a packet at each router based on its DSCP marking. Defined by IETF. Two main PHBs: (1) EF (Expedited Forwarding) — for real-time traffic. Provides low loss, low delay, low jitter. Conceptually: departure rate ≥ arrival rate (queue never grows). When combined with admission control, guarantees are achieved. (2) AF (Assured Forwarding) — for important but non-real-time traffic. Four AF classes (AF1x–AF4x), three drop precedences each. Higher drop precedence = first to be dropped under congestion.
9EF PHB (DSCP = 101110 = 46): Expedited Forwarding. Provides Premium Service — low delay, low jitter, low loss, assured bandwidth. Packets with EF DSCP are always prioritised over BE (best-effort). Router must serve EF queue so that it never accumulates — conceptually, the router's EF departure rate ≥ the configured EF arrival rate. If admission control ensures EF traffic ≤ reserved rate, the delay bound is provably small.
10AF PHB (DSCP = AFxy): Assured Forwarding. x = class (1–4, four queues with different bandwidth allocations). y = drop precedence (1=low=least dropped, 2=medium, 3=high=first dropped). Example: AF11 (class 1, low drop precedence) vs AF13 (class 1, high drop precedence). Under congestion, AF13 packets are dropped before AF12, before AF11. Allows ISPs to offer tiered services within the same class.
11DiffServ domains and SLA: DiffServ is deployed within a DiffServ domain (e.g., an ISP's network). At the boundary between two domains, edge routers re-mark DSCP values according to the inter-domain SLA (Service Level Agreement). Edge routers also perform traffic conditioning (metering, shaping, policing) to ensure traffic conforms to the agreed rate before forwarding into the domain.
12IntServ + DiffServ together: Modern networks often use IntServ at the edge (applications signal requirements via RSVP) combined with DiffServ in the core (routers use DSCP-based PHBs for aggregate handling). This gives the benefits of both: application-level control at edge, scalable simple processing at core. The edge router maps RSVP reservations to appropriate DSCP markings for DiffServ processing by core routers.

Absolute must-know facts

  • QoS needed for real-time traffic: VoIP, video conf, gaming — best-effort fails these
  • Elastic traffic (TCP): adapts to congestion. Inelastic traffic (VoIP): cannot adapt rate
  • VoIP requirements: delay <150ms, jitter <20ms, loss <1% (ITU-T G.114)
  • Jitter = variation in packet inter-arrival delay. Absorbed by jitter buffer (playout buffer) at receiver
  • CBR: fixed rate (e.g., uncompressed voice). VBR: variable rate (compressed video). Bursty: web/data traffic
  • Delay components: propagation + transmission + queuing + processing. Queuing delay is most variable (causes jitter)
  • Leaky Bucket: constant output rate r. Absorbs bursts entirely. Output always smooth
  • Token Bucket: tokens accumulate at rate r, bucket size b. Allows burst up to b bytes. Average rate = r
  • Token Bucket max burst = b (bucket capacity). Tokens do NOT overflow beyond b
  • Shaping = delay excess packets (buffer). Policing = drop/re-mark excess packets
  • FIFO: no QoS differentiation, head-of-line blocking
  • Priority Queuing: high priority first → starvation of low-priority queues if high is always busy
  • Round Robin: packet-fair but NOT byte-fair
  • WFQ: proportional bandwidth sharing = w_i / Σw_j × R. No starvation. Delay bound = b/R_i
  • RED: probabilistic drop before queue full → prevents global TCP synchronisation. Better than tail-drop
  • IntServ: per-flow reservation via RSVP. Hard guarantees. Does NOT scale (per-flow state at every router)
  • RSVP: PATH (sender → receiver), RESV (receiver → sender, reserves resources). Soft state (must refresh every ~30s)
  • IntServ service classes: Guaranteed Service (hard bounds) and Controlled Load Service (no-load-like)
  • DiffServ: per-class. Uses DSCP (6 bits of TOS field). PHBs at each router. No per-flow state at core
  • EF PHB (DSCP=46): Expedited Forwarding — low delay, low jitter, low loss. For real-time traffic
  • AF PHB: AFxy. x=class(1–4), y=drop precedence(1=least dropped, 3=most dropped). AF13 dropped before AF11
  • DSCP = top 6 bits of TOS/DS field. ECN = bottom 2 bits
  • IntServ + DiffServ combined: RSVP at edge → maps to DSCP → DiffServ in core
  • Statistical multiplexing gain: not all users active simultaneously → capacity < sum of peak rates

Tricky NPTEL MCQ traps — don't get these wrong

1Leaky bucket: output is ALWAYS at rate r. Even if nothing is in the bucket, output = 0 (not r). Output = min(current_data, r×dt)
2Token bucket: tokens overflow (capped at b). If sender is idle 100 seconds, max burst is still only b, NOT r×100
3Starvation: happens in Priority Queuing, NOT in WFQ or Round Robin
4WFQ: bandwidth proportional to WEIGHT, not inversely. Higher weight = MORE bandwidth
5RSVP PATH goes sender → receiver. RESV goes receiver → sender (reservation is receiver-initiated)
6IntServ = per-flow guarantees. DiffServ = per-class (aggregate). DiffServ scales; IntServ does not
7EF DSCP = 46 (101110 binary). AF drop precedence: 1=least dropped, 3=MOST dropped
8DiffServ DOES NOT use RSVP. RSVP is IntServ's signaling protocol
9RED drops probabilistically BEFORE queue is full (signals congestion early). Tail-drop waits until FULL
10Queuing delay is the MOST VARIABLE component of total delay — causes jitter under congestion
11RSVP uses SOFT STATE (expires if not refreshed). TCP uses hard state (requires explicit teardown)
12VoIP recommended delay: <150ms (preferred), <400ms (absolute max). Source: ITU-T G.114

Exam strategy — Week 7

  • Leaky vs Token bucket: core exam topic. Know the burst behaviour difference. Practice the token accumulation calculation

  • Priority Queuing: always leads to starvation of low queues if high queue is busy. Know why WFQ fixes this
  • RED vs tail-drop: RED = probabilistic early drop = avoid global synchronisation. Tail-drop = drop when full = oscillation

  • RSVP: PATH direction (→ receiver), RESV direction (← sender). Soft state = must refresh periodically
  • DiffServ EF: low delay/jitter. AF: four classes, three drop precedences within each. Higher drop precedence → dropped first
