Topic 1: Transport Layer — Role & Implementation Very High Weightage

What is the Transport Layer?

The Transport Layer is the 4th layer of the TCP/IP protocol stack. It sits on top of the Network Layer and provides end-to-end connectivity between two host machines. Unlike all lower layers which work hop-by-hop, the transport layer is only present at the two end hosts — never in intermediate routers or switches.

Key explanation points

1End host vs intermediate devices: Both end hosts have all 5 layers (Application → Physical). Layer 2 switches only have up to Data Link layer. Layer 3 switches and Routers have up to Network layer. The Transport layer is exclusively at end hosts.
2Hardware layer (Physical): Implemented in the NIC (Network Interface Card). The NIC provides hardware-level signaling, modulation, and coding. Physical layer = fully in hardware.
3Firmware / Device Driver (Data Link): The device driver you install for your NIC. Contains the MAC layer (Media Access Control) implementation. In wireless networks, parts of the Data Link layer may also be in hardware for performance. The upper part — LLC (Logical Link Control) — is in software.
4Kernel software (Network + Transport): In Unix-type OS, the Network layer (IP) and Transport layer (TCP, UDP) are implemented inside the operating system kernel. Applications talk to transport through the kernel interface (sockets).
5Application layer: Runs on top of the kernel. Includes browser apps, chat apps, social media apps (Facebook, YouTube, Twitter). Each is designed by application developers using socket programming APIs to communicate over the network.
6Encapsulation — how headers are added: Application generates data → Transport Layer adds TCP/UDP header → Network Layer adds IP header → Data Link adds MAC header → Physical converts to bits and transmits. At receiver side, each layer strips its own header (decapsulation).
7Why Transport Layer is needed: The Network Layer (IP) provides only unreliable datagram delivery. Intermediate routers have finite buffers that can overflow, causing packet loss. Physical channels can corrupt data. Transport Layer sits on this unreliable base and provides reliable, ordered, flow-controlled end-to-end delivery.
8Socket programming: You can write your own programs to configure and use the transport layer to make two end devices communicate. This is done via the socket API — we will look at socket programming in subsequent classes.

Common mistakes

  • Thinking routers have a Transport layer — they do NOT
  • Confusing the Transport layer with the Network layer — Network = hop-by-hop, Transport = end-to-end
  • Thinking the Physical layer is in software — it is in hardware (NIC)
  • Forgetting that LLC (upper Data Link) is in kernel software, while MAC can be in hardware or driver

Topic 2: Transport Services — TCP vs UDP Very High Weightage

Services provided by the Transport Layer

The Transport Layer must provide services on top of the unreliable Network Layer. These services ensure that application data is delivered correctly, in order, and efficiently between two remote hosts — even if those hosts are in different countries (e.g., IIT Kharagpur to Google USA).

Key explanation points

1Connection Establishment: Setting up a logical "pipe" between two end hosts before sending data. Like saying "Hello" on a phone call to confirm the other side is alive and ready to receive. The other end must acknowledge the hello before data transfer begins.
2Reliable Data Delivery: The Transport layer monitors whether data sent is received at the other end. If not received (detected via timeout), it retransmits. Eventually, every message sent by the application is guaranteed to be delivered to the receiver.
3Flow Control: Ensures the sender does not overwhelm the receiver. Example: if Transmitter (T) sends at 10 Mbps but Receiver (R) can only accept at 1 Mbps, T must slow down to 1 Mbps. Without flow control, excess data overloads the network and gets dropped — wasting resources.
4Congestion Control: Prevents the entire network from becoming congested. Like road traffic management at a junction where vehicles come from multiple roads. If data from multiple sources floods an intermediate router, it congests and drops packets. Transport layer signals senders to slow down.
5Ordered Packet Delivery: Packets may travel different paths through the network and arrive out of order (e.g., Packet 3 arrives before Packet 2). The Transport layer buffers and re-orders packets before delivering to the application — so application always sees data in the correct sequence.
6UDP — User Datagram Protocol: Works as a simple wrapper over the IP layer. Does NOT provide connection establishment, reliability, flow control, congestion control, or ordered delivery. Best for applications where real-time delivery matters more than reliability (e.g., video streaming, VoIP, DNS, gaming). Tolerates some packet loss.
7TCP — Transmission Control Protocol: Provides ALL transport services: connection establishment, reliable delivery, flow control, congestion control, and ordered packet delivery. Used for HTTP, FTP, SMTP — applications where every byte must arrive correctly. Introduces processing delay compared to UDP.
8Why multimedia uses UDP: In video/audio streaming, if a frame is lost, the player can interpolate (average frame 1 and frame 3 to reconstruct frame 2). Retransmitting a lost frame takes extra time, making the experience worse. So delay tolerance (UDP) is preferred over reliability (TCP) for multimedia.

Common mistakes

  • Thinking UDP is "broken" — it is intentionally simple for performance-critical apps
  • Thinking TCP is always better — TCP adds delay which is unacceptable for real-time applications
  • Confusing Flow Control (sender vs receiver) with Congestion Control (sender vs network)
  • Forgetting that DNS uses UDP — it is a very small query/response, UDP overhead is sufficient

Topic 3: Connection Establishment Most Important

What is Connection Establishment?

Before sending data, the two end hosts must confirm that each is alive and ready to receive. This is like saying "Hello" on a phone call. In a circuit-switched telephone network this is trivial, but in a packet-switched network it is non-trivial due to packet loss, delays, and duplicate packets.

Key explanation points

1Problems in packet switching (why simple hello fails): (a) Packets can be LOST (buffer overflow). (b) Packets can be arbitrarily DELAYED (congestion). (c) DUPLICATE packets can appear — sender retransmits thinking a packet was lost, but the original was just stuck in a queue and arrives later. All of these create confusion during connection setup.
22-way handshake problem: Client sends CR (Connection Request), Server replies with CA (Connection Acknowledgement). This naive protocol FAILS because a delayed duplicate CR from an old crashed session can fool the server into thinking a new connection is being requested.
3Solution 1 — Throwaway transport addresses (port numbers): If a machine crashes, restart using a NEW port number. The old delayed duplicate will arrive on the old port which no longer exists, so it is discarded. PROBLEM: We have a finite number of port numbers (0–65535). You cannot throw away a port forever — not feasible.
4Solution 2 — Unique connection identifiers: Give each connection a globally unique ID, put it in every packet. PROBLEM: Ensuring global uniqueness is complex, especially across restarts. Requires special hardware/algorithms and adds significant overhead.
5Solution 3 — Restrict packet lifetime (BEST): Associate a maximum lifetime T with every packet. Once a packet's lifetime expires it is automatically removed from the network. This eliminates the possibility of having delayed duplicates — the old packet is guaranteed dead before a new connection starts. This is the feasible solution used in practice.
63 methods to restrict packet lifetime: (a) Restricted network design — prevent looping; set max queuing delay per packet. (b) Hop count (TTL) — each router decrements a hop count field; when it reaches 0, the packet is dropped. This is what Internet uses today (TTL in IP header). (c) Timestamp — attach a lifetime timestamp; expired packets are dropped. Requires clock synchronization across all devices — impractical at internet scale.
7Tomlinson's 2 requirements (1975) — "Selecting Sequence Numbers": Requirement 1: A sequence number must never refer to more than one byte at any time (using BYTE sequence numbers, not packet sequence numbers). Requirement 2: The valid range of sequence numbers must be positively synchronized between sender and receiver — handled by the flow control algorithm.
8Byte sequence numbers vs packet sequence numbers: TCP uses BYTE sequence numbers — every individual byte gets a unique sequence number. A packet header contains: Sequence Number (e.g., 500) + Length (e.g., 100 bytes) → means bytes 501–600 are in this packet. This allows exact byte-level identification and ordered delivery.
9Forbidden Region: Once a sequence number is used in a connection, that same sequence number must NOT be reused within the maximum packet lifetime T. The range of sequence numbers that are "in use" (including their lifetime T) is called the Forbidden Region. New connections must use sequence numbers OUTSIDE the forbidden region of old connections.
10Initial Sequence Number (ISN) selection — 2 strategies: (a) Wait for duration T before starting a new connection — ensures all old packets with old sequence numbers are dead. (b) Choose a new ISN that is HIGH ENOUGH above the previous sequence number — sequence number spaces of two connections don't overlap.
11Bounded rate requirement: Sequence numbers must be generated at a BOUNDED rate — not too fast (might wrap around and re-enter old forbidden region) and not too slow (new connection overlaps old slow connection). TCP uses a self-clocking mechanism: generate a new sequence number only when an ACK is received, keeping rate bounded.
12TCP uses 32-bit sequence numbers: Can represent 2^32 ≈ 4 billion unique sequence numbers. Must ensure these don't "wrap around" too quickly. At high data rates, the 32-bit space can wrap within the lifetime T — this would cause confusion. Rate control prevents wrap-around within T.

3-Way Handshake — How it works

Host 1 → CR (SEQ=x)
→
Host 2 → ACK(x) + CR (SEQ=y)
→
Host 1 → ACK(y) + DATA

Host 1 validates the ACK: if the sequence number x in the ACK matches the CR it sent, the connection is genuine. If different, it was a delayed duplicate — Host 1 sends REJECT. Host 2 validates: if ACK number from Host 1 matches y it proposed, genuine; if not, delayed duplicate — discard.

Handling delayed duplicates with 3-Way Handshake

  • Case 1 — CR is delayed duplicate: Host 2 receives old CR with seq x. Sends ACK(x). Host 1 sees ACK for x but never intended to connect — sends REJECT. Host 2 receives REJECT — abandons connection. ✓ Correctly handled.
  • Case 2 — Both CR and ACK are delayed duplicates: Host 2 receives delayed duplicate CR, sends ACK(x)+proposes y. Later gets a different ACK with wrong acknowledgement number (z ≠ y). Host 2 discards the mismatched ACK. Host 1 sends REJECT for its part. ✓ Correctly handled.

Common mistakes

  • Thinking 2-way handshake is sufficient — it cannot handle delayed duplicates in packet-switched networks
  • Confusing packet sequence numbers with byte sequence numbers — TCP uses BYTE sequence numbers
  • Forgetting that throwaway port solution is NOT feasible — finite port space
  • Thinking timestamp-based lifetime is practical — it requires clock synchronization which is very difficult
  • Thinking the forbidden region problem only arises from crashes — fast or slow data rates can also cause overlap

Topic 4: Connection Release Important

What is Connection Release?

After data transfer is complete, the logical connection between two hosts must be released. Connection release is simpler than connection establishment (no sequence number problem), but introduces its own challenge — ensuring both sides agree to close without data loss over an unreliable channel.

Key explanation points

1Asymmetric Release: One party simply hangs up (sends a connection release message) and the connection is broken immediately. Simple, but DANGEROUS — if Host 1 closes while Host 2 still has data to send, that data is LOST. Like hanging up a phone call while the other person is still speaking. Leads to data loss.
2Symmetric Release: Treats the connection as TWO separate unidirectional connections. Each direction must be released independently. Host 1 sends DR (Disconnect Request) for Host1→Host2 direction; Host 2 sends DR for Host2→Host1 direction. Only when BOTH sides have sent and acknowledged DR is the connection fully closed. Avoids data loss.
3The Two-Army Problem: Two armies (Blue) are on two hills, separated by a valley controlled by the White army. Blue Army 1 wants to coordinate a simultaneous attack with Blue Army 2. Any messenger crossing the valley may be captured (message lost). The problem proves: no protocol can guarantee consensus over an unreliable channel. This is directly analogous to the symmetric release problem.
4Why Two-Army Problem applies: In symmetric release, each side needs to be sure the other side received the DR and agreed to close. But since the final ACK can always be lost, the sender can never be 100% sure. The Two-Army Problem proves this is theoretically unsolvable with perfect correctness.
5Practical solution — Timers: Each host sends a DR and starts a timer. If ACK is received within the timeout, release the connection. If not, retransmit DR. After N retries without success, forcefully release the connection. This is not 100% correct but is the best practical approach — there is always a tradeoff between correctness and performance.
6Lost ACK scenario: Host 2 sends DR, Host 1 receives DR, sends ACK, but ACK is lost. Host 2 times out → retransmits DR. Host 1 receives another DR, resends ACK, releases. Host 2 gets ACK → releases. Works correctly via retransmission.
7Both DR and ACK lost scenario: Both hosts will eventually time out after N retries and independently release the connection. There is a small possibility of data loss in this corner case. This is accepted as the practical tradeoff — correctness vs. performance.
Property Asymmetric Release Symmetric Release
Who closes One party unilaterally Both parties independently
Data loss risk High — other side may still be sending Low — both sides finish before closing
Protocol complexity Simple More complex
Correct over unreliable channel No No (Two-Army Problem) — but better
Used in practice Rarely preferred Yes (with timers) — used in TCP FIN handshake

Common mistakes

  • Thinking symmetric release is 100% correct — it is NOT (Two-Army Problem)
  • Confusing asymmetric and symmetric release — asymmetric = one hangs up; symmetric = both independently close
  • Forgetting that after N timeouts, the host forcefully releases — there is always a small data loss risk
  • Thinking Two-Army Problem applies only to military scenarios — it is a formal proof about any unreliable channel

Topic 5: Flow Control, Reliability & ARQ High Weightage

Why Flow Control AND Reliability at Transport Layer?

Flow control and reliability are implemented at BOTH the Data Link layer (hop-by-hop) AND the Transport layer (end-to-end). They serve different purposes and neither alone is sufficient. The key paper motivating end-to-end protocols is by Saltzer, Reed, and Clark — foundational reading for networking principles.

Key explanation points

1Why hop-by-hop flow control alone is insufficient: Say link S→R1=10Mbps, R1→R2=5Mbps, R2→R3=3Mbps, R3→D=1Mbps. Source S pushes 10 Mbps. Even with hop-by-hop control, packets accumulate at R1 (can only forward at 5 Mbps). Buffer overflows, packets drop. An end-to-end control is needed to tell S to send at only 1 Mbps (the bottleneck rate).
2Why end-to-end flow control alone is insufficient: Even if S and D agree on 1 Mbps, multiple sources may converge at one router from different links. If total incoming rate exceeds outgoing capacity, packets are dropped. Hop-by-hop flow control reduces congestion at individual routers, complementing the end-to-end mechanism.
3Summary: Flow control/error control at Transport layer = ESSENTIAL for correctness. Flow control/error control at Data Link layer = IMPROVES performance. Both are needed and serve complementary roles.
4Stop-and-Wait Flow Control (error-free channel): Simplest protocol. Sender sends ONE frame, waits for ACK before sending the next. Sender only sends Frame N+1 after receiving ACK for Frame N. Works perfectly on error-free channel. Major disadvantage: very low utilization — link is idle while waiting for ACK.
5Stop-and-Wait ARQ (noisy channel): Same as Stop-and-Wait but adds sequence numbers (0 and 1 only — 1-bit sequence number is enough because only one outstanding frame). If a frame is lost or corrupted, sender times out and retransmits. If ACK is lost, sender retransmits and receiver uses sequence number to detect duplicate. ARQ = Automatic Repeat Request.
6Stop-and-Wait sequence numbers — why 1-bit is enough: At any time, only ONE frame is outstanding in the network. Sequence numbers alternate: 0, 1, 0, 1... An ACK of 1 means "I received frame 0, expecting frame 1." At most 2 distinct sequence numbers needed → 1 bit suffices.
7Problem with Stop-and-Wait: For every packet sent, sender must wait for ACK before sending next. If the link has high bandwidth-delay product, the link is mostly idle. Example: 1 Gbps link, 100ms RTT → sender can only pipeline one packet every 100ms — extremely wasteful. This leads to Sliding Window protocols.
8Sliding Window Protocol — broad idea: Sender maintains a SENDING WINDOW — a set of sequence numbers it is permitted to send without waiting for ACK. Receiver maintains a RECEIVING WINDOW — a set of sequence numbers it is permitted to accept. Multiple frames are sent in a pipeline fashion. ACKs arrive parallelly. Window slides forward as ACKs are received.
9Sliding Window — how it works: Say sender window size = 7. Sender can send frames 0–6 without waiting. When ACK for frame 0 arrives, window slides to 1–7 (can now send frame 7). Both sending window and receiving window slide as ACKs arrive. This "pipeline" fully utilizes the link capacity.
10Piggybacking: In a bidirectional connection, instead of sending ACK as a separate message, the ACK is included ("piggybacked") into the data frame being sent in the reverse direction. This reduces overhead by combining data + ACK into one packet.

Common mistakes

  • Thinking flow control at Data Link layer alone is enough — it handles only hop-by-hop, not end-to-end rate mismatch
  • Thinking Stop-and-Wait uses more than 1-bit sequence number — only 1 bit (0 and 1) needed
  • Confusing Stop-and-Wait with Go-Back-N or Selective Repeat (those are specific Sliding Window variants)
  • Forgetting that ARQ = Automatic Repeat reQuest — the reliability mechanism for noisy channels
  • Thinking piggybacking means sending the same data twice — it means attaching ACK to data going in reverse direction

Advanced Topics — Transport Layer Completion IIT-Level

Delayed Duplicate Problem (CRITICAL)

In packet switching networks, packets can be delayed, duplicated, or reordered. This creates confusion during connection establishment.

  • Server may receive SAME connection request twice
  • Cannot distinguish:
    • New request ❓
    • Delayed duplicate ❓
  • Occurs due to:
    • Network congestion
    • Retransmissions
    • Buffer delays

Sequence Number Concept

Each byte/segment is assigned a unique sequence number to ensure proper ordering and duplicate detection.

  • Helps detect duplicate packets
  • Ensures correct ordering
  • Used in reliability and flow control

Forbidden Region Concept

Sequence numbers must not overlap between old and new connections.

  • Each packet has lifetime = T
  • Within time T, same sequence number must NOT repeat
  • Overlap → confusion between old and new packets

Initial Sequence Number Selection

  • Must avoid reuse of old sequence numbers
  • Two solutions:
    • Wait for time T
    • Use higher sequence number

Problems in Packet Switching Network

  • Packet Loss (buffer overflow)
  • Packet Corruption
  • Arbitrary Delay
  • Duplicate Packets

Why Simple Handshake Fails?

  • Packets can be delayed
  • Duplicate requests confuse server
  • Client crash scenario creates ambiguity
