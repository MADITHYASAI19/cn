Topic 1: Routing Fundamentals Core Foundation

What is routing?

Routing is the process of determining the best path a packet should take from source to destination across a network of interconnected routers. This is distinct from forwarding (the actual act of moving a packet from an input port to an output port using the forwarding table). Routing protocols build and maintain the routing table; forwarding uses the built routing/forwarding table.

Key explanation points — Fundamentals

1Routing vs Forwarding: Routing = control-plane function. Routers exchange information with neighbours, run routing algorithms, and build routing tables. Happens periodically or on topology changes. Forwarding = data-plane function. For each arriving packet, look up destination IP in forwarding table, identify output port, send packet. Routing is slow (seconds); forwarding must be fast (nanoseconds per packet).
2Routing table vs Forwarding table: Routing table: built by routing protocols. Contains: Destination network, next-hop router, metric (cost), and which routing protocol installed this route. May have multiple routes to same destination. Forwarding table (FIB — Forwarding Information Base): derived from routing table. Contains: Destination prefix → output interface + next-hop MAC. Optimised for fast lookup (hardware TCAM).
3Autonomous System (AS): A collection of IP networks and routers under the control of a single organisation (ISP, university, enterprise) with a common routing policy. Each AS is assigned a unique ASN (Autonomous System Number) by IANA/RIRs. 16-bit (1–65535) historically; now 32-bit (4-byte ASNs). Private ASNs: 64512–65534. Example: Google AS15169, Airtel AS24560. Internet = collection of thousands of interconnected ASes.
4Intra-domain vs Inter-domain routing: Intra-domain (IGP — Interior Gateway Protocol): routing within a single AS. Goal: find shortest/least-cost path. Examples: RIP (Distance Vector), OSPF (Link State), EIGRP (Hybrid). Inter-domain (EGP — Exterior Gateway Protocol): routing between different ASes. Goal: follow policy (not just shortest path). Only one standard today: BGP (Border Gateway Protocol).
5Graph model of a network: The network is modelled as a graph G = (N, E) where N = set of routers (nodes) and E = set of links (edges). Each edge has a cost/weight (can represent delay, bandwidth, hop count, administrative metric). Goal of routing algorithm: find the minimum-cost path from source to destination in this graph. Different routing protocols use different algorithms to compute this.
6Static vs Dynamic routing: Static routing: routes entered manually by administrators. Simple, predictable, zero protocol overhead. Problems: does not adapt to topology changes (link failures), does not scale. Used for small networks or default routes. Dynamic routing: routing protocols automatically discover neighbours, exchange network reachability information, recompute routes on topology changes. Scales to large networks. Converges automatically after failures.
7Convergence: The time taken for all routers in a network to have a consistent, correct view of the network topology after a change (link failure, new link, new router). Fast convergence = less routing instability, less packet loss during changes. Distance Vector protocols converge slowly (count-to-infinity problem). Link State protocols converge faster. BGP convergence can take minutes in the internet — impractical for real-time failure recovery.
8Routing algorithm classification: (1) Centralised vs Decentralised: Link State = centralised knowledge (each router has full topology); Distance Vector = decentralised (each router only knows neighbours). (2) Static vs Dynamic: static routes vs protocols that respond to topology changes. (3) Load-sensitive vs Load-insensitive: most internet protocols are load-insensitive (don't consider current congestion in link cost). ARPANET originally used load-sensitive — caused oscillations.

Topic 2: Distance Vector Routing & RIP Most Important

Distance Vector — overview

Distance Vector (DV) routing is a distributed, iterative, asynchronous algorithm. Each router maintains a table (vector) of distances (costs) to every destination in the network. Routers share their tables only with directly connected neighbours. Based on received neighbour tables, each router recomputes its own table using the Bellman-Ford equation.

Key explanation points — Distance Vector Algorithm

1Bellman-Ford equation: The core of DV routing. For each router x, and for each destination y: D_x(y) = min over all neighbours v { c(x,v) + D_v(y) }. Where c(x,v) = cost of link from x to its neighbour v, and D_v(y) = v's estimated distance to y. Router x tries all its neighbours and picks the one that gives the minimum total cost to y. That neighbour becomes the next-hop for destination y.
2DV algorithm operation (distributed Bellman-Ford): Each router: (1) Initialises its distance to itself = 0, to all others = ∞. (2) Sends its distance vector to all directly connected neighbours. (3) On receiving a neighbour's DV, recomputes D_x(y) for all y using Bellman-Ford. (4) If D_x(y) changes for any y → update table and send updated DV to all neighbours. (5) Repeat until no updates — algorithm has converged. Works asynchronously — routers don't need to be in sync.
3Good news travels fast: When a link cost decreases (e.g., new faster link), the improvement propagates rapidly. Affected router immediately computes improved distance and tells its neighbours. Neighbours update and tell their neighbours. In O(number of hops) rounds, all routers converge on the new optimal paths. This is the "good news" scenario — DV handles it efficiently.
4Count-to-infinity problem (bad news travels slowly): When a link fails or cost increases, DV routing can converge extremely slowly. Example: A—B—C. A-B link fails. A knows it can't reach C directly. But A hears B's table which says B can reach C with cost 2 (via A). A thinks: "I can reach C via B at cost 1+2=3." A updates table. B gets this and thinks: "I can reach C via A at cost 1+3=4." This loop continues, incrementing by 2 each round, until cost reaches ∞ (some max value like 16 in RIP). Very slow convergence for bad news.
5Solutions to count-to-infinity: (1) Split Horizon: if router B learned a route to destination X from router A, then B does NOT advertise this route back to A. "Don't tell me what you learned from me." Breaks simple 2-node routing loops. (2) Poison Reverse (stronger): if B learned route to X from A, B advertises the route to A but with metric = ∞ (poison = explicitly advertise the route as unreachable back to the source). Better than simple split horizon — handles more loop scenarios. (3) Hold-down timers: when a route is invalidated, don't accept updates for it for a hold-down period. Prevents premature reinstallation of stale routes.
6Maximum hop count (infinity = 16 in RIP): RIP defines infinity = 16 hops. Any route with metric ≥ 16 is considered unreachable. This limits count-to-infinity to at most 16 iterations. Consequence: RIP cannot support networks larger than 15 hops diameter. Any destination more than 15 hops away is unreachable in RIP. This is why RIP is limited to small networks.
7Triggered updates: Normal DV sends updates periodically (every 30s in RIP). Triggered updates send an immediate update whenever a route changes, before the next periodic update. This speeds up convergence significantly. If a link fails at time t, router detects it immediately and sends a triggered update rather than waiting up to 30 seconds for the next periodic update.

Key explanation points — RIP (Routing Information Protocol)

8RIP overview: RIP (Routing Information Protocol) is the oldest and simplest distance vector IGP. Defined in RFC 1058 (RIPv1), RFC 2453 (RIPv2). Metric: hop count (number of routers a packet must traverse). Simple — does not consider bandwidth, delay, or congestion. Maximum metric = 15 (16 = infinity = unreachable). Maximum network diameter = 15 hops. Suitable only for small networks.
9RIP operation details: Routers send full routing table to neighbours every 30 seconds (periodic updates). Transport: UDP port 520. After 180 seconds (6 missed updates) without hearing about a route → route marked invalid (metric set to 16). After 240 seconds (hold-down) → route removed from table. RIPv1: classful (no subnet mask in updates). RIPv2: classless (includes subnet mask, supports VLSM and CIDR). RIPng: RIP for IPv6.
10RIPv1 vs RIPv2 differences: RIPv1: classful only (no subnet mask in advertisements), no authentication, broadcast updates. RIPv2: subnet masks included (VLSM/CIDR support), MD5 authentication, multicast updates (224.0.0.9 instead of broadcast). Both use hop count metric. Both limited to 15 hops diameter. Both have slow convergence and count-to-infinity problem.
11RIP convergence time: Worst case convergence time for RIP ≈ 6 × 30 = 180 seconds (must miss 6 updates before declaring a route invalid). In a 15-hop network, count-to-infinity can take many minutes. This makes RIP unsuitable for any network where fast failover is required. Modern alternatives (OSPF, EIGRP, IS-IS) converge in seconds.
12EIGRP (Enhanced IGRP): Cisco proprietary hybrid protocol — combines DV simplicity with some LS features. Uses DUAL (Diffusing Update Algorithm) — guarantees loop-free paths. Metric: composite (bandwidth + delay + reliability + load). Sends only incremental updates (not full table). Very fast convergence. Supports VLSM, CIDR. Not fully open standard (though later partially opened), so less universal than OSPF.

Topic 3: Link State Routing & OSPF Most Important

Link State — overview

Link State (LS) routing gives every router a complete map of the entire network topology. Each router floods information about its directly connected links to all other routers. Every router independently runs Dijkstra's shortest path algorithm on this complete map to compute the shortest path tree rooted at itself — giving it the optimal next-hop to every destination.

Key explanation points — Link State Algorithm

1Link State Advertisement (LSA): Each router creates an LSA (Link State Advertisement) that contains: (1) Router ID (its own identity). (2) List of all directly connected neighbours with link costs. (3) Sequence number (to detect stale LSAs). (4) Age (TTL — LSAs expire). Routers flood LSAs using reliable flooding: when a router gets an LSA, it forwards it out all interfaces except the one it arrived on. Duplicate LSAs (seen before) are dropped.
2Link State Database (LSDB): Each router collects LSAs from all routers in the network and stores them in its LSDB (Link State Database). When the LSDB is complete (contains LSAs from all routers), each router has an identical view of the entire network topology — a complete weighted graph. Then each router independently runs Dijkstra's algorithm on this graph to compute shortest paths.
3Dijkstra's algorithm — concept: A greedy shortest path algorithm. Start at source node s. Maintain: (1) Visited set N' — nodes whose shortest path from s is confirmed. (2) Distance array D(v) — current best known distance from s to each node v. Initialise: D(s)=0, D(all others)=∞. Each step: pick node w not in N' with minimum D(w) → add to N'. Then relax: for each neighbour u of w, if D(w)+c(w,u) < D(u) → update D(u). Repeat until all nodes in N'. Result: shortest path tree.
4Dijkstra's algorithm — example: Network: A-B cost 2, A-C cost 5, B-C cost 1, B-D cost 3, C-D cost 1. Find shortest paths from A. Start: D(A)=0, others=∞. Step 1: visit A (min). Update: D(B)=2, D(C)=5. Step 2: visit B (D=2). Update: D(C)=min(5, 2+1)=3, D(D)=2+3=5. Step 3: visit C (D=3). Update: D(D)=min(5, 3+1)=4. Step 4: visit D (D=4). Done. Shortest paths: A→B=2, A→C=3 (via B), A→D=4 (via B,C).
5Dijkstra's complexity: Naive implementation: O(n²) where n = number of nodes. With a priority queue (binary heap): O((n+E) log n) where E = edges. With Fibonacci heap: O(E + n log n). For practical router implementations with thousands of nodes, the priority queue version is used. Each router independently computes the same result — no coordination needed after LSA flooding is complete.
6LS vs DV — key comparison: LS: each router has full topology → runs Dijkstra locally. More memory (full LSDB), more computation (Dijkstra), but converges fast (milliseconds with triggered LSAs), no count-to-infinity. DV: each router only knows neighbours → distributed Bellman-Ford. Less memory, less computation per iteration, but converges slowly, count-to-infinity problem. LS is preferred for real networks; DV (RIP) only for small/simple networks.
7Oscillation problem in LS: If link cost is based on current traffic load (load-sensitive), and all routers simultaneously recompute paths → they all shift to the less-loaded path → that path becomes overloaded → back again. This oscillation makes the network unstable. Solution: use load-insensitive metrics (fixed administrative costs). Modern OSPF uses static link costs (usually based on bandwidth only, not current load).

Key explanation points — OSPF

8OSPF — Open Shortest Path First: The standard link-state IGP. RFC 2328 (OSPFv2 for IPv4), RFC 5340 (OSPFv3 for IPv6). Open standard (not proprietary). Metric: cost (typically 10⁸ / link_bandwidth_bps — so faster links have lower cost). Example: 100Mbps link = cost 1; 10Mbps link = cost 10. Sends LSAs using IP protocol number 89 (not TCP or UDP — built directly on IP). Uses multicast: 224.0.0.5 (all OSPF routers), 224.0.0.6 (DR/BDR only).
9OSPF areas: Large OSPF networks are divided into areas to limit the scope of LSA flooding and reduce LSDB size. Each area has its own LSDB — routers only flood LSAs within their area. Areas connect through Area 0 (Backbone area). Inter-area routing: routers on area boundaries (ABRs — Area Border Routers) summarise routes and advertise summaries into Area 0. Benefits: smaller LSDB, faster SPF computation, less flooding overhead, better scalability.
10OSPF router types: (1) Internal Router: all interfaces in one area. (2) Backbone Router: has interface in Area 0. (3) ABR (Area Border Router): connects two or more areas. Maintains LSDB for each connected area. Summarises routes between areas. (4) ASBR (AS Boundary Router): connects OSPF domain to external routing domain (e.g., redistributes BGP routes into OSPF). (5) DR/BDR: roles on broadcast/multi-access networks (see below).
11DR and BDR on broadcast networks: On broadcast networks (Ethernet LAN), if N routers on same segment all form OSPF adjacencies with each other → O(N²) adjacencies and LSAs. Solution: elect a DR (Designated Router) and BDR (Backup DR). All routers form adjacency only with DR and BDR. All routing updates flow through DR. DR's LSA represents the broadcast segment and all routers on it. Reduces adjacencies from O(N²) to O(N). Election: highest OSPF priority wins; tie-break: highest Router ID (highest IP address).
12OSPF packet types: 5 types: (1) Hello — neighbour discovery and keepalive. (2) DBD (Database Description) — summary of LSDB contents during adjacency formation. (3) LSR (Link State Request) — request specific LSAs from neighbour. (4) LSU (Link State Update) — contains one or more LSAs. (5) LSAck (Link State Acknowledgement) — acknowledge received LSUs. Reliable flooding: uses sequence numbers + explicit ACKs.
13OSPF adjacency formation (Hello protocol): Routers send Hello packets every hello interval (default 10s on point-to-point; 10s on broadcast). A neighbour is declared dead if no Hello received for dead interval (default 4 × hello = 40s). Hello packets contain: Router ID, Area ID, Hello/Dead intervals, DR/BDR, Neighbour list. Two routers become neighbours only if: same Area ID, same subnet, same hello/dead intervals, same authentication. Adjacency formed after LSDB synchronisation (DBD + LSR + LSU exchange).
14OSPF convergence: When a link fails: router detects failure (dead interval expires or physical link down). Immediately sends LSU with updated LSA (link removed, sequence number incremented). LSA flooded throughout the area. All routers rerun Dijkstra and compute new forwarding table. Total time: typically <1 second for link-down detection + SPF computation (sub-second with modern implementations). Much faster than RIP's 180-second convergence.

Topic 4: Path Vector Routing & BGP Most Important

BGP — the glue of the internet

BGP (Border Gateway Protocol) is the only EGP deployed on the internet. It is a Path Vector protocol — like distance vector, but instead of advertising a cost/metric, BGP advertises the complete AS path (sequence of ASes) to reach each destination prefix. This eliminates routing loops (if your own AS is in the AS path, drop the route) and enables rich policy control.

Key explanation points — Path Vector & BGP Architecture

1Path Vector concept: Each router advertises not just the distance/cost to a destination, but the complete path (sequence of ASes) taken to reach it. Example: AS1 says "I can reach prefix 10.0.0.0/8 via path [AS1, AS2, AS5]." When AS3 receives this, it prepends itself: "I can reach 10.0.0.0/8 via path [AS3, AS1, AS2, AS5]." Loop prevention: if a router sees its own AS in the AS_PATH attribute → drop the route (it loops back). Solves count-to-infinity completely — loops are detected immediately.
2EBGP vs IBGP: EBGP (External BGP): BGP session between routers in DIFFERENT ASes. Typically: directly connected border routers of neighbouring ASes. Routes exchanged between ASes. TTL is 1 (must be directly connected by default). IBGP (Internal BGP): BGP session between routers within the SAME AS. Used to propagate externally-learned routes across the AS to all routers. Can be between non-directly-connected routers (multi-hop IBGP). Requires full-mesh or Route Reflectors.
3IBGP full-mesh problem: IBGP rule: a route learned from an IBGP peer must NOT be re-advertised to another IBGP peer (to prevent loops within a single AS, since IBGP doesn't add AS number in path). This means: in an AS with N routers, every router must have IBGP sessions with every other router = N(N-1)/2 sessions (full mesh). Does not scale for large ASes. Solutions: (1) Route Reflectors (RR): designated router reflects IBGP routes to clients (like BGP DR). (2) Confederations: split large AS into smaller sub-ASes.
4BGP session establishment: BGP runs on top of TCP port 179. Provides reliable, ordered delivery of BGP messages. Two routers wanting to exchange BGP routes must first establish a TCP connection, then exchange BGP OPEN messages (agree on parameters), then exchange routing information via UPDATE messages. BGP peers are called BGP neighbours or peers. A BGP session is called a BGP peering.
5BGP message types: (1) OPEN: establish BGP session (version, ASN, hold time, BGP router ID). (2) UPDATE: advertise new routes or withdraw withdrawn routes. Contains NLRI (Network Layer Reachability Information = prefixes) + path attributes. (3) NOTIFICATION: error message — terminates the BGP session. (4) KEEPALIVE: sent periodically to maintain the session (default every 60s; hold timer = 180s — session drops if no KEEPALIVE for 180s).
6BGP path attributes: When advertising a prefix, BGP attaches path attributes: (1) AS_PATH: sequence of ASes traversed (mandatory). (2) NEXT_HOP: IP of next router to reach the prefix (mandatory). (3) LOCAL_PREF: preference within an AS (for IBGP — higher = preferred). (4) MED (Multi-Exit Discriminator): hint to neighbouring AS about preferred entry point (lower MED = preferred). (5) ORIGIN: how route was learned (IGP=i, EGP=e, Incomplete=?). (6) COMMUNITY: tag for grouping routes for policy.
7BGP best-path selection: When BGP receives multiple paths to the same prefix from different peers, it selects ONE best path using a decision process (in order): (1) Highest WEIGHT (Cisco proprietary, local to router). (2) Highest LOCAL_PREF. (3) Locally originated (network command or redistribute). (4) Shortest AS_PATH. (5) Lowest ORIGIN (i < e < ?). (6) Lowest MED. (7) EBGP over IBGP. (8) Lowest IGP metric to next-hop. (9) Lowest Router ID. Only ONE best path is installed and advertised. Mnemonic: We Love Oranges AS Oranges Mean Pure Refreshment.
8BGP policy — why it matters: Unlike OSPF which finds the shortest path, BGP finds the path that satisfies policy. An AS may prefer routes based on: commercial agreements (prefer routes via provider A who pays more), traffic engineering (load balancing outbound traffic across multiple exit points), security (avoid routing through certain ASes). BGP provides the tools: LOCAL_PREF controls outbound path selection. MED influences how neighbours send traffic in. AS_PATH prepending makes your route look longer (less preferred by others).
9BGP route aggregation: ISPs aggregate many prefixes into a shorter prefix before advertising to neighbours (to reduce the global BGP routing table size). Example: an ISP with 10.1.0.0/24, 10.2.0.0/24 ... 10.255.0.0/24 can advertise just 10.0.0.0/8 to the wider internet. Aggregation reduces the global routing table (currently ~900k IPv4 routes). Without aggregation, the internet routing table would be orders of magnitude larger.
10BGP convergence & stability issues: BGP convergence after a topology change can take minutes (much slower than OSPF, which takes seconds). Reason: BGP uses TCP (reliable delivery), changes propagate hop-by-hop across ASes, MRAI (Minimum Route Advertisement Interval) = 30s for EBGP, 5s for IBGP — limits how often updates can be sent. BGP instability (route flap) can be damped with route dampening (suppress frequently flapping routes). BGP carries the full internet routing table (~1 million IPv4 prefixes) — convergence of all of these after a major event is complex.
11BGP and internet routing security: BGP has no built-in authentication for route correctness. BGP hijacking: a malicious/misconfigured AS advertises prefixes it doesn't own → traffic redirected to the hijacker (e.g., Pakistan Telecom hijacked YouTube in 2008; China Telecom hijacked US military traffic in 2010). BGP route leaks: customer accidentally advertises provider routes to another provider → becomes unintended transit. Solution: RPKI (Resource Public Key Infrastructure) — cryptographically signs prefix-to-ASN mappings. Route Origin Authorization (ROA) validates that an AS is authorised to originate a prefix.
12Hot potato routing: An intra-AS routing policy where a router hands off a packet (to exit the AS) as quickly as possible — through the nearest IBGP exit point (lowest IGP cost to EBGP next-hop), even if a farther exit would result in a shorter end-to-end path. Each AS passes the "hot potato" (processing cost) to the next AS as fast as possible. Opposite: cold potato routing (early exit through the AS to deliver the packet via the best overall path — used in some CDNs).

  • Routing = control plane (build tables). Forwarding = data plane (move packets using table)
  • AS (Autonomous System) = network under single organisation. ASN uniquely identifies it
  • IGP (intra-AS): RIP, OSPF, EIGRP. EGP (inter-AS): only BGP in use today
  • DV (Distance Vector): distributed Bellman-Ford. Share table with neighbours only
  • Bellman-Ford: D_x(y) = min over v { c(x,v) + D_v(y) }. v = direct neighbours
  • Count-to-infinity: bad news (link failure) propagates slowly in DV — routing loops increment metric to ∞
  • Solutions: Split Horizon (don't advertise route back to source), Poison Reverse (advertise as ∞), Hold-down timers, Triggered updates
  • RIP: hop count metric. Max metric=15 (16=∞=unreachable). UDP port 520. Updates every 30s. RIPv2 = classless (subnet mask included)
  • RIPv1 = classful (no mask). RIPv2 = classless + MD5 auth + multicast 224.0.0.9
  • LS (Link State): flood LSAs to all. Each router has complete topology (LSDB). Run Dijkstra locally
  • Dijkstra: greedy algorithm. O(n²) naive; O((n+E)log n) with heap. Finalized distances in N'
  • OSPF: IP protocol 89 (not UDP/TCP). Cost = 10⁸/bandwidth. Multicast 224.0.0.5 (all) / 224.0.0.6 (DR/BDR)
  • OSPF areas: Area 0 = backbone. ABR connects areas. ASBR connects to external. LSAs flooded only within area
  • OSPF DR/BDR: elected on broadcast segments. Reduces adjacencies from O(N²) to O(N). Highest priority wins; tie-break = highest Router ID
  • OSPF hello=10s, dead=40s (4×hello). Packets: Hello, DBD, LSR, LSU, LSAck. Convergence: seconds
  • BGP: only EGP. Path Vector protocol. TCP port 179. RFC 4271 (BGP-4)

  • EBGP = between different ASes (TTL=1). IBGP = same AS (multi-hop OK)
  • IBGP full-mesh: can't re-advertise IBGP routes to other IBGP peers → full-mesh or Route Reflectors
  • BGP messages: OPEN, UPDATE, NOTIFICATION, KEEPALIVE. Keepalive every 60s, hold=180s
  • BGP path attributes: AS_PATH (mandatory), NEXT_HOP, LOCAL_PREF, MED, ORIGIN, COMMUNITY

1RIP uses UDP (port 520). OSPF uses IP directly (protocol 89 — NOT TCP/UDP). BGP uses TCP (port 179)
2RIP infinity = 16 → max hop count in network = 15 (not 16). 16th hop = unreachable
3Split Horizon = omit route. Poison Reverse = advertise route with metric ∞. Both prevent loops
4Dijkstra cannot handle negative-weight edges. Bellman-Ford can (detects negative cycles). Internet uses positive costs — Dijkstra is fine
5OSPF DR election: highest PRIORITY wins. Tie-break = highest ROUTER ID (usually highest IP). Priority 0 = never becomes DR
6EBGP: directly connected (TTL=1 by default). IBGP: multi-hop allowed (TTL=255 typically)
7BGP best-path: LOCAL_PREF checked BEFORE AS_PATH length. Higher LOCAL_PREF wins even over shorter path
8IBGP full-mesh: N routers = N(N-1)/2 sessions. Solution: Route Reflectors, not DR (DR is OSPF concept)
9LS flooding = LSAs sent out ALL interfaces except arrival interface. Not a broadcast to all — done hop-by-hop
10Intra-domain = IGP (OSPF, RIP). Inter-domain = EGP (BGP only). BGP is NOT an IGP
11Good news travels fast / Bad news travels slowly — this refers to DV (count-to-infinity). LS has no such problem
12OSPF cost calculation: if link=100Mbps, ref=10⁸, cost=10⁸/10⁸=1. If link=10Mbps, cost=10⁸/10⁷=10. If link=1Gbps, cost=0.1→rounds to 1 (minimum)
13Path Vector = BGP. Distance Vector = RIP, EIGRP. Link State = OSPF, IS-IS. Hybrid = EIGRP. Don't mix them up
14BGP KEEPALIVE = 60s. Hold timer = 180s. OSPF Hello = 10s. OSPF Dead = 40s. RIP update = 30s. RIP invalid = 180s

Exam strategy — Week 8

  • Bellman-Ford equation: always write it out. Practice trace through 3–4 node examples step by step
  • Count-to-infinity: trace the A–B–C failure example. Know why split horizon helps and why it's not complete
  • Dijkstra: practice on 5-node example. Trace N' set expansion step-by-step. Show D(v) updates clearly
  • OSPF cost calculation: formula = 10⁸/BW. Know that 1Gbps and 10Gbps both give cost=1 (problem with default ref bandwidth)
  • DV vs LS comparison table: memory, topology knowledge, convergence speed, loop prevention. Very likely exam question
  • BGP best-path order: memorise top 4 (Weight → LOCAL_PREF → local origin → AS_PATH). Exam typically asks about LOCAL_PREF vs AS_PATH priority
  • IBGP vs EBGP: same AS vs different AS. Full-mesh problem and Route Reflectors as solution
  • Protocol transport: RIP=UDP 520, OSPF=IP proto 89, BGP=TCP 179. Frequently tested
  • BGP loop prevention via AS_PATH: if own ASN appears → drop. Simple and effective
