Topic 1: Data Link Layer — Role, Functions & LLC Services Exam Favorite

What Does the Data Link Layer Do?

The Data Link Layer (Layer 2) is responsible for reliable transfer of frames between adjacent nodes (hop-by-hop). It bridges the gap between the physical layer (bits) and the network layer (packets). Key functions: Framing, Error Detection/Correction, Flow Control, and Medium Access Control (MAC).

Four Key Functions of DLL

1Framing: Divides the bit stream from the physical layer into discrete units called FRAMES. Defines where each frame begins and ends.
2Error Detection and Correction: Detects (and sometimes corrects) errors introduced by the physical link. Uses redundancy bits: parity, CRC, checksums, Hamming codes.
3Flow Control: Ensures the sender does not overwhelm the receiver. Key protocols: Stop-and-Wait, Sliding Window.
4Medium Access Control (MAC): Controls who gets to use the shared medium at any given time. Resolves contention on shared links (bus, wireless).

DLL Sublayers

  • LLC (Logical Link Control) — Upper sublayer: Provides the interface to the network layer. Multiplexes network protocols. Handles flow control and error control (in some implementations). Defined by IEEE 802.2.
  • MAC (Medium Access Control) — Lower sublayer: Controls access to the shared physical medium. Contains the hardware MAC address (48-bit = 6 bytes). Handles framing at the hardware level.

LLC Service Types Direct Theory Question

Type 1 — Unacknowledged Connectionless Service

  • Definition: Frames are sent without establishing a connection and without any acknowledgment. The sender does not know if the frame was received correctly.
  • ACK used? No — no acknowledgment sent by receiver.
  • Reliability: Unreliable. Error recovery (if needed) must be done by a higher layer (e.g., TCP).
  • Use case: Real-time traffic (voice, video streaming) where retransmission would cause more harm than the lost frame. Also used in Ethernet — fast, low overhead. The most widely used LLC service.

Type 2 — Connection-Oriented Service

  • Definition: A logical connection is established between sender and receiver before data transfer begins. Data is delivered reliably, in order, with error correction and flow control.
  • ACK used? Yes — every frame is acknowledged. Lost frames are retransmitted.
  • Reliability: Reliable. Similar to TCP but at the link layer.
  • Use case: Older WAN protocols (X.25, LAPB, HDLC) where the link itself was unreliable and a reliable DLL was needed. Less common today because transport-layer reliability (TCP) is preferred.

Type 3 — Acknowledged Connectionless Service

  • Definition: No connection is established (connectionless), but each frame IS individually acknowledged by the receiver. If no ACK is received within a timeout, the frame is retransmitted.
  • ACK used? Yes — each individual frame is acknowledged, but no prior connection setup.
  • Reliability: Reliable on a per-frame basis, but without the overhead of full connection establishment.
  • Use case: Industrial control networks (e.g., factory floor automation) where transactions are infrequent but each must be guaranteed. Connecting to a device that may be temporarily switched off.

DLL in the Context of Different Links

  • Point-to-point links: Only two nodes share the link. No contention. DLL simpler — PPP, HDLC.
  • Broadcast/shared links: Multiple nodes share same medium. Contention possible. DLL needs MAC sublayer. Examples: Ethernet (802.3), Wi-Fi (802.11).

Topic 2: Framing Methods High Weightage

Why Framing?

The physical layer delivers a raw stream of bits. The DLL must organize these bits into manageable, distinguishable units — frames. The receiver needs to know where each frame starts and ends.

Framing Methods

1. Character Count

  • First field of the frame header contains the NUMBER of characters (or bytes) in the frame.
  • Receiver uses this count to determine frame boundaries.
  • Problem: If count field is corrupted by an error, the receiver loses synchronization and cannot recover. NOT reliable.

2. Flag Bytes with Byte Stuffing (Character-Oriented)

  • Frames begin and end with a special FLAG byte (e.g., 01111110 = 0x7E in HDLC).
  • Problem: Flag pattern might appear in data. Solution: Byte Stuffing (character stuffing).
  • Before transmission: if the flag byte appears in data, an ESCAPE byte is inserted before it.
  • If the escape byte itself appears in data, ANOTHER escape byte is inserted before it.
  • Receiver removes extra escape bytes (destuffing).
  • Used in: PPP (Point-to-Point Protocol).

3. Flag Bits with Bit Stuffing (Bit-Oriented)

  • Frame delimiter = special bit pattern, e.g., 01111110 (flag = six 1s between two 0s).
  • Problem: Six consecutive 1s in data would look like a flag. Solution: Bit Stuffing.
  • Sender: After every five consecutive 1s in data, automatically inserts a 0 bit.
  • Receiver: After every five consecutive 1s, if the next bit is 0, it is removed. If it is 1 followed by 0, it is a flag (end of frame).
  • Used in: HDLC, SDLC. Bit-efficient — works at bit level.

4. Physical Layer Coding Violations

  • Uses special signal patterns that are INVALID in the normal encoding scheme to mark frame boundaries.
  • Example: In Manchester encoding, a deliberate violation (no transition) marks start/end of frame.
  • No stuffing needed. Only works when the physical layer encoding has unused signal patterns.
  • Used in: IEEE 802.5 (Token Ring), 100Base-TX Ethernet (uses 4B/5B encoding to mark delimiters).

Topic 3: Error Detection & Correction Most Important

Why Error Detection?

Physical links are imperfect — noise, interference, and hardware limitations introduce bit errors. The DLL uses redundancy bits to detect (and sometimes correct) errors. Key principle: send more bits than necessary so the receiver can verify correctness.

Error Detection Methods

1Parity Bit (Single Bit): Add one extra bit to make total number of 1s even (even parity) or odd (odd parity). Detects ALL single-bit errors. CANNOT detect 2-bit errors (two flips cancel out). Cannot correct errors.
22D Parity (Longitudinal & Transverse): Arrange data in a matrix. Compute parity for each row AND each column. Can detect AND correct single-bit errors. Can detect ALL 2-bit errors.
3Checksum: Treat data as 16-bit integers, add them up (one's complement arithmetic), and send the complement. Receiver adds all including checksum — should get all 1s. Used in TCP/UDP/IP headers. Weak — cannot detect many error patterns.
4CRC (Cyclic Redundancy Check): Most powerful standard error detection. Treat data as a polynomial, divide by a GENERATOR polynomial (e.g., CRC-32). Send remainder as the FCS (Frame Check Sequence). Receiver divides: if remainder = 0, frame is correct. Detects ALL single-bit errors, ALL burst errors ≤ degree of generator, most other burst errors.

Hamming Code — Error Correction

  • Developed by Richard Hamming at Bell Labs.
  • Inserts PARITY BITS at positions that are powers of 2 (positions 1, 2, 4, 8, 16…).
  • Each parity bit covers a specific set of data bits (based on binary representation of bit positions).
  • Can DETECT and CORRECT single-bit errors using the syndrome (parity check results).
  • Syndrome non-zero → identifies EXACTLY which bit is wrong → flip it.
  • For n data bits, need r parity bits such that 2^r ≥ n + r + 1.
  • Used in: RAM chips (ECC memory), some disk systems. Extended Hamming can detect 2-bit errors.

CRC — Key Details (NPTEL Favorite)

  • Generator polynomial examples: CRC-16 (x¹⁶+x¹⁵+x²+1), CRC-32 (used in Ethernet).
  • Sender appends r zeros to data (r = degree of generator), divides by generator using XOR division, gets remainder → this is the FCS.
  • Receiver appends FCS to data and divides by generator → if remainder = 0, no error detected.
  • CRC can detect: All 1-bit errors, All 2-bit errors (with correct generator), All odd number of bit errors, All burst errors of length ≤ r, most burst errors of length > r.

Topic 4: Flow Control, ARQ Protocols & Piggybacking Very Important

What is Flow Control?

Flow control prevents a fast sender from overwhelming a slow receiver. Without flow control, the receiver's buffer overflows and frames are lost. At DLL, flow control is per-hop.

Piggybacking Important — Missing from Many Notes

Piggybacking — Combining ACK with Data

  • Definition: The technique of attaching (piggybacking) an acknowledgment to an outgoing data frame, rather than sending the ACK as a separate frame.
  • Working example: Node A sends Frame 0 to Node B. Instead of Node B sending a bare ACK 0 back, Node B waits briefly until it has its OWN data frame to send to A. Then Node B sends "Data Frame + ACK 0" in a single combined frame. This way, the ACK "rides" on the data frame for free.
  • Advantage: Saves bandwidth — instead of 2 frames (1 data + 1 bare ACK), only 1 frame is sent. Especially significant on bidirectional (full-duplex) links.
  • Disadvantage: If Node B has no data to send, it must eventually send a bare ACK anyway (after a piggybacking timeout) to avoid the sender timing out and retransmitting.
  • Used in: TCP (the ACK field in every TCP segment is a piggybacked ACK), HDLC, and all full-duplex sliding window protocols.

Stop-and-Wait ARQ — Three Critical Scenarios

Stop-and-Wait Basic Operation

  • Sender transmits ONE frame, then STOPS and waits for an ACK.
  • If ACK received → send next frame. If timeout → retransmit same frame.
  • Efficiency = 1 / (1 + 2a) where a = Tp/Tf (propagation delay / transmission time).
  • Sequence numbers needed: only 1 bit (0 or 1) — alternating bit protocol.

Scenario 1 — Lost Frame

  • What happens: Sender transmits Frame N. Frame is lost in transit — receiver never receives it.
  • Receiver behavior: Receiver sees nothing. It does not send any ACK.
  • Sender behavior: Sender's retransmission timer expires (timeout). Sender retransmits Frame N.
  • Resolution: Retransmitted frame arrives at receiver → receiver sends ACK N → sender advances.
  • Key point: The timer is the ONLY mechanism the sender has to detect a lost frame. If no timer, sender waits forever.

Scenario 2 — Lost ACK

  • What happens: Sender transmits Frame N. Receiver receives Frame N correctly and sends ACK N. But the ACK is lost in transit.
  • Receiver behavior: Receiver has already delivered Frame N to the network layer. It is now waiting for Frame N+1.
  • Sender behavior: Sender's timer expires (ACK not received). Sender retransmits Frame N (same frame again).
  • Problem: Receiver gets Frame N again — it thinks this is a NEW frame N+1, but it is actually a DUPLICATE of Frame N.
  • Solution: Sequence numbers (1-bit alternating). Receiver checks the sequence number — Frame 0 again after expecting Frame 1 → it is a duplicate → receiver discards it but still sends ACK 0 again.

Scenario 3 — Delayed ACK (Premature Timeout)

  • What happens: ACK is NOT lost but is significantly delayed in the network (queued at an intermediate switch, etc.).
  • Sender behavior: Timer expires before the delayed ACK arrives. Sender retransmits Frame N.
  • Receiver behavior: Receiver may now receive TWO copies of Frame N — the retransmission AND the original delayed ACK arrives at the sender after retransmission.
  • Complication: Sender now receives two ACKs for Frame N. It may start sending Frame N+1 after the first ACK — then the second ACK arrives and could be misinterpreted as an ACK for Frame N+1.
  • Solution: 1-bit sequence numbers + careful ACK interpretation. Sender ignores duplicate ACKs for a frame it has already moved past.
  • Timer setting: Timer must be set appropriately — too short → unnecessary retransmissions; too long → wasted idle time after a real loss.

Sliding Window ARQ

Sliding Window Variables

  • Sender can have up to W frames "in flight" (unacknowledged) simultaneously. W = window size.
  • Efficiency = W / (1 + 2a) if W < 1+2a; = 1 (100%) if W ≥ 1 + 2a.

Go-Back-N Window Variables:

  • S (Send Base / SF): Sequence number of the oldest unacknowledged frame (the start of the sender's window). Also called the "send base".
  • SL (Send Last / Last frame sent): Sequence number of the most recently sent frame. The window spans from S to SL.
  • Window slides forward when the sender receives an ACK for S — S advances to S+1, and SL can advance by one more.
  • GBN Sender Window = 2^n − 1 (with n-bit sequence numbers).

Selective Repeat Window Variables:

  • RF (Receive First): The lowest sequence number the receiver is waiting for (the start of the receiver's window).
  • RL (Receive Last): The highest sequence number the receiver will accept. Frames outside [RF, RL] are discarded.
  • SR Receiver Window = 2^(n-1). Receiver buffers out-of-order frames within [RF, RL].

Go-Back-N — Cumulative ACK

  • In GBN, acknowledgments are cumulative: ACK N means "I have correctly received ALL frames up to and including frame N."
  • If the receiver sends ACK 5, it means frames 0, 1, 2, 3, 4, 5 have all been received correctly.
  • Benefit: Reduces ACK overhead — if multiple frames arrive before the ACK is sent, a single ACK covers all of them.
  • Used in TCP: TCP uses cumulative ACKs. "ACK 1000" means "I have received all bytes up to sequence 999; send me byte 1000 next."
  • GBN receiver has window size = 1 → can only accept next expected frame; out-of-order frames are discarded.

Bandwidth-Delay Product (BDP)

  • BDP = Bandwidth × Round-Trip Propagation Delay
  • Represents how many bits can be "in flight" at any given time.
  • For SAW to be 100% efficient, the frame must be large enough to fill the pipe.
  • For sliding window: Window size W ≥ BDP / Frame size = 1 + 2a.
  • Satellite links have huge BDP → large windows needed.

ALOHA Protocols — Efficiency Numbers High-Probability Numerical

Pure ALOHA

  • Working: When a node has a frame, it transmits immediately — no carrier sensing, no slot synchronization.
  • Collision: If two nodes transmit at any overlapping time, both frames are destroyed. A node that transmitted at time t can collide with any frame transmitted in the interval [t−Tf, t+Tf].
  • Vulnerable period: 2 × frame transmission time (2Tf) — twice as long as slotted ALOHA.
  • Maximum efficiency = 1/(2e) ≈ 18.4% — achieved when the channel load G = 0.5. This means at best, only 18% of the channel capacity is used for successful transmissions.
  • Throughput S = G × e^(−2G) where G = offered load (frames/slot).

Slotted ALOHA

  • Working: Time is divided into equal SLOTS, each exactly one frame long. Nodes can only begin transmission at the START of a slot. All nodes are synchronized.
  • Collision: A collision can only occur within the same slot — if two or more nodes start simultaneously. Outside the same slot, no collision is possible.
  • Vulnerable period: 1 × frame transmission time (Tf) — HALF that of pure ALOHA because a node's transmission only risks collision within one slot, not across two.
  • Maximum efficiency = 1/e ≈ 36.8% — achieved when the channel load G = 1. This is exactly DOUBLE the efficiency of pure ALOHA.
  • Throughput S = G × e^(−G).
  • Why slotted is better: By synchronizing transmission starts, the vulnerable period is cut in half, reducing the probability of collision and doubling useful throughput.

Token Passing — Step-by-Step Flow

How Token Ring / Token Passing Works

A special control frame called the token circulates around the ring (or logical ring). Only the node holding the token may transmit.

1Wait for Token: Each node monitors the ring/bus waiting for the token to arrive. The token is a short, distinctly formatted frame (no data payload) that identifies itself as "free" or "busy". Node cannot transmit until it captures the token.
2Capture Token: When the node's turn arrives and the free token passes, the node "captures" it by either removing it from the ring (in 802.4 Token Bus) or marking it as "busy" and appending data (in 802.5 Token Ring). The node now has exclusive transmission rights.
3Transmit Data: The capturing node sends its data frame(s) around the ring. Other nodes see the "busy" token and do NOT transmit — they simply regenerate and forward the frame.
4Time Limit or All Data Sent: The node is allowed to transmit for up to a token holding time (THT). When the time limit expires OR all queued frames are sent (whichever comes first), the node stops transmitting.
5Release Token: The node generates and releases a new FREE token back onto the ring. The token circulates to the next node, which can now capture it and repeat the process.

Why no collision? Only the token holder can transmit. Since only one node holds the token at any moment, there is never more than one node transmitting — so collision is physically impossible.

Limitation: If the token is lost (due to transmission error, node crash, or ring break), the entire network stalls. A designated monitor node detects token loss (by timeout) and generates a new token.

CSMA/CD — The Ethernet MAC Protocol

CSMA/CD Step-by-Step

  • CSMA = Carrier Sense Multiple Access: "Listen before you talk" — check if channel is idle before transmitting.
  • CD = Collision Detection: While transmitting, monitor the channel. If a collision is detected (another signal mixed in), STOP immediately.
  • Jam Signal: After detecting a collision, broadcast a 48-bit jam signal to notify all stations.
  • Binary Exponential Backoff: After collision attempt k, wait a random number of slot times in range [0, 2^k − 1], then retry. After 10 collisions, max range fixed at 1023. After 16 failures, give up and report error.
  • Minimum Frame Size (Ethernet): 64 bytes = 512 bits. Guarantees sender can detect collisions before finishing transmission.

ARP — Address Resolution Protocol Frequently Asked

What is ARP and Why is it Needed?

  • ARP resolves a known IP address (Layer 3) to its corresponding MAC address (Layer 2).
  • Why needed? When a host wants to send an IP packet to another host on the same LAN, it knows the destination IP address (from DNS or application) but needs the destination MAC address to create the Ethernet frame.
  • Position: ARP sits between Layer 2 (DLL) and Layer 3 (Network). It is sometimes called a "glue" protocol — it enables the IP layer to work over Ethernet.

ARP Working — Step by Step

1Check ARP Cache: Sender host looks up its ARP table (cache) for an existing MAC address entry for the destination IP. The ARP cache stores recently resolved IP→MAC mappings with a timeout (typically 20 minutes).
2Cache Miss → Broadcast ARP Request: If no cache entry found, sender broadcasts an ARP Request frame to all nodes on the LAN. Destination MAC = FF:FF:FF:FF:FF:FF (broadcast). Payload: "Who has IP address 192.168.1.5? Tell 192.168.1.1." Every device on the LAN receives this.
3Target Replies: The host with IP 192.168.1.5 recognizes its IP address in the request and sends a unicast ARP Reply directly back to the requester: "I have IP 192.168.1.5; my MAC is AA:BB:CC:DD:EE:FF." All other hosts ignore the request.
4Cache Update: Requester receives the ARP Reply, stores the IP→MAC mapping in its ARP cache with a timestamp. Future packets to the same IP use the cached entry without broadcasting again. The cache entry expires after the timeout period.
5Frame Sent: Sender now constructs the Ethernet frame with the resolved destination MAC address and sends the IP packet inside it to the intended host.

Hub — Multi-Port Repeater

Hub Architecture and Limitations

  • A hub is a Layer 1 (Physical layer) device — it has no intelligence, does not read MAC or IP addresses.
  • Multi-port repeater: When a frame arrives on any port, the hub repeats (regenerates and amplifies) it out of ALL other ports simultaneously.
  • Same collision domain: All nodes connected to a hub share one collision domain. Any two nodes that transmit simultaneously will collide. Hubs are half-duplex only.
  • Same broadcast domain: Broadcast frames are forwarded out all ports by the hub.
  • Result: As more nodes are added to a hub, collision frequency increases and overall throughput decreases. Hubs are now obsolete — replaced by switches in all modern networks.

Bridge & Bridged Ethernet

Bridge — LAN Segment Divider

  • A bridge connects two or more LAN segments together, operating at Layer 2 (MAC layer).
  • MAC learning: A bridge learns which MAC addresses are on which port by observing source MAC addresses of incoming frames. Builds a MAC address table.
  • Filtering: If the destination MAC is known and on the same port the frame arrived from, the bridge filters (drops) the frame — no need to forward it to other segments.
  • Forwarding: If the destination is on a different port, the bridge forwards the frame to that port only — reducing traffic on other segments.
  • Reduces collisions: Each bridge port is its own collision domain. Nodes on different bridge ports cannot collide with each other.
  • Multi-port bridge = Switch: A switch is simply a bridge with many ports — one collision domain per port (per connected node), which is why switches essentially eliminate collisions in modern Ethernet.

Full-Duplex Ethernet

Full Duplex — Collisions Eliminated

  • Half-duplex: A node can either send OR receive at a given time, but not both. CSMA/CD is required because collisions are possible when multiple nodes share the medium.
  • Full-duplex: A node can send AND receive simultaneously. Requires separate wires (or fiber strands) for each direction — no shared medium, so signals do not mix.
  • CSMA/CD not needed: Since full-duplex has a dedicated send path and a dedicated receive path point-to-point (switch port ↔ NIC), two signals can never collide. CSMA/CD is disabled.
  • Effective bandwidth doubles: A 100 Mbps full-duplex link gives 100 Mbps in each direction simultaneously = effective 200 Mbps aggregate.
  • Requirement: Both ends must support full duplex — negotiated via auto-negotiation. Used in all modern switched Ethernet (each switch port = dedicated full-duplex link to one node).

Fast Ethernet Auto-Negotiation

Auto-Negotiation (IEEE 802.3u)

  • Defined in IEEE 802.3u (Fast Ethernet standard, 100 Mbps).
  • Two Ethernet devices automatically negotiate the best compatible operating mode when they connect — without manual configuration.
  • Parameters negotiated: Speed (10 Mbps or 100 Mbps), Duplex (half-duplex or full-duplex).
  • Mechanism: Devices exchange Fast Link Pulses (FLPs) during the link establishment phase before data transmission begins. FLPs encode the capabilities of each device.
  • Highest common capability wins: If one device supports 100 Mbps full-duplex and the other supports only 10 Mbps half-duplex, the link operates at 10 Mbps half-duplex.
  • Auto-negotiation failure: If one side has auto-negotiation disabled and manually set to 100 Mbps full-duplex, while the other uses auto-negotiation and defaults to half-duplex, a duplex mismatch occurs — causing poor performance and many late collisions.

Ethernet Frame Format (IEEE 802.3)

Frame Structure

  • Preamble (7 bytes): 10101010... — synchronizes clocks of sender and receiver.
  • SFD (Start Frame Delimiter, 1 byte): 10101011 — signals start of actual frame.
  • Destination MAC (6 bytes): 48-bit hardware address of recipient.
  • Source MAC (6 bytes): 48-bit hardware address of sender.
  • Type/Length (2 bytes): Identifies the upper-layer protocol (e.g., 0x0800 = IPv4, 0x0806 = ARP).
  • Data (46–1500 bytes): Payload (minimum 46 bytes; padding added if less).
  • FCS (4 bytes): CRC-32 for error detection.
  • Total minimum frame size: 64 bytes (excluding preamble + SFD).

MAC Address (Hardware Address)

  • 48 bits = 6 bytes = 12 hexadecimal digits. Example: AA:BB:CC:DD:EE:FF
  • Burned into the NIC (Network Interface Card) at manufacture — also called hardware address or physical address.
  • First 3 bytes = OUI (Organizationally Unique Identifier) — assigned to manufacturer by IEEE.
  • Last 3 bytes = device-specific portion assigned by manufacturer.
  • Broadcast MAC address: FF:FF:FF:FF:FF:FF (all ones) — received by ALL stations.
  • Multicast MAC address: least significant bit of first byte = 1.

Error Detection Method Comparison

→Parity (1-bit): 1 extra bit; detects 1-bit and all odd-count errors; NO correction.
→2D Parity: Row + column parity; detects+corrects 1-bit; detects all 2-bit errors.
→Checksum: 16-bit; used in IP/TCP/UDP; weak detection; one's complement sum.
→CRC: Strongest; r-bit FCS (remainder); used in Ethernet (CRC-32), Wi-Fi.
→Hamming: Parity at powers-of-2 positions; corrects single-bit errors.

Flow Control & MAC — Quick Reference

→Stop-and-Wait: 1 frame at a time; efficiency = 1/(1+2a); 1-bit seq nums.
→Go-Back-N: Window = 2^n − 1; cumulative ACK; retransmit from error; simple receiver.
→Selective Repeat: Window = 2^(n-1); individual ACK; retransmit only errored; needs larger receiver buffer.
→Optimal window = 1 + 2a to achieve 100% link utilization.
→Pure ALOHA: 18% max; Slotted ALOHA: 36% max. Both are random access (asynchronous contention).
→Token Ring: Round-robin; deterministic; no collision; token loss = stall.

ARP, Hub vs Switch vs Router

→ARP: check cache → broadcast request → unicast reply → cache entry → send frame.
→Hub: Layer 1, 1 collision domain for ALL ports, broadcasts to all — obsolete.
→Switch: Layer 2, collision domain per port, one broadcast domain — modern standard.
→Router: Layer 3, separate collision AND broadcast domain per port — connects networks.
→Bridge = 2-port switch; multi-port bridge = switch.

Exam Strategy — Week 10

  • LLC type questions: Type 1 = no ACK (Ethernet); Type 2 = full connection+ACK (HDLC); Type 3 = ACK, no connection (industrial)
  • ALOHA numericals: Pure ALOHA = 18%; Slotted ALOHA = 36%. Memorize: slotted = 2× pure.
  • For window size formula: GBN = 2^n − 1; SR = 2^(n−1). These WILL be asked.
  • Piggybacking saves bandwidth — ACK rides on data frame in the reverse direction.
  • For CRC questions: always check if remainder = 0 → no error. Non-zero → error detected.
  • CSMA/CD is Ethernet; CSMA/CA is Wi-Fi (no collision detection in wireless).
  • ARP Request = broadcast; ARP Reply = unicast. This distinction is commonly tested.
  • Hub vs Switch: Hub = all ports same collision domain; Switch = each port separate collision domain.
  • Router separates BROADCAST domains; switch does NOT (without VLANs).
  • Full-duplex: no CSMA/CD needed because separate send/receive wires prevent collisions.
